# 系统设计题内容库补全设计

**日期：** 2026-07-17  
**状态：** 已批准  
**范围：** 在不扩建实验框架的前提下，为 `scope.yaml` 中尚无正文的 74 个案例补齐可独立阅读的系统设计分析。

## 1. 背景与问题本质

仓库当前已经冻结 10 个问题族和 75 个规范案例，但公开可读内容只有 `distributed-rate-limiter` 一题。其余工作主要集中在 schema、校验器、实验 harness、证据存储和发布审计。作为 W0 纵向样板，这些设施证明了一条案例可以从推导走到证据；作为面向读者的系统设计知识库，它们却没有交付足够的题目内容。

这次工作的目标不是继续证明框架完整，而是把用户真正要阅读和练习的 74 道题补出来。系统设计题的客观本质统一写成：

```text
必须守住的不变量
+ 稀缺资源
+ 无法立即获知的信息
+ 业务允许牺牲的属性
+ 触发设计切换的可观察条件
```

文章必须从这五项推导架构，不能从 Redis、Kafka、某家公司或一张现成架构图开始。

## 2. 双口径完成度

现有 manifest 的 `status: complete` 表示完整依赖闭包：非空的 principle、claim、scenario lab、evidence requirement 和 source，以及所有反向绑定和实验依赖。它不是“正文已经写完”的状态。

本批次冻结该语义，不改 schema、不新增生命周期，也不为追求数字制造占位实验。新增的 74 个案例使用：

```yaml
required: true
status: draft
```

每个 manifest 仍填写题名、主问题族、适用维度和至少两个可证伪主张；不存在的 `principles`、`labs`、`evidence_requirements` 与 `sources` 不写。正文只使用 `DEDUCED`，或在满足原因与变化影响说明时使用 `ASSUMED`；没有实验不得写 `MEASURED`，没有直接权威来源不得写 `SOURCED`。

对外同时报告两个互不冒充的进度：

- **题目内容覆盖：75/75。** 规范案例均有 manifest 与完整八章正文。
- **实验与证据闭环：1/75。** 继续由 `generated/coverage.md` 和现有 `complete` 语义计算。

## 3. 每篇文章的固定结构

74 篇文章沿用金样板的八个二级标题，标题、顺序与含义保持不变：

1. `表面题目`
2. `反问与边界`
3. `客观模型`
4. `必然约束`
5. `从简单方案演进`
6. `设计决定`
7. `运行与演进`
8. `面试考察本质`

虽然 `draft` 当前不会触发正式内容校验器，本批次仍按 complete 内容门槛写作：八节正文分别不少于 120、180、240、240、300、240、240、240 个非代码字符。常规题目标长度为 2,400–4,200 个中文可见字符；分布式 SQL、支付、证券交易、容器编排和 AI 服务等复杂题可达到 4,800 字。

### 3.1 表面题目

说明谁发起什么操作、哪个状态变化、怎样算成功，以及品牌化题名背后的规范问题。明确主流程和至少一个容易被题名掩盖的歧义。

### 3.2 反问与边界

确定成功语义、SLO、读写比例、峰值与偏斜、一致性或误差要求、地域、成本、安全和非目标。生产数字未知时使用符号或显式假设，不能把猜测写成事实。

### 3.3 客观模型

定义最小 API 或命令、核心实体、状态所有者、状态机、版本关系和不变量。每篇至少包含一个容量、放大、延迟或误差公式，并说明热点、长尾或倾斜如何进入模型。

### 3.4 必然约束

至少给出两个 case-owned 的 `DEDUCED` 主张。推导必须包含前提、信息边界或物理下界、最小反例、结论和适用边界，不能只罗列 CAP、扩展性或高可用等名词。

### 3.5 从简单方案演进

从最简单正确基线开始，指出哪个可观察指标先越界，再引入一种必要复杂度。每次演进都写出解决的问题、新增成本、新增失败模式和下一次切换条件；至少有两个可观察的切换条件。

### 3.6 设计决定

在已声明假设下做出明确选择，说明正常关键路径、正确性边界、超时、重试、幂等、降级和故障语义。至少写一个未选择方案以及它重新变优的条件。

### 3.7 运行与演进

给出真正反映用户体验或不变量的 SLI、过载和降级顺序、一次具体故障演练、灰度与回滚、数据或协议迁移，以及适用的多地域、安全、隐私、多租户和成本边界。

### 3.8 面试考察本质

第一段直接回答：给定什么不变量，因为何种信息或资源边界，候选人应推导出什么不可兼得关系，再依据什么业务风险做决定。随后说明优秀信号、常见误区、追问路径和 20/40/60 分钟表达方式。

## 4. 十个问题族的第一性原理核心

| 问题族 | 客观核心 |
| --- | --- |
| 寻址与流量 | 稳定身份如何映射到变化中的位置、预算或规则，并控制映射过期、偏斜和迁移成本。 |
| 分布式存储 | 一个逻辑写入如何在部分失败下获得可声明的持久性、可见性与演进能力。 |
| 信息流、社交与排序 | 个性化有序结果应在写时还是读时物化，以及偏斜如何放大计算、存储和公平性问题。 |
| 实时通信与协作 | 断连、重试和多写者条件下，消息怎样交付、排序、去重或最终收敛。 |
| 搜索、抓取与地理 | 派生索引如何近似不断变化的现实，在召回、时效、延迟、误差和成本间取舍。 |
| 媒体处理与分发 | 大字节对象及其派生版本如何在带宽、缓存命中、处理队列和启动延迟之间流动。 |
| 交易与争用 | 钱、库存、座位或所有权如何在并发与重试下守恒，并通过账本、幂等和对账恢复。 |
| 流处理、分析与可观测性 | 无界事件流如何保持有界进展，处理乱序、重放、背压、近似和状态恢复。 |
| 调度与控制平面 | 期望状态如何通过幂等协调收敛到实际状态，并以租约、fencing 和隔离避免双主。 |
| AI 与向量 | 概率质量、GPU 成本和尾延迟如何共同约束服务，同时保持模型、数据、提示词和索引版本可追踪。 |

同一问题族内的文章用六元组区分：`核心状态 / 不变量 / 主导放大 / 协调点 / 标志性故障 / 切换指标`。两个兄弟案例若六项中没有至少三项显著不同，视为内容重复，需要重写。

## 5. Manifest 契约

每个新增案例使用下面的最小实质结构：

```yaml
schema_version: 1
id: ticketing
title: 票务系统
primary_family: transactions-contention
required: true
status: draft
dimensions:
  - problem-slo
  - capacity-cost-skew
  - contracts-data-invariants
  - concurrency-transactions-idempotency
  - failure-recovery-disaster
claims:
  - id: ticketing-capacity-must-have-one-owner
    statement: 同一可售资源在任一时刻必须只有一个有效分配权威。
  - id: ticketing-hold-trades-utilization-for-contention-control
    statement: 临时占座降低并发超卖风险，但会制造库存闲置与回收问题。
```

dimension 只列正文实质覆盖的维度，不机械复制全部十二项。claim ID 必须以 case ID 为前缀并保持全仓库唯一。每个 claim 必须在正文至少出现一次，且标签类别保持一致。

## 6. 防模板化质量门槛

每篇必须通过以下检查：

1. **换标题测试：** 替换成同族另一题后仍成立的段落必须改写。
2. **唯一不变量：** 至少一个不变量只能属于本题。
3. **最小反例：** 至少一个具体数字或事件序列能击穿天真方案。
4. **两次量化推导：** 至少一次容量或放大计算，以及一次延迟、恢复或误差边界。
5. **事实源与写权限：** 明确谁拥有状态，哪里是线性化点或最终仲裁点。
6. **故障时间线：** 说明故障前后重试、内部状态和用户可见结果。
7. **切换条件：** 至少两个可观察阈值，禁止“规模大了再分片”。
8. **反选方案：** 写出未采用方案和它重新变优的条件。
9. **本质句不可泛化：** 必须同时含本题不变量、信息边界与主导取舍。
10. **组件后置：** 前半篇完成模型和推导，产品名只能作为后置实现映射。
11. **证据诚实：** 不伪造测量、来源、实验或引用。
12. **兄弟题差异：** 同族逐对审阅六元组，避免换名复用。

## 7. 导航与呈现

新增 `cases/README.md`，按 `scope.yaml` 的十个问题族列出全部 75 题并链接正文。根 `README.md` 的完成度改为双口径，首页先引导读者进入题目索引，再保留 W0 实验与证据说明。`docs/methodology/evidence.md` 同步解释内容覆盖不等于实验认证。

`generated/coverage.md` 继续由 CLI 生成并保持 1/75，因为它表达的是现有严格 `complete` 契约；不得手工将其改成内容覆盖率。

## 8. 实施批次

先用四个形状差异明显的案例校准写作：

- `url-shortener`
- `key-value-store`
- `ticketing`
- `llm-chat-serving`

校准不需要用户 review，由实现者按本设计自审。随后按问题族并行：

1. 寻址与流量、分布式存储、交易与争用；
2. 信息流与排序、实时协作、流处理与可观测性；
3. 搜索与地理、媒体分发、调度与控制平面；
4. AI 与向量、全库去模板化审计、导航与说明。

每个问题族由一个作者独占目录，公共索引和根文档最后统一修改。

## 9. 验收标准

- `cases/` 下存在与 `scope.yaml` 精确对应的 75 个 case 目录；
- 每个目录同时包含 `case.yaml` 与 `README.md`；
- 新增 74 篇均含八个精确有序 H2，且达到 complete 正文的逐节最低字符数；
- 每篇至少有两个 case-owned claim，manifest 与正文标签一一闭合；
- 不存在未完成标记、虚构来源、虚构实验或无效内部链接；
- `cases/README.md` 对 75 题均有可达链接；
- 根 README 明确报告内容 75/75、证据闭环 1/75；
- `go run ./cmd/whiteboard validate --root . --content` 通过；
- `go test ./...` 通过；
- `generated/coverage.md` 经 CLI 重新生成后仍诚实显示 strict complete 1/75。

## 10. 明确不做

- 不新增 principle、lab、adapter 或 evidence；
- 不修改 lifecycle、coverage、release、CLI 或 schema 语义；
- 不为 74 题实现生产服务或重复框架代码；
- 不把品牌架构当标准答案；
- 不把正文存在包装成实验已经通过；
- 不在本批次开发文档网站或额外 UI。
