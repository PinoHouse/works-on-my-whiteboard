# 大文件传输

## 表面题目

设计大文件传输系统，是让客户端在网络中断、进程重启和多次重试下上传或下载数 GB 到数 TB 的对象，恢复时只重传必要字节，并最终得到可验证的完整文件。上传一个块成功、所有 HTTP 请求成功和文件可见是三种状态；只有服务端确认完整块集合、每块摘要和整体摘要后原子发布 manifest，读取方才可看到成功对象。

本文以分块断点上传为主，也给出对称下载。文件内容不可变，更新产生新 generation；上传会话只是暂存容器，不能直接成为可下载对象。题名常掩盖“偏移量属于哪个文件版本”“complete 响应丢失后重试如何判定”和“并行越多是否越快”，这些才是正确性与资源边界。

## 反问与边界

先问文件大小分布、网络带宽与丢包、移动端/数据中心、峰值并发、单文件完成时限、允许的暂存保留、是否端到端加密及服务端能否计算明文摘要。块大小固定还是协商，客户端是否可信，重复块能否去重，下载是否需 Range 和多源。上传成功语义应明确是单故障域落盘、复制完成还是 manifest 发布；本文不把异步跨区复制冒充已完成。

设文件大小 `B`、块大小 `c`，块数 `n=ceil(B/c)`，每块元数据 `m`，索引开销约 `n×m`。无断点时在比例 `x` 处失败会重传约 `xB`；分块后理想重传上界接近未确认块加协议开销。并行度 `p` 的吞吐最多为 `min(client_upstream, network_path, service_share, p×single_stream_rate)`，超过最窄项不会增加有效字节，却使内存约 `p×c`、连接和失败重传上升。

## 客观模型

核心实体为 `UploadSession(session_id,generation,expected_size,expected_digest,chunking,expires)`、`Chunk(index,offset,length,digest,state)`、`FileManifest` 和 `PublishedObjectPointer`。创建会话冻结文件身份、总长、分块算法与租户；`PutChunk` 携带 session generation、index、长度与 digest，同身份同摘要幂等，不同摘要冲突。`Complete` 检查所有必需 index 恰好覆盖 `[0,B)`、无重叠/空洞、块可读，再计算或验证 Merkle/整体摘要并原子提交不可变 manifest。

状态机是 OPEN、VERIFYING、COMMITTED(manifest_digest)、ABORTED/EXPIRED。complete 带幂等 operation id；若响应丢失，查询同一 session 应返回已提交 manifest，而非新建文件。暂存块可内容寻址，但引用计数只有 manifest commit 后才成为永久引用；清理过期会话不能删除已被别的 manifest 引用的块。下载先读取一个 manifest generation，再按摘要验证块，不能在中途跟随逻辑名的新版本。

热点来自少数超大文件占据 worker/带宽以及大量小块放大元数据。若 `c` 太大，失败重传多且并行不足；太小，`n`、签名和请求开销爆炸。块大小应按带宽时延积、失败率与内存预算选择，而不是固定套用某个常数。

## 必然约束

[DEDUCED:large-file-transfer-resume-state-must-bind-to-content-identity] 客户端先上传文件 A 的前 1 GB，随后本地路径被文件 B 覆盖，若恢复记录只有“偏移 1 GB”，它会把 B 的后半段接到 A 前半段，长度甚至可能正确。会话必须绑定预期整体身份、分块参数和 generation，每块再验证摘要；身份变化应新建会话。仅靠文件名、修改时间或字节偏移无法证明内容连续。

[DEDUCED:large-file-transfer-chunk-success-does-not-imply-file-success] 三块文件中 0、1、2 分别返回成功，但块 1 被错误会话引用，或 complete 前块 2 尚未复制可读，直接暴露逻辑名会得到混合/截断对象。只有 manifest 对完整位置集合和 digest 做一次权威提交，才能把暂存传输与读可见性隔离。各块 ACK 只说明该块在声明的耐久域被接受。

[DEDUCED:large-file-transfer-parallelism-is-bounded-by-bottleneck-and-fairness] 客户端上行 100 Mb/s 时，将连接从 4 增至 40 不会突破 100 Mb/s；反而四十个 64 MB 缓冲要 2.5 GB，并在链路抖动时同时重传。共享服务若让一个 TB 文件占满所有 slot，小文件尾延迟无限增长。调度必须按租户/会话分配带宽与 in-flight 字节，动态并行只利用未饱和资源。

## 从简单方案演进

最简单正确基线是单请求流式上传到临时对象，完整摘要通过后 rename/manifest 发布。它适合文件小于重试预算。若失败重传字节连续一周超过有效上传的 5%，或单请求时长 `p95` 超过负载均衡器连接上限的一半，切到分块会话；新增会话状态、清理和块索引。百分比按流量成本与用户耐心校准。

当单流吞吐低于可用链路的 50% 且服务端尚有份额时启用有限并行；当 in-flight 内存或租户带宽达到预算 70%，自适应降低 `p`。第二组待压测指标是块大小从 8 MB 到 128 MB 时元数据 QPS、失败重传和内存的拐点，以及过期时间多长既覆盖离线恢复又不让暂存容量失控。全局固定块大小只适合文件和网络分布窄的场景。

未选“把每块当独立可读并由客户端猜完整性”，因为它泄露部分文件；若业务本来就是稀疏数据集且消费者接受显式分区状态，它重新变优。未选跨租户全局去重，因为摘要存在性和计时可泄露别人的文件；同信任域、加密与合规允许且存储节省显著时才考虑。

## 设计决定

客户端创建 session，得到冻结的 chunk plan 与短期上传凭据；可并行上传但每块都带 digest。服务端写临时块、校验长度/摘要后 ACK，客户端查询 bitmap 恢复。Complete 以 session generation 进入 VERIFYING，检查完整覆盖并原子提交 manifest；重复 Complete 返回同结果。逻辑文件名若需要更新，再用 expected pointer generation CAS 到新 manifest，旧版本保留供在途读者。

过载由低到高降级：减少后台跨区复制与预取；降低单会话并行度；缩短新会话最大 in-flight 并按租户排队；暂停超大低优先级会话；最后拒绝新 session，但继续允许已接近完成者提交 manifest。不能通过跳过摘要或提前发布来“提升可用性”。反选服务器单线程串行拼接所有块，因为它形成第二次 `B` 字节复制；若存储不支持 manifest/复合对象且文件较小，串行拼装可重新更简单。

## 运行与演进

SLI 包括有效吞吐、完成时延、重试字节比、chunk digest 失败、会话恢复率、complete 幂等命中、暂存/发布容量比、过期清理年龄、租户公平性与下载校验失败。两组校准阈值：重试比超过多少值得更小块；metadata QPS 和内存到何处需要更大块或分层索引。安全告警区分恶意随机块、超大声明和正常弱网恢复。

故障时间线：0 秒创建 generation 5，上传块 0/1；10 秒客户端断网；一小时后查询 bitmap 只补块 2；Complete 已提交 manifest M 但响应丢失；客户端重试同 operation id，返回 M 而不生成 M2。再让旧进程迟到提交块 1 的不同摘要，因 session 已 COMMITTED 且身份冲突拒绝。下载时切换逻辑指针到 generation 6，已开始读 M 的客户端仍完成 M，不混入新块。

协议迁移先让新服务同时接受旧单流和新 session，新客户端灰度启用；回滚仍能读已提交 manifest，绝不删除新格式数据。端到端加密时整体明文摘要由客户端声明，服务端至少校验密文字节与 Merkle 结构；恶意客户端风险由隔离和配额承担。删除先撤销逻辑指针与凭据，再回收 manifest 引用和块，报告控制接受与物理擦除的不同阶段。

## 面试考察本质

本题本质是：给定“发布文件必须是一个位置完整、摘要一致的不可变 generation”，因为网络中断后服务端与客户端对已接收字节的知识不同，而带宽、内存和暂存容量有限，候选人应推导带身份的会话、块级幂等与最终 manifest commit，并在块大小、并行、重传和公平间取舍。

优秀回答会算 `ceil(B/c)` 与 `p×c`，给出路径被覆盖后的错拼反例，区分 chunk ACK、Complete 和逻辑指针更新，处理响应丢失。常见误区是只说分片并行、用偏移代表身份、所有块成功便暴露目录、并行无限增大，或清理过期会话时误删共享块。追问可落到 Merkle、跨区耐久、客户端恶意声明和 TB 文件公平性。

二十分钟完成 API、会话状态、容量与原子发布；四十分钟加入并行调节、断点、清理和故障；六十分钟再谈加密、去重、多地域与协议迁移。考察的不是某云上传接口，而是能否说清一次 ACK 穿过了哪个故障域、什么证据证明整个文件成立，以及重试为何不会产生第二个逻辑成功。
