# 银行转账

## 表面题目

设计银行转账，表面上是付款人输入收款账户与金额，系统扣款并显示到账。真正的链路包含付款行借记、制裁和风险决定、清算报文受理、收款行贷记、跨行结算与可能退汇；这些状态由不同机构拥有，发生时间也不同。“已提交”“收款方可用”和“最终结算”不能共享一个完成标志。

本设计覆盖账户内与跨行贷记转账、端到端参考号、在途头寸、回执、对账和退回。支付意图幂等只是入口条件；跨行自动重放还要求清算网络与收款行共同承诺按稳定端到端 reference 去重并返回原结果。核心是每次借记最终能由贷记、退回或仍可解释的在途状态说明。具体清算网络格式和外汇交易不是主流程。

## 反问与边界

先确定同行还是跨行、实时还是批量、币种与地区、到账和最终不可撤销的合同、截点与节假日、手续费承担、限额、制裁筛查及收款账户校验。用户取消在借记前、报文发送后还是收款贷记后分别意味着拒绝、撤回请求或新退回。还要问端到端 reference 由谁生成、清算网络与收款行的重复识别窗口是否覆盖最大重放期，以及重复 reference 是否保证返回原受理或贷记结果。

容量要看指令率、批量代发峰值、金额分布、外部 ACK 未知率、在途年龄、日内流动性和对账吞吐。SLO 要拆成受理、借记、发送、收款可用和结算；高金额可能走不同人工路径。安全要求强认证、收款人变更保护、敏感账户脱敏和审计；过载不能跳过制裁检查。非目标是由本地数据库一次提交保证外部机构已经到账。

## 客观模型

最小命令为 `CreateTransfer(debtor, creditor, amount, key)`、`Authorize`、`Debit`、`Send(reference)`、`IngestAck`、`RecordCredit`、`Return` 与 `Reconcile`. 付款账户 owner 拥有可用余额与借记，转账编排拥有 instruction 和 reference，清算网络拥有报文受理与结算位置，收款行拥有受益人贷记，本地账本拥有在途和清算分录。

状态可为 CREATED、AUTHORIZED、DEBITED、SENT、ACCEPTED、CREDITED、SETTLED、RETURN_PENDING、RETURNED 与 UNKNOWN。对同一 transfer_id 至多借记一次；同一端到端 reference 在支持幂等的网络和收款行至多产生一次受理与一次受益人贷记，重复报文返回原结果；终态中借记对应受益人贷记或等额退回；任何 SENT 未终结项都保留在途科目。删除和覆盖会破坏资金路径。

若有效转账速率为 `λ`、平均终态时间为 `T`、平均金额为 `a`，期望在途金额约 `E[float]=λ×T×a`；具体头寸为所有“付款已借记且未收到终态”的 `Σ amount_i`。这既是流动性预算，也是对账队列的正确性容量。

## 必然约束

[DEDUCED:bank-transfer-debit-credit-and-interbank-settlement-have-different-finality] 付款行能原子借记自己的客户，却不能同一瞬间控制收款行余额和清算机构头寸。即使收款行先垫资使 B 可用，跨行结算仍可能稍后完成；反之报文受理也不代表 B 已可用。因此用户状态必须标注“已扣款、处理中、已到账或已结算”的层级。同行转账在同一账本内可以合并借贷，但不能把这个特例冒充所有通道。

[DEDUCED:bank-transfer-ambiguous-network-outcome-requires-reference-based-reconciliation] 跨行超时不证明对方未处理。最小时间线：10:00 付款行借记 A 100 元并以 R7 发送；10:01 收款行已贷记 B，但 ACK 丢失；10:02 付款行超时。用新 ID 重发可能让 B 再收 100，停止并标失败则与 A 的借记冲突。若清算网络和收款行的合同都保证在识别窗口内按 R7 幂等，outbox 可以原样重放 R7，对方必须返回第一次的 ACCEPTED/CREDITED 结果而不再次贷记；若任一环节没有这个合同，不确定发送就只能保留 R7 为 UNKNOWN 并查询、对账或转人工，不能自动重发，绝不凭超时猜结果。

[DEDUCED:bank-transfer-return-is-a-new-compensating-event-not-a-delete] 已借记或贷记的历史已影响余额与机构头寸，退回不能让它从日志消失。收款账户关闭时，应记录 B 侧退汇、清算反向与 A 侧入账，并引用原 R7；删除原指令会使审计无法解释中间在途和费用。只有尚未借记、未发送的草稿才可安全取消而不产生资金事件。

## 从简单方案演进

最简单正确基线是同行转账：在同一账户账本事务原子提交 A 借方与 B 贷方，transfer_id 唯一。扩展跨行时，先原子借记 A 与贷记“通道在途”科目，持久化稳定 reference，再由 outbox 发送；ACK、查询和批次对账推进状态。启用 outbox 自动重放前必须登记通道合同：网络转发和收款行都以 reference 为幂等键、重复报文返回原结果且识别窗口覆盖重放期。缺少任一保证时，outbox 只允许首次发送，结果不确定后转查询或人工队列。把网络调用塞进账户事务既会长锁，也不能回滚外部已处理报文。

第一个待校准指标是某通道“已发送但终态未知”连续十五分钟超过 0.1%。达到后停止该通道的自动重放，即使原 reference 理论上受保护，也优先扩大查询；超过三十分钟仍未知则进入优先对账并向客户显示在途。任何情况下都不生成新 ID。0.1% 与三十分钟需按通道合同校准，调低增加查询成本，调高扩大重放压力和客服风险。

第二个待校准指标是在途金额超过通道日内流动性额度的 60%，或在途 `p99` 年龄超过承诺到账时间的两倍。此时降低新转账限额、优先处理旧批次，合规允许时引导同行即时转账；60% 和两倍是风险缓冲起点。新增代价是拒绝合法流量，但继续放行可能耗尽结算头寸。

没有选择对所有 timeout 自动冲正，因为对方可能已经贷记，单边把 A 加回会创造资金。只有收到网络明确拒绝或可证明未受理的 fence，才可直接反向本地借记。

## 设计决定

本设计在付款账户版本上检查可用余额和授权，原子写借记、在途贷记、transfer_id 与 outbox。发送使用稳定端到端 reference；重复客户端请求返回原 transfer，参数不同拒绝。对支持幂等的通道，outbox 每次投递都携带同一 R7，网络和收款行命中 R7 后返回原处理结果，不创建第二笔受理或贷记；对不支持该合同的通道，outbox 首次发送后即封住自动重投，UNKNOWN 只能查询或人工处置。清算 ACK、收款确认、退回和对账都作为不可变事件推进单调状态，本地超时只写 UNKNOWN。

收款方可用与跨行 SETTLED 分开对用户和财务发布。在确认贷记后，将在途科目转成通道应收或结算科目；退回产生反向 journal，保留费用规则与原 reference。制裁或风险授权发生在发送前，但授权不等于借记，借记也不等于收款。通道故障时暂停高风险新指令，已在途继续查询。

反选方案是先向用户显示成功、日终批量净额纠错，它体验快却把拒绝和流动性风险隐藏到机构资产负债表。只有银行明确提供无条件到账担保并为失败垫资时，产品层才可提前承诺，账本仍需保留真实层级。

## 运行与演进

关键 SLI 包括重复借记和双贷、UNKNOWN 比例与年龄、在途金额、ACK 与收款延迟、对账差异、退回率、制裁队列和客户状态错误。过载时先降报表与普通通知，再限制低优先级批量；账户借记、外部回执、对账和退回不能被新指令饿死。

故障演练分两种通道合同：10:00 A 借记 100 且在途 journal 提交，R7 发送；10:01 对方贷记 B，ACK 丢失；10:02 编排重启，客户重复原键。支持 reference 幂等时，outbox 原样重放 R7，收款行返回第一次的 CREDITED，B 仍只加 100；不支持时，系统返回 R7 的 UNKNOWN，不再次发送，只查询或进人工。10:05 对账确认 CREDITED，单次消除在途并更新客户；随后迟到 ACK 只补充证据，不重复账本。演练必须注入“重复 R7”并在收款方断言仍是一笔贷记，而不只检查付款行未重复借记。

通道格式升级先双解析回执并影子核对 reference，再按少量批次灰度；回滚保留新字段。密钥和证书轮换覆盖在途最大生命周期。灾备切换前同步 outbox 与 reference 索引，用 epoch 禁止两站同时发送；日志脱敏账户并按法人和租户隔离流动性额度。

## 面试考察本质

这题考察的是：给定“同一指令不重复借记、稳定 reference 在收款侧不重复贷记、每笔借记最终有收款或可审计退回”的不变量，因为付款行无法同步知道收款行和清算网络走到哪一步，候选人应推导出本地原子借记、端到端幂等合同、无合同时禁止不确定重发、分层最终性与在途对账，并在即时体验、双贷风险和流动性占用之间取舍。

优秀回答会区分业务意图、授权、借记、报文、贷记、结算和退回，用 ACK 丢失时间线说明有端到端 reference 合同时如何原样重放、无合同时为何只能查询，并用在途科目证明资金去向。常见误区是 timeout 即失败、只让付款行去重却假设收款行不会双贷、自动单边冲正、删除退回交易，或把入口幂等键当成跨机构全局去重。

二十分钟应讲清本地账本与 reference；四十分钟加入未知状态、对账、退回和流动性公式；六十分钟再讨论清算截点、灾备 fencing、制裁与协议迁移。追问可要求在“B 已到账但 ACK 丢失”时逐项说明 A、B、在途和客户页面。
