# 分布式邮件服务

## 表面题目

设计分布式邮件服务，主流程是接受发件人提交的 envelope 与正文，按收件域解析路由，持久排队并通过 SMTP 转交，最终产生成功、延迟或退信状态。这里的“已发送”至少包含本服务接受保管、远端邮件服务器接管、邮件进入收件箱以及收件人阅读四层；本服务只能对自己承担的 custody 和观察到的协议响应作精确声明。

邮件不同于在线聊天：收件方可能数小时不可用，附件占据长期 spool，多个收件人按域拆成不同状态，远端反垃圾系统也不受发送方控制。`250 accepted` 是责任边界而不是用户可见性的证明，Message-ID 是稳定身份提示而不是跨互联网的恰好一次事务键。

## 反问与边界

先问服务是提交 API、完整 MTA 还是受控企业域内中继；本地返回接受前需要怎样的耐久性；单封大小、收件人数、投递期限、退信语义和优先级如何定义。容量按邮件率、收件人展开、平均/最大字节、域分布、重试次数和最长保留计算，还要确认单域并发、信誉限制、TLS 与跨境数据要求。

成功 SLO 应区分本地接受延迟、首次远端尝试时间、最终转交比例和队列最老年龄。安全边界包括防开放中继、发件身份验证、恶意附件隔离、退信反射防护和租户配额。非目标是保证远端收件箱排序、绕过垃圾过滤、证明阅读，或要求所有外部 MTA 按 Message-ID 去重。

## 客观模型

最小接口为 `Submit(envelope,body,idempotency_key)`、`GetRecipientStatus(message_id,recipient)` 和内部 `Attempt(domain,message_id,recipient)`。权威状态包括 `message -> {message_id,body_digest,envelope,custody_state}`、每收件人状态与尝试链；耐久正文/spool 的所有者在本地接受时承担 custody，远端最终 SMTP 响应标记一次责任转交。路由缓存与连接池不拥有投递事实。

不变量是：返回本地接受后，正文与 envelope 在可证明交接、可证明未交接的终态退信或明确的歧义处置完成前可恢复；同一提交幂等键不生成两份不同邮件；同一 `(message_id,recipient)` 的重试保持正文摘要；一个故障域的退避不阻塞健康域。只有从未出现“远端可能已接收”的尝试时，确定 5xx 才能形成 `DEFINITE_FAILURE`；一旦进入 `AMBIGUOUS_MAY_HAVE_DELIVERED`，后续 4xx/5xx 只能改变重试决策，不能倒推出首次尝试失败或生成声称“未送达”的虚假 DSN。

对域 `d`，积压近似 `Q_d(t+Δ)=max(0,Q_d(t)+(λ_d-μ_d)×Δ)`；第 `k` 次退避可为 `min(B_max,B_0×2^k)` 加抖动。总 spool 字节按 `Σ_d Q_d×avg_message_bytes` 预算，而非只看邮件数。少数大附件与单个大型收件域的限速会同时放大容量和尾部交付时间。

## 必然约束

[DEDUCED:distributed-email-service-acceptance-transfers-custody-not-recipient-visibility] 本服务只有在正文和每收件人责任已耐久记录后，才能诚实返回接受；此时可保证断连后继续尝试，并在证据充分时给出转交或失败结论，在响应歧义时则只能报告不确定。远端 `250` 只证明其 MTA 接管，后续过滤、收件箱展示和阅读均在另一权限域。结论是观察到 250 时 custody 可以精确迁移，用户可见性只能 best-effort 观察。

[DEDUCED:distributed-email-service-ambiguous-smtp-response-forces-at-least-once-transfer] 本服务向 `b.example` 发送 Message-ID `M`，远端已入队但 `250` 回包被断网吞掉；发送方必须立即把该收件人置为 `AMBIGUOUS_MAY_HAVE_DELIVERED`。五分钟后若不重传，可能静默丢信，若重传，远端又可能显示两封。即使第二次收到 5xx，它也只证明第二次被拒绝，不能证明第一次未被接管；策略可以停止继续重试，但状态仍须保留歧义，不能生成声称该收件人确定未收到的 DSN。Message-ID 可帮助远端去重，却不是其必须执行的事务约束。

[DEDUCED:distributed-email-service-retry-must-be-isolated-by-recipient-domain] 一个域的 4xx 临时失败表示该域当前接收能力或信誉边界，不代表其他域不可用。若所有收件人共用严格 FIFO，队首故障域会让健康域等待并占满 spool。按域隔离队列、并发和退避可将故障限制在责任边界；只有单一受控收件域才无需这层隔离。

## 从简单方案演进

最简单正确基线是一份耐久 spool 加顺序 worker：先落正文与收件人任务，再返回接受，按 MX 投递并记录响应。它适合受控小流量，却会被某个域的临时失败阻塞。当单域 4xx 比例超过待校准的 20%，且该域最老邮件超过 15 分钟，拆为域级队列、降低该域并发并扩大退避，健康域继续前进。

第二步把正文对象与收件人状态分离，共享正文摘要、按域分片任务。它降低多收件人复制，却新增对象引用、垃圾回收和部分终态。只有所有收件人终态且保留期结束后才能删除正文。当 spool 字节达到安全容量的 70%，或大附件占出站字节超过 60%，先拒绝尚未接受的低优先批量邮件，并对新大附件使用受控下载链接；绝不丢弃已接受 custody。

第三步引入域信誉、连接复用、TLS 策略和地域路由。它提高吞吐，却新增 DNS/MX 陈旧、证书失败和跨域策略差异。域队列年龄接近交付窗口一半时向发件人报告延迟，而非无限加并发。20%、15 分钟、70% 与 60% 都是待压测和运营校准的初始参数；调低保护 spool 但更早拒绝，调高提高利用率却缩小故障余量。

未选择提交请求同步等待远端 SMTP；只有单一受控域、低 RTT 且调用方必须即时获得远端拒绝时，它才重新变优。

## 设计决定

本设计在本地接受前原子写入正文摘要、envelope、提交幂等键和每收件人 `PENDING` 状态。调度器按收件域取任务，以稳定 Message-ID 尝试。状态机为：观察到 250 才进入 `TRANSFERRED`；在任何歧义尝试之前收到确定 5xx 才可进入 `DEFINITE_FAILURE` 并生成失败 DSN；4xx 保持可重试；正文可能已被接收而 250 丢失则进入 `AMBIGUOUS_MAY_HAVE_DELIVERED`。从该状态重试后收到 4xx/5xx，只能把 `retry_state` 设为延迟或 `STOPPED`，不得把 recipient 改写为确定失败；若运营策略不再重试，只能向发件人报告结果不确定，不能发送虚假未送达 DSN。

故障时间线是：0 秒本地落盘并接受；20 秒远端接收正文；21 秒 `250` 响应随断链丢失，recipient 立即成为 `AMBIGUOUS_MAY_HAVE_DELIVERED`；5 分钟后以同一 ID 重试并收到 550，这只让 `retry_state=STOPPED`，recipient 仍保持歧义，也不产生“未送达”DSN。可精确保证本地 custody 记录与完整尝试链可恢复；不能确定远端是否已有一封、何时显示、排序或阅读。

未采用把全部域放在一个全局队列的方案，因为故障域会造成队首阻塞。若服务只转发到一个内部网关，且该网关提供事务幂等接口，单队列才可能足够。

## 运行与演进

核心 SLI 是本地接受持久化成功率、首次尝试延迟、按域队列年龄、4xx/5xx 比例、`AMBIGUOUS_MAY_HAVE_DELIVERED` 数量与年龄、可证明转交/失败/不确定结局比例、spool 字节和正文引用泄漏。按租户、收件域、大小与优先级分桶，不能把歧义并入失败率，也不能用健康域平均值掩盖某域积压。过载时先暂停新批量与大附件，再限制低信誉租户；已接受邮件保留投递预算。

演练让远端在接收后丢弃 250，再让同一收件人的重试返回 550，验证 recipient 仍为 `AMBIGUOUS_MAY_HAVE_DELIVERED`、只停止重试且不生成失败 DSN；另一个域持续 4xx 时应独立退避，健康域不受阻。恢复按域逐步放大并发，避免同时重放形成重试风暴。正文格式迁移采用双读、先新写小流量、校验摘要后放量；回滚不得遗失新格式中已接受的 custody。

地域选择同时考虑发件数据驻留、到收件域路径与信誉；凭据、正文和投递日志分级加密，退信不回显敏感正文。租户配额在接受前执行，接受后不能因账单或容量变化静默丢弃。成本按长期 spool 字节、附件出口和每域连接重试分别归因。

## 面试考察本质

给定“本服务一旦接受就必须可恢复地承担每个收件人的投递责任，且可能已交接的事实不能被后续失败改写”的不变量，因为远端响应可能在其接管后丢失，发送方无法同时避免静默丢信与重复远端邮件，候选人应推导出 custody、`AMBIGUOUS_MAY_HAVE_DELIVERED` 的单向状态、至少一次交接和每域队列；再依据 spool、交付窗口与域信誉决定继续重试或停止，但不能伪造确定失败。

优秀信号包括区分本地接受与远端 `250`、按字节而非邮件数规划、说明 Message-ID 的有限作用、隔离故障域，并禁止丢弃已接受邮件。常见误区是把邮件当即时 socket、全局 FIFO、无限重试，或声称远端接受等于收件人阅读。

二十分钟回答覆盖接受边界、spool、收件人状态机；四十分钟加入域级退避、歧义响应和退信；六十分钟再讨论信誉、附件、跨地域与格式迁移。追问始终要求指出当前 custody 在谁手中、断链后知道什么，以及哪两个指标触发限流。
