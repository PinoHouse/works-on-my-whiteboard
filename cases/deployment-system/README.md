# 部署系统

## 表面题目

设计部署系统，表面上是把一个新制品推广到测试、区域和生产副本，并在失败时回滚。真正的状态变化是某个服务的目标 rollout generation 从旧版本切到新版本，控制器依据延迟且有噪声的健康观测逐批扩大暴露；成功意味着目标批次和流量都收敛，旧控制器不能在回滚后再次推进坏版本。

题名掩盖了“进程存活”“实例 ready”“业务健康”和“发布安全”并不相同。设计覆盖制品不可变引用、批次、切流、观察窗、暂停、回滚和审计；容器放置由编排器完成，应用数据迁移必须作为显式兼容协议处理，而不是假设二进制回滚能撤销。

## 反问与边界

先问部署对象是无状态服务、有状态集群还是客户端；可接受最大错误暴露量、发布总时长、可用容量下降和区域故障是多少。健康信号包括探针、错误率、延迟、业务转化和人工确认中的哪些；低流量服务如何取得统计置信度；告警缺数据算健康、失败还是未知。还要确定 canary 百分比、并发区域数、回滚触发者和审批边界。

期望状态是服务、环境、制品摘要、配置版本、目标副本与 rollout policy；观测状态是每个实例版本、流量权重、健康窗口和编排结果。每个服务/环境只有当前 rollout controller generation 能发 placement 与 routing 命令；健康评估器拥有窗口结果，但不能独立改变目标。外部负载均衡和编排器必须校验 rollout epoch。

SLO 是从批准到安全完成、暂停检测时间、回滚到错误暴露停止的时间和可用容量。非目标是在任意 schema 破坏下“一键回滚”，也不把一次探针 200 当业务正确。所有后文百分比均为待压测或业务校准策略。

## 客观模型

实体为 `Release(artifact_digest, config_version)`、`Rollout(generation, stages, state)`、`Target(replica_set, traffic_weight)` 和 `HealthWindow(signal, interval, verdict)`。状态 `PLANNED -> PROGRESSING(stage) -> PAUSED -> COMPLETED`，任意阶段可进入 `ROLLING_BACK(new_generation)`；回滚是新 generation，不是让旧控制器倒着执行。命令携带 `{service, env, rollout_generation, step}`。

不变量是只有当前 generation 能改变目标副本和流量；流量不能超过当前 ready 容量的安全界；promotion 必须引用固定健康窗口与策略版本；制品摘要不可变；旧版本能读取新版本可能写出的兼容数据。若每批暴露比例 `b`、错误概率增量 `e`、观察窗请求量 `R`，预期额外错误约 `b*R*e`；批次越小风险面越小，但得到足够样本的时间约与 `1/b` 增长。

反例：G8 canary 健康评估超时，操作者创建 G9 回滚。旧 controller 恢复后收到迟到“健康”并把 G8 从 10% 推到 50%。若路由端只看服务名，回滚被反转；路由和编排命令都必须拒绝低 generation。旧 worker 的日志可审计，但其推进结果无效。

## 必然约束

[DEDUCED:deployment-system-progress-must-bind-rollout-generation] 扩容、切流和回滚都必须绑定当前 rollout generation，否则迟到控制器会把已经回滚的坏版本再次推进。数据库里的 rollout status 不能 fence 已经排队到外部路由器的旧命令；下游目标也要比较 epoch。最小反例正是 G8/G9 交错。若部署完全串行且单进程故障后所有旧命令可被物理撤销，边界才可简化。

[DEDUCED:deployment-system-health-is-a-delayed-noisy-observation] 探针成功只能证明局部瞬时存活，部署推进必须在业务信号的有界观测窗中权衡检测速度、误报与故障暴露量。低流量 canary 五分钟只有十个请求，零错误不能证明万分之一目标；等待更久增加发布时长，放大流量增加潜在伤害。系统只能按风险选择窗口和先验，不能从缺少错误推出绝对安全。

[DEDUCED:deployment-system-rollback-does-not-reverse-incompatible-state] 二进制回滚无法自动撤销新版本已写的数据、消息或外部副作用，协议演进必须先保持前后兼容。若新版本把字段从文本不可逆改成整数并删除原值，旧版本启动后无法读；即使实例全换回，数据事实未回滚。因此采用 expand/migrate/contract 或向前修复，发布系统只协调代码与流量。

## 从简单方案演进

最简单正确基线是单环境、串行替换一台副本，确认 ready 后再替换下一台，并保持最小可用容量。它适合小集群且错误易察觉。第一个待演练切换指标是一次全量发布的预期额外错误超过错误预算的百分之五，或回滚检测时间超过十分钟；此时引入 1%/10%/50% 分批和业务健康窗，新增样本不足、版本并存和路由复杂度。

第二个待校准指标是发布等待时间超过两小时且百分之八十停留在低流量取样，或 canary 请求少于判定所需样本的四分之一；此时使用合成流量、历史基线或按实例而非流量的小批次，但不能伪造统计确定性。另一个切换点是单区域事故会消耗全局预算的百分之三十时改为区域串行；调低阈值更安全但拉长全球发布。

未选择默认蓝绿一次切全流，因为容量翻倍且切换瞬间暴露大；若服务副本少、状态外置、预热可验证且回切极快，它重新变优。未选择所有区域并行，因为相关制品缺陷会同时放大；当变更已在等价环境长期运行且紧急安全修复要求速度时可临时采用。

## 设计决定

发布固定 artifact digest、配置和策略，创建单调 rollout generation。controller 反复对比 desired target 与编排/路由 observed state，先创建无流量 canary、验证 startup，再以阶段目标切流；每步记录开始时间、样本与 verdict。promotion 是匹配 generation 和前一步的条件写。健康未知默认暂停，高风险信号触发新 generation 回滚。

旧 controller 租约过期后返回，其健康报告作为历史样本存储，但任何 scale、route、complete 命令因 generation 过期被编排器和路由器拒绝。回滚先停止增加暴露、把新请求切回已知良好版本，再回收坏副本；已进入请求允许有界排空。若路由响应丢失，按 generation 查询实际权重，不能盲目增量重试。

反选是仅用实例探针自动推进，简单且快速；当应用无关键业务语义、流量足够、探针能覆盖主要故障时可重新变优。生产默认还需要错误率和延迟，并把缺数据明确定义为 unknown。

## 运行与演进

SLI 包括每 generation 的 stage age、版本流量分布、desired/observed 副本差、health verdict 延迟、错误预算消耗、旧 epoch 命令拒绝、回滚启动与完成时间、容量余量和版本共存时长。过载时暂停新发布，优先完成正在回滚和安全修复；不能因控制器队列拥塞让旧坏版本继续扩张。

演练：T0 G20 在 10% 流量；T1 controller C1 失联；T2 错误率越阈，C2 创建 G21 回滚并切到 0%；T3 C1 恢复，拿到 T0 窗口的迟到 green，尝试把 G20 切至 50%。路由必须拒绝 G20，编排器不新增副本，审计显示 stale。另一轮让路由更新成功但响应丢失，reconciler 查询 observed 10% 后不得再加十个百分点。

待演练指标一是 canary 错误预算 burn rate 超过基线四倍并持续两个窗口时自动回滚；二是 desired/observed 差超过副本总数百分之十或持续五分钟时暂停 promotion。四倍、两个窗口、10% 都须按请求量和故障成本校准。协议升级先 expand，双读写灰度，再 contract；回滚窗口内禁止删除旧格式。密钥与配置版本随 release 记录，审计可重放每次决策。

## 面试考察本质

给定“只有当前 rollout generation 能扩大故障暴露，且旧版本必须在回滚窗口内仍能解释系统状态”这一不变量，因为健康是延迟、低样本且有噪声的观测，候选人应推导出 generation fencing、分批暴露、观测窗口和兼容迁移，并依据错误预算交换发布速度、容量和置信度。

优秀回答会区分 desired target、observed instances、traffic 与 health，画出旧控制器在回滚后恢复的时间线，并说明为什么 rollback 不是数据时间机器。常见误区是 readiness 通过就全量、把暂停当失败、只在数据库锁 controller，却让路由器接受旧命令。

二十分钟完成 rollout 状态和批次；四十分钟加入业务健康、路由 fencing 与回滚；六十分钟讨论低流量统计、区域顺序、schema 演进和审计。本题独特本质是用有限故障暴露换取对新版本的可观察证据，而非高效地“把包发到机器”。
