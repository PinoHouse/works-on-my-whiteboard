# 社交信息流

## 表面题目

设计社交信息流，表面流程是用户发布帖子，关注者打开首页后看到若干条按时间或相关性排列的内容。真正的状态变化分成两类：内容库确认帖子、可见性与删除版本，关系图确认关注、取关和拉黑；首页只是从这些事实产生的候选 ID 与派生排序。发布成功不等于每个关注者已经收到副本，首页返回成功也不等于其中候选永远可见。

题名掩盖的关键歧义是“顺序”含义。同一作者可以用 `publish_seq` 给出权威发布序，跨作者首页却没有一条全局真相；相关性分、时间衰减和去重都只能生成可重算 rank。本设计聚焦 home feed，不把搜索、私信、广告竞价或通知送达混入同一正确性边界。

## 反问与边界

先问可见性承诺：删除、取关、拉黑和私密切换后多久必须停止展示，允许首页短暂缺项还是允许越权；同一会话是否要求读己之写，分页是否接受候选在两页之间移动。再问新鲜度与延迟：发帖到合格关注者可见的 `p95/p99`、首页生成 `p99`、每页数量、刷新频率与冷启动降级。吞吐必须按作者关注数分布和用户关注数分布表达，平均值会同时隐藏名人单帖和超级读者读时合并。

容量变量包括作者 `a` 的有效关注者数 `F_a`、发帖率 `P_a`、读者关注源数 `n`、每源预取数 `k` 和活跃读者比例。还需确认多地域关系边的写权限、机器人发布限制、租户配额、帖子与互动保留期限。安全上，候选缓存不能充当授权；日志只记录内容 ID、版本和排名原因摘要。非目标是审计级全局时间线、保证每次刷新完全稳定，以及用首页投递状态证明用户已读。

## 客观模型

最小命令为 `Publish(author, body_ref, visibility, idempotency_key)`、`ChangeVisibility(post, expected_version)`、`Follow(viewer, author, expected_version)` 与 `GetHome(viewer, cursor)`。内容库拥有 `post -> (author,publish_seq,visibility_version,state,created_at)`；关系图拥有 `(viewer,author) -> (state,edge_version)`；首页服务只拥有带来源版本的候选集合、排序特征和游标。作者发布序与各权威记录版本可比较，跨作者 home rank 是派生结果。

不变量是读取时帖子仍存在且对读者可见；版本较旧的投递不得覆盖删除墓碑；取关或拉黑后旧 inbox 条目不能重新授予访问；分页游标只能声明某次候选快照，不能伪装成全局发布位置。写时物化把 `(viewer,post,visibility_version)` 放入 inbox，读时物化则从关注源近况拉取并合并，混合方案对普通作者预推、对名人读时取回。

作者一帖的写时工作量近似 `W_a=F_a*(1-c_a)`，其中 `c_a` 是无需持久预推的安全比例；读时从 `n` 个源各取 `k` 条时，合并前候选上界为 `C=n*k`，并非最终展示的 `k` 条。偏斜决定成本位置：名人使写队列集中，关注数极多的读者使读放大集中，热门帖的媒体和互动查询又形成次生热点。

## 必然约束

[DEDUCED:social-news-feed-home-order-is-derived-not-a-global-publish-log] 内容库只为每个作者分配发布序，排名器在读取时才掌握某个读者的候选、特征与策略版本，因此无法从两个作者的本地序推出审计级全序。最小反例是作者甲的序号 8 在 10:00 提交，作者乙的序号 3 在 09:59 提交但 10:01 才跨区到达；相关性模型又把乙排在甲前。首页可以合理返回乙、甲，也可在模型升级后反转，但不能据此声称谁先全局发布。需要审计的事件应读取各自权威日志，首页顺序只是一份带版本的读模型。

[DEDUCED:social-news-feed-celebrity-skew-makes-unconditional-write-fanout-unbounded] 若每次发布同步写完所有关注者 inbox 才确认，单帖工作量直接由 `F_a` 决定。最小反例是作者 A 有一千万关注者，投递池每秒处理十万条：一帖至少需要一百秒队列时间；增加普通作者平均吞吐不能缩短该下界。若 A 两秒后删除，删除清理还会与剩余九百八十万条旧投递竞争。由此名人必须采用读时合并、仅预推活跃用户或有界异步投递；只有关注数有硬上限且写入池按峰值预留时，全量写扇出才是有界选择。

[DEDUCED:social-news-feed-visibility-must-be-checked-after-materialized-candidate-delivery] inbox 中的候选只证明某个旧时刻有投递资格，不掌握读取时的删除、拉黑和关系变化。具体序列是 0 秒帖子 P 以可见版本 8 发布，1 秒写入部分 `(viewer,P,8)`，2 秒作者删除并提交墓碑 9，3 秒队列重复旧版本 8，4 秒用户刷新。若首页只信 inbox，已投递用户看到已撤回内容；若先向权威查询版本 9则隐藏 P，迟到版本 8也只能成为可清理垃圾。因而派生候选永远不能扩大权威可见性。

## 从简单方案演进

最简单正确基线是读时合并：查询关注关系，对每个作者取最近帖子，读取时复核可见性，再按明确策略排序。它没有投递一致性问题，却在 `n*k` 候选过大时拉高首页尾延迟。普通用户增多后，可为活跃读者预物化最近窗口，减少首页读放大；新增成本是 inbox 存储、重复任务、关系变更清理和候选陈旧。

第一个待压测校准指标是某作者单帖 `F_a` 超过写入池一分钟安全容量的 `20%`，或其 fan-out 最老任务年龄超过首页新鲜度 SLO 的一半：该作者切为读时合并或只向近期活跃关注者预推。第二个待回放校准指标是读时合并 `p99` 超过 `250 ms`，或候选不足请求比例超过 `2%`：扩大普通作者预推窗口或预计算候选，但保留展示前可见性复核。`20%/半个 SLO` 与 `250 ms/2%` 都是初始策略参数，不是生产测量；调低会更早支付混合路径复杂度，调高会扩大队列或读延迟风险。

再扩展时按 viewer 分片 inbox、按 author 分片发布与近期索引，排名可离线准备稳定特征、在线拼入关系与新鲜度。切换动作必须以作者偏斜和读者偏斜分别执行，不能把所有流量都改成同一种物化。模型升级只重算派生 rank，不改 `publish_seq`、可见性版本或关系所有权。

## 设计决定

本设计采用混合物化：普通作者异步预推候选 ID，名人和积压作者在读时并入；每次读取用关系图和内容库当前版本执行硬过滤，再做排序与分页。发布确认发生在内容库提交后，不等待全量 fan-out。投递以 `(viewer,post,visibility_version)` 幂等，删除墓碑可抢占普通投递优先级；超时仅导致候选缺失或稍晚，不允许绕过权限返回。

缓存保存候选与特征，不保存最终授权。权威内容或关系服务不可达、无法确认当前删除/拉黑/隐私版本时一律 fail-closed：隐藏相关候选，候选不足则返回较少结果或明确的暂时错误，不能用旧的“公开”状态降级展示。公开内容若要允许短暂陈旧，必须另有不受删除与权限变化约束的产品类别；它不属于本题的用户内容可见性契约。游标绑定候选快照与 ranking version，模型切换不重写旧页。

反选“所有作者一律同步写时全量扇出”，它能降低稳定小图上的读延迟，却让名人单帖决定确认延迟与队列容量；只有关注数有严格上限、删除极少且首页必须极低延迟时重新变优。也不选择纯读时方案作为高关注读者默认路径，因为它把每次刷新都放大成大量源读取。

## 运行与演进

核心 SLI 是发帖到合格读者可见的延迟、首页生成 `p99`、候选不足率、按作者 fan-out backlog、读取时被可见性过滤比例、删除后曝光数和旧版本投递拒绝数。过载时先丢弃低价值特征与深页预计算，再缩小普通作者预推窗口，最后对冷读者返回较少候选；删除、拉黑与私密检查永不降级成软分。

故障演练固定时间线：12:00:00 P/v8 发布，12:00:01 工作者甲投递二十万条，12:00:02 墓碑 v9 提交，12:00:03 队列恢复并重放 v8，12:00:04 两个区域刷新。读到 v9 的区域隐藏 P；尚不能权威确认当前版本的区域同样隐藏 P 而不是回退到 v8，旧任务以版本不匹配拒绝。恢复作业按 post 与版本批量清理，不能让“最终删除”替代已读路径的权限边界。演练记录用户可见结果、fail-closed 候选不足、墓碑传播年龄与队列公平性。

排名版本先影子计算候选差异，再按租户灰度；回滚只切 ranking version，不回滚权威事实。跨地域迁移关系边时保留单一写 owner 和单调 edge version。隐私删除覆盖正文、候选、特征、调试样本和缓存，热门作者不得占尽所有工作者，队列按作者设置配额和老化优先级。

## 面试考察本质

给定“任何刷新都不能越过当前帖子与关系可见性”这一不变量，因为读者节点既不知道所有作者的完整全局顺序，也不能立即获知删除和拉黑，候选人应推导出权威内容/关系与派生首页的分层，并在名人写放大与超级读者读放大之间选择混合物化。业务最终依据越权损失、新鲜度和尾延迟决定缺项、陈旧与成本的取舍。

优秀回答会明确作者序是权威而 home rank 是派生，写出 `F_a` 和 `n*k` 两种放大，用一千万关注者删除事件说明候选必须复核，并把名人从普通扇出路径隔离。常见误区是把消息队列当成权限真相、为稳定分页制造全局时钟，或认为缓存失效一定及时。

二十分钟完成状态 owner、不变量和读写物化选择；四十分钟加入偏斜公式、混合路径、幂等投递与删除时间线；六十分钟再讨论模型版本、分页、跨地域关系、隐私删除和队列公平。追问应持续要求说明某个版本由谁提交、哪个顺序可审计、哪个 rank 可丢弃重算。
