# 问答与新闻聚合

## 表面题目

设计问答与新闻聚合，表面上是接收问题、回答、新闻链接和投票，再给用户展示热门内容。真正的状态变化包括来源修订、题主接受回答、版主审核、投票关系和可撤销的去重聚类；首页热度只是这些事实上的派生 rank。成功不能只定义成内容入库，还要保证来源可追溯、接受裁决不被票数覆盖、撤稿或拆分后旧摘要不会继续冒充权威事实。

题名同时包含两种顺序：问答中的 accepted answer 是有写权限的状态，新闻的事件时间是来源声称的事实；票数、抓取时间和相关性分只是展示信号。本设计不承诺判断新闻绝对真伪，也不把搜索索引、推荐画像或外部网页存档混入核心。

## 反问与边界

先问谁能投稿、编辑、接受回答和撤销，版主裁决多久必须生效，来源撤稿后允许保留历史记录还是必须隐藏正文。再问首页新鲜度、聚类准确率、举报处理时延、分页稳定性与热点流量。新闻发布时间是否可信、页面更新是否形成新 revision、同一 URL 多语言版本是否合并，都决定身份模型。

容量按抓取源数、每源修订率、候选相似比较数、投票与评论热点以及人工审核队列规划，不能只按首页 QPS。安全上保留来源 URL、抓取摘要和决策审计但避免无限复制受限正文；投票要限速、防刷并按 actor 去重。非目标是用多数票取代题主或版主权限，以及把文本相似阈值包装成无误差的全局主键。

## 客观模型

最小命令为 `SubmitQuestion`、`SubmitAnswer(question, revision)`、`AcceptAnswer(question, answer, expected_version)`、`IngestSource(url, source_time, digest)`、`Moderate(target, decision)` 与 `Vote(actor,target)`. 问答 owner/版主拥有 accepted 与审核状态；来源仓拥有每个 revision 及撤稿标记；聚类服务拥有可撤销的 `(member,cluster,generation,reason)` 边；排名器只拥有带模型版本的派生分。

不变量是同一问题版本只有一个获授权 accepted answer；不同来源和修订不得因聚类而丢失身份；撤稿、封禁和拆分优先于旧缓存；`source_time` 与 `ingest_time` 分别保存；热度不能赋予编辑或接受权限。问题内回答的接受状态权威，首页与 cluster 内顺序派生。

可用的展示分可写成 `score=α*freshness(source_time)+β*source_quality+γ*engagement-duplicate_penalty`，但它不是事实顺序。若自动聚类 precision 为 `p`、每日合并 `N` 对，则预期误合并约为 `(1-p)*N`；热点事件还使每项需要比较的候选数和人工复核同时上升。

## 必然约束

[DEDUCED:qa-news-aggregation-accepted-answer-is-authoritative-while-hotness-is-derived] 接受回答需要题主或版主写权限，而热度服务只掌握投票、时效与模型特征。最小反例是回答 A 获题主接受但 10 票，回答 B 有 1000 票；若排名器因 B 分高而把 accepted 状态改给 B，它越过了授权裁决。可以让 B 展示在前，也可提示社区推荐，但 accepted 标记只能由权威状态转换。若产品定义“最高票自动接受”，那是另一份明确合同，仍需在同票、刷票与撤票时给唯一规则。

[DEDUCED:qa-news-aggregation-canonicalization-is-a-fallible-decision-not-a-lossless-key] URL 规范化、标题相似或向量接近都只是证据，不能证明两个版本描述同一事实。反例是 10:00 来源 X 报“列车延误”，10:05 来源 Y 报“列车恢复”，标题高度相似；系统若合并且 10:06 X 撤稿，删除 cluster 会误删 Y，保留 X 摘要又错误。聚类必须保留成员版本、证据和 generation，允许版主拆分。只有来源提供稳定且可信的共同事件 ID 时，才能把它当更强身份键。

[DEDUCED:qa-news-aggregation-event-time-and-ingest-time-must-remain-separate] 聚合器知道何时抓到页面，却无法仅凭到达时间知道事件发生时间；来源时间又可能迟到或被修改。最小反例是昨日 08:00 的旧报道因网络故障今天 12:00 才抓取，若按 ingest 排名会伪装成刚发生；若盲信被篡改成未来的 source time，又会永久置顶。系统应同时保存两者、限制异常来源时间并向读者标示，rank 可选择组合但不能覆盖原始时间谱系。

## 从简单方案演进

最简单正确基线是每个提交和来源 revision 独立存储，按抓取时间列出，不自动合并；问答接受和审核走条件更新。它可追溯但重复多、发现体验差。候选量增长后加入相似关联，先把模型结果当“可能相关”，达到足够证据才生成可撤销 cluster；新增误合并、拆分回滚、缓存 generation 与人工复核。

第一个待回放校准指标是人工确认的错误合并率连续一周超过 `0.5%`，或撤稿后错误摘要曝光超过 `0.05%`：提高自动合并门槛，把跨来源聚类降为候选关联并增加复核。第二个待运营校准指标是未审核投稿或举报的最老年龄超过 `15 min`，且高风险主题举报率超过基线 `2 倍`：限制该主题新内容进入公共 rank，只展示已验证来源或占位。两组值不是生产测量；调低会牺牲召回和发布速度，调高会扩大误导与审核风险。

读量增加时缓存 cluster 视图与热榜，但条目携带审核版本和 generation；写时只物化来源、投票关系与聚类边，热度按周期离线聚合并在读取时拼入新鲜度。人工裁决始终优先，不因模型升级重写来源历史。

## 设计决定

本设计保存每个来源 revision 和两种时间，规范 URL 只做候选查找；聚类器生成带置信原因的可撤销边，版主拆分推进 generation。问答 accepted 与审核在权威库条件提交。首页读取先过滤撤稿和封禁，再读取当前 cluster generation，最后计算派生 rank；缓存不匹配即重取。

抓取、投票和聚类任务以来源摘要或 actor-target 幂等；超时重试不创建第二个 revision 或重复票。审核服务不可用时高风险新内容 fail-closed，低风险已审核内容可用旧 rank 展示。反选“只按单一全局热度自动聚合”，在来源高度可信、主题低风险且产品只追求发现时可能重新变优，但仍不能抹掉来源身份。

也未选择对相似页面永久合并存储，因为它让一次误判不可逆，并破坏撤稿和多样性。付出成员级历史与 generation 的存储，换取可解释拆分和旧缓存 fencing。

## 运行与演进

SLI 包括来源抓取与 revision 延迟、误合并/误拆分反馈率、撤稿后曝光、举报最老年龄、accepted 冲突、cluster generation 缓存命中和热点主题审核占用。过载时先降低低价值相似比较和个性化特征，再限制未经验证来源进入公共 rank；审核、撤稿和来源谱系优先。

故障演练：0 秒 X revision 2 加入 C/g7，30 秒模型把 Y revision 1 加入，60 秒版主拆分并封禁 X，generation 推进到 8，90 秒边缘仍缓存 g7。读路径发现 generation 不符并重取，只展示 Y；回放保留加入、拆分与撤稿事件，不能用当前摘要覆盖历史。演练检查两个地区均不会把 X 的旧文本附着到 Y。

模型和规范化规则升级先影子输出边差异，再灰度新 generation；回滚停止新决策但继续理解已写边。跨地域让来源历史单主写、排名多地重建。删除、版权与隐私请求按来源成员执行，不能粗暴删除整个 cluster；租户和版主操作保留审计。

## 面试考察本质

给定“来源、修订和有权限的社区裁决必须可追溯且不能被热门度覆盖”这一不变量，因为聚合器无法仅凭相似文本知道两个页面是否为同一事实，也无法把抓取时刻当事件时刻，候选人应推导出权威状态与可撤销聚类、派生热度的分层，并在召回、新鲜度、审核时延与误合并之间取舍。

优秀回答会区分 accepted 与 rank、保存 source/ingest time，用“延误/恢复”最小反例证明聚类边可撤销，并量化 `(1-p)*N`。常见误区是认为 canonical URL 无损、用票数替代权限、或删除一个来源时整簇消失。

二十分钟完成状态 owner、revision 和接受不变量；四十分钟加入聚类误差、审核、缓存 generation 与故障线；六十分钟再讨论来源信誉、跨地域、版权删除和模型迁移。追问应让候选人逐项说明哪个事实可审计、哪个顺序只是展示。
