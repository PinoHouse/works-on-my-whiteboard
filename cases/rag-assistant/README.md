# RAG 助手

## 表面题目

设计一个检索增强生成助手：用户提出问题，系统从企业文档中找出相关片段，把证据交给生成模型，再返回带引用的答案。一次成功不是“模型吐出了一段通顺文字”，而是查询时有权访问的资料被及时检索，答案中的关键断言能回到原文的确定版本，并在权限撤销后不再继续披露旧证据。品牌化的“知识库问答”掩盖了三个不同结果：检索找没找到、模型是否忠实使用、引用是否真的支持断言。

主流程包括文档接入、切块与嵌入、索引发布、授权过滤、候选重排、上下文组装、生成与引用映射。这里不设计基础向量数据库和通用模型托管，也不承诺回答必然为真；题目关注派生知识在变化文档与当前权限下如何可追踪地进入概率输出。

## 反问与边界

先问语料属于公开网页、单租户知识库还是细粒度企业权限；ACL 是文档级、段落级还是行级，撤权后允许多长披露窗口。质量要同时定义检索 `recall@k`、引用覆盖率、引用蕴含率、无依据回答率和任务成功率，体验则区分检索 `p95`、首令牌 `p95` 与完整答案延迟。再问语料规模、更新与删除速率、查询峰值、语言分布、长文偏斜、上下文上限和每次回答的令牌成本预算。

成功语义需说明“找不到”能否明确拒答，是否允许使用模型参数中的常识，引用要精确到文档、版本、页码还是字符区间。权限以查询开始、检索结束还是发布答案时为准；本设计选择发布前当前授权为披露边界。多地域是否复制原文，日志可否保留检索片段，删除是否包含缓存、评测样本与对话历史，也必须写入契约。非目标是训练模型、保证确定性生成或把高引用数量当作事实正确。

## 客观模型

最小接口为 `Ask(principal, question, conversation_version, policy_epoch, request_id)`、`GetAnswer(attempt_id)` 与 `Revoke(document_id, acl_epoch)`。文档版本保存 `(document_id, source_revision, content_digest, acl_id, acl_epoch)`；chunk 保存 `(chunk_id, source_revision, byte_range, chunker_version, embedding_model_digest, vector_digest)`；索引 manifest 保存 `(index_generation, segment_digests, embedding_space, build_watermark)`；answer attempt 保存查询、授权快照、候选 chunk、重排器、提示模板、生成模型、安全策略、采样配置与引用映射的联合谱系。

设文档总字符量为 `D`，平均块长 `c`、重叠 `o`，近似 chunk 数为 `N≈D/(c-o)`；嵌入维度 `d`、每维 `b` 字节时，仅原始向量约占 `N*d*b`，ANN 图、元数据和副本会再乘放大系数。一次请求成本可写为 `C = C_embed(query)+C_search(N,ef)+C_rerank(k)+C_generate(Tin,Tout)`。长文重叠过大同时放大存储和提示令牌，相同文档的高频查询又使热门 ACL 与热门片段形成缓存偏斜。

状态所有者分别是源系统的文档 revision、权限服务的当前 acl_epoch、索引目录的可服务 generation 和回答记录的 provenance。关键不变量是：任何引用都能映射到未篡改的源 revision 与精确区间；查询向量和文档向量属于兼容空间；发布时每个进入提示的私有 chunk 仍获当前授权；一次回答只绑定一个完整索引 generation。确定的流水线与版本记录能复现“当时看见了什么”，却不能令概率模型在不同批次、内核或采样路径上逐字确定。

## 必然约束

[DEDUCED:rag-assistant-grounding-requires-versioned-lineage] 可追溯回答必须绑定文档版本、切块器、嵌入模型、索引代际、提示模板与生成模型，只有最终文本无法证明依据来自哪里。最小反例是文档 v7 写“退款期 30 天”，回答生成后源文档更新为 v8 的 14 天；若引用只保存 URL，审计时看到 v8 会错误判断答案从未有依据。保存 source_revision 与 byte range 可以重建证据，保存模型和提示版本用于解释为何相同证据产生不同表述，但仍不等于证明断言正确。

[DEDUCED:rag-assistant-current-authorization-must-fence-generation] 检索时通过权限检查不代表稍后仍可披露；回答发布前必须以当前授权世代重新约束证据与输出。事件序列是零毫秒检索到机密 chunk，二十毫秒管理员撤销用户权限并推进 acl_epoch，五十毫秒模型开始流出该片段。若只信检索快照，撤权后仍泄露；因此私有证据在发送首个相关 token 前必须通过当前世代检查，长答案还需短期发布租约或可撤销缓冲。无法保证全局瞬时停止时，契约应明确最大暴露窗，而不能把缓存 TTL 叫作安全边界。

[DEDUCED:rag-assistant-retrieval-quality-is-a-budgeted-recall-problem] 在有界检索工作量下扩大候选、重排和上下文会提高召回机会，却同时增加延迟、令牌成本与无关证据干扰。若真正相关片段排在 ANN 第 21，而候选预算为 20，后续再强的生成器也看不到它；把预算升到 200 可能找回，却使重排和上下文成本约增十倍，并可能把相互冲突的旧版本带入提示。质量、延迟和成本必须以同一查询分布联合评估。

## 从简单方案演进

最简单正确基线是小规模文档逐篇全文扫描，先用权限服务过滤，再把匹配段落与 source revision 交给模型。它慢但语义清楚，可作为离线评测真值近似。当语料扫描 `p95` 达到问答延迟预算的百分之三十，或每天全文计算量超过单机可用 CPU 小时，才引入倒排/向量候选与 immutable index generation。索引降低在线工作量，却新增异步新鲜度、近似漏召回和删除传播。

第二步将候选检索与交叉编码重排分离，并按 ACL/时间/文档类型规划过滤。若固定评测集 `recall@20` 低于目标两个百分点以上，先增加候选或改进切块；若重排 `p95` 超过二百毫秒，则按查询类别缩减候选或用分级模型。这些百分点和二百毫秒是待离线评测与压测校准的初始切换条件：调低召回容忍会增加算力，调低延迟阈值会更早牺牲长尾证据。

第三步为频繁更新语料建立增量索引和代际发布。若新文档可检索 lag 的 `p99` 超过业务新鲜度目标，或墓碑积累超过 segment 条目的百分之十五使过滤开销明显上升，则触发 compaction/增量代发布；阈值需通过删除演练和真实更新分布校准。双代只在验证窗口内并存，查询绑定一个 generation，不能逐分片混读。

未选择“把整个知识库塞入超长上下文”，因为令牌成本随语料增长且权限、版本与引用更难约束；当语料小于上下文上限、更新稀少且每次都需全局比较时它会重新变优。也未选择只按租户建立独立索引，因为文档级 ACL 变化会复制大量向量；若租户完全隔离且共享率接近零，物理分索引反而简化安全审计。

## 设计决定

接入器从权威源读取 immutable source revision，生成带内容摘要的 chunk，以固定 chunker_version 和 embedding_model_digest 写入不可变 segment；目录在校验完整性后原子发布 index_generation。查询先取得 principal 的当前 policy_epoch，用同空间 query embedding 搜候选，并让 ACL 和版本过滤进入候选规划，再重排。上下文包记录每个 chunk 的 digest、原文区间与授权依据，提示模板要求引用 chunk_id 并允许“证据不足”。

生成 attempt 固定 `(index_generation, policy_epoch, retriever, reranker, prompt, model, tokenizer, safety)`。首批私有输出进入短缓冲，发布器重新读取当前 ACL；若 epoch 变化，重新验证所有证据，剔除后上下文不足则终止并返回权限变化，而非继续使用已经进入 GPU 的文本。重试沿用 request_id 时返回原 attempt 状态；需要重新检索时创建新 attempt，不能把两个 generation 的流拼接成一个答案。

过载先降低候选上限和使用较轻重排器，同时标记质量等级；再拒绝低优先级长问题，不能绕过 ACL 或引用校验。反选“检索后立即流式生成、不做发布再校验”延迟更低，但只适用于公开且不可撤权语料。确定性索引构建能保证同 generation 的候选数据一致，生成仍受采样、批次和数值执行影响，因此谱系支持归因而非逐字保证。

## 运行与演进

线上 SLI 分为质量、体验和成本：按权限过滤后的 `recall@k`、引用覆盖/蕴含、拒答准确率、无依据断言率；检索、重排、首令牌和完成延迟；每个成功答案的搜索 CPU、重排 GPU 秒与输入输出令牌。安全侧监控撤权到不可披露的最大时间、旧 acl_epoch 命中、跨租户候选数和删除后残留。所有质量指标按 index/model/prompt generation 分桶，避免平均数掩盖某代回归。

故障演练：t0 用户获权并检索文档 v7；t1 候选已装入上下文；t2 管理员撤权并把 epoch 从 41 推到 42；t3 生成器完成含机密短语的草稿；t4 发布器看到 42 与 attempt 的 41 不同，重新校验失败并丢弃缓冲，用户只收到权限已变化。随后模拟权限服务不可用：私有库 fail-closed，公开库可按缓存策略继续。演练验证日志不记录正文、旧 chunk 不从答案缓存泄露。

升级先离线固定一组带权限、时效和无答案问题，影子构建新 chunk/model/index generation，比较 recall、groundedness、延迟和单位答案成本；灰度查询整次绑定新代，回滚只改目录指针。每月做删除穿透与旧 segment 恢复演练。业务切换条件包括无依据断言率超过风险预算即回滚生成配置，以及每个有效答案成本超过人工替代收益时缩小候选/模型；具体预算由业务校准，不能虚构评测结论。

## 面试考察本质

给定“私有证据只能在当前授权下披露，且每个关键断言必须回到确定原文版本”这一不变量，因为相关性只能在有限候选工作量内近似获知、文档与权限持续变化、生成又具有概率性，候选人应推导出版本化检索谱系、发布前授权栅栏和质量—延迟—成本联合预算，而不是把向量库接到模型就宣布完成。

优秀信号是把 retrieval recall 与 groundedness 分开，明确 source revision、chunk/index/model lineage，讨论 ACL 在候选规划和发布时的双重作用，并承认可重放输入不等于确定输出。常见误区包括只量生成质量、用答案缓存绕过当前权限、引用一个会变化的 URL、索引升级时混合不兼容空间，或声称温度零即可审计复现。

二十分钟回答应给出文档—chunk—index—answer 数据链、成本公式和权限事件反例；四十分钟补充候选/重排、拒答、更新删除与双代发布；六十分钟再谈多租户隔离、评测集、灰度、撤权演练和成本降级。追问可用“检索后立即撤权”检验安全边界，用“相关片段排第 21”检验召回预算，用“原文 30 天改为 14 天”检验引用是否真正可追溯。
