# 短链接服务

## 表面题目

设计短链接服务，表面流程是接收一个长网址，返回较短别名，并在读者访问别名时重定向到原目标。真正的状态变化是“某个公开别名从未分配变为绑定到一个经过校验的目标”，成功不仅指创建接口返回，还意味着随后所有合格解析请求都不会把该别名解释成另一个租户的目标。读路径通常远多于写路径，且一次错误重定向可能直接越过安全边界。

题名掩盖了一个决定架构的歧义：短链究竟是创建后永久不可变的引用，还是所有者可以修改、停用和设置有效期的入口。前者近似内容寻址后的稳定映射，后者则是带缓存传播语义的可变路由规则。自定义别名、滥用拦截、访问统计和链接预览可以加入，但都不能改变“一个别名在一个版本上只有一个权威目标”这一核心。

## 反问与边界

首先确认成功语义：创建是“任意生成一个别名”还是“占用指定别名”，同一个幂等键重试是否必须返回同一结果；解析是返回 HTTP 跳转、内部目标对象，还是在目标被封禁时拒绝。还要明确可变性、有效期、删除后能否复用别名、读后多久必须看见更新，以及误跳转与暂时不可用哪个风险更大。若删除后的别名立即复用，旧邮件、浏览器历史和边缘缓存就可能把旧访问者带到新租户，因此本设计不复用曾公开的别名。

容量边界用变量而非伪造生产数字表达。设峰值创建率为 `W`、解析率为 `R`、存活映射数为 `K`、别名空间为 `M`、热点链接占读流量比例为 `h`。需要询问目标可用性和解析 `p99`、允许的缓存陈旧窗口、地域数、恶意扫号速率、单租户配额以及保留期限。目标网址是敏感输入，写入前应限制协议、长度和危险域名；日志默认记录别名与目标域名摘要，不记录完整查询串。非目标包括替用户抓取任意页面、保证第三方目标可用，以及把点击分析做成强一致计数器。

## 客观模型

最小接口是 `Create(target, custom_alias, idempotency_key)`、`Resolve(alias)`、`Update(alias, expected_version, new_target)` 与 `Disable(alias, expected_version)`。核心记录为 `(alias, target, owner, mode, version, status, expires_at)`。别名目录分片是映射状态的最终所有者；创建时的唯一条件写入是自动别名和自定义别名共同的线性化点。可变记录的条件更新把版本从 `v` 推进到 `v+1`，边缘缓存只保存带版本和到期时间的解析结果。

不变量包括：任一别名不能同时指向两个已生效目标；不可变模式创建后目标不再变化；可变模式只有匹配当前版本的所有者操作才能更新；停用不能被延迟重试重新激活；未授权租户不能枚举或修改映射。对于均匀随机生成，下一次尝试碰撞的概率近似 `K/M`，一次成功所需尝试次数的期望为 `1/(1-K/M)`。若六位 base62 空间约为五百六十八亿而占用率达到百分之九十，平均尝试已从接近一次升到十次，写入尾延迟和唯一索引争用同时放大，增加创建节点并不会增大 `M`。

解析容量不能只按平均 `R` 规划。一个名人链接使 `h` 接近一时，单别名会穿透到同一目录分片；边缘缓存命中率记为 `c`，则回源解析约为 `R*(1-c)`，但更新后的主动失效和冷启动会瞬间降低 `c`。缓存键必须包含命名空间和规范化后的别名字节，不能对大小写做未经契约声明的折叠。

## 必然约束

[DEDUCED:url-shortener-live-keyspace-determines-collision-cost] 前提是自动别名从有限空间独立抽取，目录中已有 `K` 个仍不可复用的键。分配节点在尝试唯一写入前无法知道候选是否空闲，因此单次碰撞概率由 `K/M` 决定。最小反例是空间只有四个码且已有三个：无论启动一台还是一百台生成器，每次随机尝试仍有四分之三失败；并发生成器反而会争用最后一个空位。结论是扩计算资源不能消除码空间饱和，必须通过增加码长、分配不重叠号段或提前迁移命名空间降低占用率。若别名来自保证唯一的序列再编码，这个随机碰撞结论不适用，但序列可枚举性和分配器可用性成为新的边界。

映射正确性也不能由缓存独立决定。可变别名在版本 `7` 从目标甲改为目标乙时，未收到失效消息的边缘节点只掌握旧事实；若响应不携带版本与陈旧上界，它无法区分“甲仍正确”和“更新通知丢失”。因此强制立即全局可见需要读时回到权威目录或同步确认所有副本，而低延迟边缘解析只能接受有界陈旧。删除后复用更危险：旧缓存中的目标甲和新所有者的目标丙都对同一别名看似合法，没有世代号便不存在安全合并方式。

## 从简单方案演进

最简单正确基线是一张带唯一索引的单库映射表：创建用条件插入，解析按别名主键读取。它适合低流量且给出清楚的唯一性，但当解析数据库 `p99` 超过端到端预算的三分之一，或读连接利用率连续十五分钟高于百分之七十时，首先引入只读缓存。缓存解决重复读取，却新增陈旧数据、热点键和穿透；空结果只短暂缓存并配合每租户速率限制，避免恶意枚举把不存在别名变成回源洪峰。

第二步把解析缓存推到区域或边缘。不可变链接可使用长到期时间，因为同一键的目标永不改变；可变链接则把版本、最大陈旧期与失效事件绑定。当更新到解析可见的 `p99` 超过业务承诺，或失效队列最老事件年龄超过最大陈旧期的一半时，应缩短到期、改为版本探测，或把要求“立即生效”的少数别名绕过边缘缓存。代价是回源增加，失败模式则从单纯缓存未命中变为部分区域长期持有旧版本。

第三步在映射数或写吞吐越过单分片容量前按别名哈希分片。目录路由必须版本化；迁移期间源分片只转发或拒绝旧纪元写入，不能让两边同时占用同一自定义别名。码空间占用率达到百分之五十且分配重试 `p99` 超过两次时，就应在用户可感知前扩展新码长，因为双码长读取兼容容易、已发短链重写不可能。一个未选择的方案是把长网址哈希直接截断作为别名：它天然去重，却会泄露相同目标关系，处理碰撞和可变目标也仍需目录；只有产品明确要求同一规范网址全局共用一个不可变短链时，它才重新占优。

## 设计决定

本设计把别名目录设为权威状态，自动别名采用足够大的随机空间加唯一条件插入，自定义别名走同一原子占用路径。创建重试由幂等键索引到第一次成功结果，避免客户端超时后制造多个公开链接。解析先读边缘缓存，未命中再按目录路由查询；目录超时不猜测目标，已有且未超过最大陈旧期的缓存可以降级返回，并在响应追踪中标记旧版本。危险协议、已禁用目标和无效别名直接拒绝，不通过重试扩大写权限。

[DEDUCED:url-shortener-mutability-requires-versioned-redirects] 不可变别名一旦发布，缓存节点无需获知后续目标变化，因而可以用长缓存换取低回源，但代价是拼写错误或目标迁移不能原地修复。允许修改后，旧值与新值在传播期间必然并存；版本化条件更新可阻止两个管理请求互相覆盖，版本化缓存与有界到期可定义用户最多看见旧目标多久，却不能同时提供无限缓存和瞬时全局更新。由此产品必须在创建时声明模式，不能把“可能以后修改”隐藏成实现细节。

可变更新采用 `expected_version`，冲突返回当前版本而非盲目最后写胜。禁用产生新的墓碑版本且保留别名，避免延迟创建重试或租户更替复活旧入口。未采用“所有解析都同步读主库”的方案，因为它让全球读延迟和目录故障直接暴露给每次点击；当链接属于支付回调等绝不允许陈旧目标的低流量场景时，同步权威读取才重新变优。

## 运行与演进

核心 SLI 是有效别名解析成功率、解析 `p50/p99`、错误目标事件数、更新可见延迟、缓存命中率、创建碰撞次数和唯一写冲突率。容量观察按分片与热点别名拆分，不能让总体高命中掩盖单区域失效积压。过载时依次丢弃点击分析、预览与非必要审计扩展，随后限制创建和自定义别名；已存在映射的解析最后降级。安全封禁优先于可用性，命中明确墓碑的缓存不得继续跳转。

一次故障演练按时间线执行：零毫秒时别名 `x` 的版本十二指向甲，五十毫秒时目录提交版本十三指向乙，六十毫秒失效发布节点宕机，边缘 A 收到事件而边缘 B 未收到；在最大陈旧期内，A 返回乙、B 可带降级标记返回甲，期限到达后 B 必须回源或失败，不能无限返回甲。恢复后按版本十三重放失效是幂等的，旧版本十二事件不能覆盖新缓存。演练检查用户可见结果、失效队列年龄和回源峰值。

发布新码长先让读取同时识别旧新格式，再只给小比例创建请求生成新码，比较碰撞率和路由错误后放量；回滚只停止新生成，不能撤回已公开别名。分片迁移使用目录纪元和逐分片校验，目标计数、墓碑数与抽样解析一致后才释放旧副本。多地域复制、目标加密、租户删除和点击数据保留各自有独立策略，映射表与分析事件不共享生命周期。

## 面试考察本质

给定“一个公开别名在一个版本上只能有一个权威目标”这一不变量，因为生成器在条件写入前不知道候选是否已占用、边缘缓存又不能立即知道目标已更新，候选人应推导出码空间饱和带来的碰撞下界，以及可变性、缓存时长和全局更新速度不可同时取满。最终选择应依据误跳转风险、更新时效和解析延迟，而不是先说使用某个缓存或数据库产品。

优秀信号包括主动区分不可变与可变链接、写出 `K/M` 碰撞模型、禁止删除后复用、说明唯一条件写入和版本更新各自的线性化点，并用热点链接而非平均流量规划读路径。常见误区是认为哈希不会碰撞、把扩容生成器当成扩大键空间、让缓存永久保存可变目标，或为统计点击在同步解析路径上维护全局计数器。

二十分钟回答应完成接口、记录、不变量和单库加缓存基线；四十分钟加入码空间计算、幂等创建、边缘失效与故障时间线；六十分钟再讨论目录分片迁移、滥用防护、多地域陈旧边界和码长演进。追问可沿着“自定义别名抢占”“更新与禁用并发”“热点穿透”“旧二维码永不消失”展开，每一步都要求说清谁有写权限、读者何时能看见哪个版本，以及什么指标触发下一种复杂度。
