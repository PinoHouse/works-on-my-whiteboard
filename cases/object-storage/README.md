# 对象存储

## 表面题目

设计对象存储，表面上是按 bucket 和 key 上传、下载、覆盖与删除大文件。真正需要保证的是：`PUT` 成功后，一个完整、校验通过的对象 generation 已在声明故障范围内可恢复；任意 `GET` 只能看到旧的完整 generation 或新的完整 generation，不能看到上传到一半的字节。命名空间元数据拥有 key 到可见 manifest 的映射，块服务只拥有不可变字节及其副本或纠删分片；只有命名空间的条件发布点可以声明某一 generation 对读者成功可见。

主流程是分配上传身份、传输和校验块、形成 manifest、满足耐久确认、原子发布名称，再异步回收无引用块。题名掩盖了“字节已收到”“对象已耐久”“名称已可见”三种状态。本题支持版本化、multipart、条件覆盖和生命周期，不提供就地修改任意字节，也不把目录列举的即时一致性自动等同于单 key 读取契约。

## 反问与边界

先问对象大小和分布、上传下载比、热点 key、请求峰值、首字节与整对象延迟、单流网络失败率，以及成功后需抵抗磁盘、节点、可用区还是地域故障。覆盖是否保留旧版本，删除是新增 delete marker 还是立即不可恢复，`GET` 是否支持 range，列表与单 key 读取的一致性是否相同，multipart 暂存多久后回收。内容校验是客户端 checksum、分片 checksum 还是端到端哈希，也必须明确。

成本边界包括复制或纠删码、跨区字节、冷层迁移、小对象元数据、修复带宽和出口流量。还要问租户配额、服务端或客户端加密、密钥轮换、保留锁与合规删除。后文阈值是待校准策略而非测量。非目标包括覆盖对象的原地字节事务、在未完成 multipart 时提供普通 GET，以及为了降低延迟在耐久分片不足时仍返回同样的成功语义。

## 客观模型

核心状态为 `{tenant,bucket,key,generation -> manifest(chunk_ids,checksums,size,policy,state)}`。块是以内容或上传身份寻址的不可变单元，块服务拥有物理分片；namespace 服务拥有 key 的当前 generation、版本历史和条件谓词；上传会话拥有暂存 part 集合。`CreateUpload`、`UploadPart(upload_id,no,bytes,checksum)`、`Complete(upload_id,ordered_parts,expected_generation)` 与 `Get(key,generation)` 把传输和发布分开。

不变量是 manifest 引用的全部块校验通过且达到声明耐久度后才能发布；名称切换原子；重试同一 complete 不产生不同 generation；覆盖和删除按预期 generation 仲裁；回收只能删除不再被任何可恢复 manifest 引用的块。一个 multipart 上传的部分到达不能改变普通读的可见对象。

逻辑对象大小为 `B`，三副本物理容量约为 `3B`；采用 `k+m` 纠删码约为 `B×(k+m)/k`，但无缓存读取至少取得 `k` 个数据等价分片，丢失 `f` 个分片只在 `f≤m` 时可重建。还需另计最小块填充、小对象元数据、校验、跨区流量和生命周期重写。对象请求均匀并不意味着字节均匀：一个十 TiB 热对象会支配网络和修复队列。

## 必然约束

[DEDUCED:object-storage-object-ack-must-mean-a-complete-checksummed-generation] 若成功只表示接收节点看到若干字节，客户端无法判断网络断开前是否完整。最小反例是 `report.csv` 有十个 part，前九个写入后第十个超时；若服务此时让 key 可读，读者取得九成文件却得到成功响应。对象长度字段也不足以发现同长度位翻转，故需要有序 part、每块校验和、总大小或端到端校验，并在完整 generation 可恢复后才确认。若接口另有“暂存成功”，可以早确认，但不能冒充对象 PUT 成功。

[DEDUCED:object-storage-manifest-commit-separates-data-transfer-from-visible-publication] 网络重试可能让旧上传的 part 10 与新上传的 part 1 至 9 同时存在；块服务只知道字节到达，不知道用户最终选哪组。只有 `Complete` 对 `{upload_id, ordered part numbers, checksums, expected_generation}` 条件提交，namespace 才能从旧 manifest 原子切到新 manifest。否则 GET 可能读取混合上传。不可见孤儿块是分离传输和发布的可回收成本，优于把部分对象暴露为业务事实。

[DEDUCED:object-storage-redundancy-choice-trades-capacity-for-small-read-and-repair-work] 三副本可从一个完整副本读取，容量为 `3B`；例如 `k=8,m=3` 的纠删码容量约为 `1.375B`，却需取得八个等价分片并在故障时编码重建。小对象会因最小分片和并发请求放大而不划算，修复也与前台读争夺带宽。因此纠删码不是“更省的复制”，而是容量、读取尾延迟和恢复工作的显式交换。

## 从简单方案演进

最简单正确基线是单节点不可变文件加原子元数据重命名：先写临时文件并校验，刷盘后把 key 指向完整文件。需要机器故障耐久时，将块同步复制到独立故障域并让 manifest 只引用已确认块；容量增长后再分片 namespace 和块布局。大对象与不稳定网络使整对象重传越界时，才加入 multipart 和可恢复 upload_id。

第一个**待校准**切换指标是：单流上传失败重传字节占比超过百分之三，或对象大小 `p95` 超过二百五十六 MiB，启用三十二到一百二十八 MiB 可校验 part，并把并发 part 数限制在租户带宽和块队列预算内。百分之三、二百五十六 MiB 和 part 大小都需按网络、校验 CPU 与请求成本压测；part 过小会放大元数据，过大会增加失败重传。

第二个**待校准**切换指标是：冷数据容量成本超过总预算百分之六十，同时对象大小 `p50` 大于六十四 MiB，且读取 `p99` 可额外容忍八十毫秒时，从全复制迁移到 `k+m` 纠删码。阈值必须按介质价格、修复目标和地域带宽校准；小对象、热点和低延迟对象仍保留副本或缓存，不能全库一刀切。

未选择把可变文件系统直接暴露为对象后端，因为就地覆盖和目录操作会模糊 generation 发布点，并使部分失败可见。若工作负载主要是小文件随机写和严格目录事务，文件系统会重新变优。也未选择所有对象立即纠删码，因上传关键路径编码和小读放大会破坏热点 SLO；后台冷却后转换更符合成本边界。

## 设计决定

本设计让上传先生成不可见 upload_id，part 写入块服务并分别校验。`Complete` 验证有序 part 列表、总大小和端到端 checksum，等待复制或纠删分片达到策略要求，再由 namespace 以 `expected_generation` 条件提交新 manifest；这个提交是对象可见性的线性化点，也是普通 PUT 成功的声明者。覆盖生成新 generation，不修改旧块；删除生成版本化 marker，并由生命周期异步回收。

超时后的 `UploadPart` 以 upload_id、part no 和 checksum 幂等；内容不同则冲突。`Complete` 超时后客户端查询 upload 状态或重试相同请求，不能启动无身份覆盖。某些块不可用时，若剩余分片仍满足重建，GET 可读并排队修复；低于恢复边界则明确失败，不返回截断字节。过载时先限制后台迁移和修复并发，再限低优先级上传，保护 namespace 与热点下载。

反选方案是“所有 part 到达就逐步更新可读 key”。它能实现流式消费，却改变了对象原子读取语义。若产品需要边写边读，应定义独立的追加日志或流对象接口，暴露 committed offset；普通对象 API 仍只发布完整 generation，不能让两种成功含义共用同一 key。

## 运行与演进

SLI 包括完整 PUT 成功率与延迟、checksum 不匹配、complete 结果未知、可见 manifest 缺块计数、单 key GET 首字节、修复债务、不可见孤儿字节、namespace 条件冲突、冷热层迁移和按租户出口流量。耐久健康按故障域和可重建分片计算，不只数进程。后台任务要有字节速率和前台延迟反馈，防止大规模修复压垮正常读取。

故障演练时间线：零毫秒上传一百个 part；九十毫秒所有块校验完成；一百毫秒 manifest 达到耐久条件但 complete 响应丢失；客户端以同一 upload_id 重试，namespace 返回已发布 generation 42，而不是生成 43。另一轮在九十九个 part 后断网，普通 GET 仍读旧 generation 41，暂存块在保留期后回收。再断开 `m` 个分片验证可重建，断开 `m+1` 个验证明确失败且不返回部分数据。

格式演进先让读者理解新旧 manifest 和编码策略，再为新对象写新版本，后台逐对象迁移并校验端到端哈希；名称只在新布局可恢复后切换，回滚保留旧 manifest。密钥轮换可以重包裹数据密钥而不重写全部块；合规保留锁由 namespace 仲裁，块回收需遵守所有引用和保留期。跨地域复制状态必须与 PUT 的地域耐久等级一起呈现，不能把“最终会复制”写成已达到地域灾难保护。

## 面试考察本质

这题考察的是：给定“对象写成功后可恢复且读取永远是一个完整 generation”的不变量，因为字节块服务不知道 multipart 是否最终完成、命名空间才知道哪个版本可见，候选人能否让 manifest 成为唯一发布仲裁，并用复制或纠删码明确交换容量、读取尾延迟和修复工作。

优秀回答会区分字节接收、耐久和发布，画出九个 part 成功而第十个超时的最小反例，用 `B×(k+m)/k` 计算容量，并说明旧完整或新完整而非逐块可见。常见误区是把块数达到当成对象成功、覆盖旧 generation 就地修改、让 complete 重试重复发布，或只比较存储单价而忽略小对象和修复放大。

二十分钟完成对象、manifest、块和 PUT/GET 语义；四十分钟加入 multipart、条件 generation、校验和、复制与纠删码；六十分钟再讨论生命周期、热点、灾难恢复、加密和格式迁移。追问始终落到响应时刻：哪些完整字节已可恢复，哪个 namespace 状态让它可见，超时重试如何找到同一结果，以及坏块出现时 GET 是否仍满足完整性。
