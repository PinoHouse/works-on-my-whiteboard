# 聊天与即时通信

## 表面题目

设计聊天与即时通信系统，主流程是用户向单聊或群聊会话发送消息，在线设备尽快显示，离线设备重连后补齐，并支持多设备、撤回、成员变更和已读游标。真正的成功必须分三层：服务接受并持久提交消息、某台设备收到消息、某个人实际看到消息。只有第一层能由服务端提交点精确定义，后两层受连接、终端进程和用户行为影响。

题名还掩盖了“顺序”的范围。业务通常需要同一会话内有可恢复顺序，并不需要所有会话共享一个总序；发送者屏幕上的本地时间也不是其他设备可验证的提交顺序。输入状态、正在输入提示和在线徽标可以丢失，已接受消息则不能因为网关断线而永久消失。

## 反问与边界

先确认发送成功是落入几个副本、返回序号，还是必须等到对端设备 ACK；重试是否携带稳定客户端消息 ID；一名用户有多少设备，离线历史保留多久，撤回与成员移除多快生效。还要确认每会话与全局峰值、最大群规模、机器人流量、消息大小、发送 `p99`、重连补拉预算和跨地域成员比例。未知生产数用 `λ`、设备数与保留窗表达，不把示例阈值说成测量结果。

安全边界包括服务端在每次拉取和广播前按成员世代授权、正文静态加密、租户隔离、封禁与审计保留。已读回执只表示某设备把游标推进到某序号，不证明本人理解内容。非目标包括跨会话全局排序、对所有设备恰好一次展示、永久保留所有输入状态，以及用聊天投递 ACK 推导用户在线。

## 客观模型

最小接口为 `Send(conversation_id, sender, client_message_id, body, member_epoch)`、`Sync(conversation_id, after_seq, limit)`、`Ack(device_id, conversation_id, seq)` 与 `Recall(message_id, expected_version)`。权威状态是会话分区拥有的 `conversation -> {member_epoch,next_seq}` 和追加记录 `message -> {seq,sender,client_message_id,body,status,version}`；设备游标、网关连接和推送结果都是派生状态。`(sender,client_message_id)` 条件占用与会话序号分配在同一提交边界完成。

不变量是：一个会话中每条已接受消息只有一个序号；同一发送者的同一客户端 ID 只占用一次；旧成员世代不能读取移除后的内容；撤回产生更高版本，延迟广播不能让旧正文重新可见。跨会话没有共同线性化点，墙钟只能用于展示，不能替代会话提交序列。

若会话消息率为 `λ`、在线设备数为 `D_on`、平均消息字节为 `b`，在线出口近似 `λ×D_on×b`；离线窗口 `T` 内每台设备的待补消息上界为 `λ×T`。持久日志只追加一次，不代表群聊网络成本只发生一次。热点由单个大群的序号分配、在线扇出和多设备回放主导，不能用全站平均消息率掩盖。

## 必然约束

[DEDUCED:chat-messenger-order-is-per-conversation-not-global] 一个会话分区能在提交时分配唯一 `seq`，因此断线后所有设备可用同一序列补拉；两个无关会话没有共同协调点，墙钟又可能漂移。最小反例是会话甲在时钟慢的区域先提交、会话乙在时钟快的区域后提交，按时间戳合并会反转真实提交关系。结论是只承诺会话内顺序；只有中央审计明确需要且写量很低时，全局总序才值得它的协调代价。

[DEDUCED:chat-messenger-commit-and-device-delivery-are-different-acknowledgements] 网关写 socket 成功时只知道字节交给本机协议栈，不知道终端进程是否存盘，更不知道用户是否阅读。服务端可以精确保证已提交消息及其版本可恢复，却只能 best-effort 推送、展示和回传已读。把设备 ACK 当提交点会让离线成员阻塞发送；把提交 ACK 当已读又会制造错误产品语义。

[DEDUCED:chat-messenger-client-message-id-is-required-for-ambiguous-send-retry] 发送者在提交后、ACK 前断线，无法知道原请求是否生效。甲发送 ID `x`，服务提交为 `seq=41` 后回包丢失；三秒后若重试被当作新请求，就产生 `seq=42` 的重复消息。稳定 `(sender,x)` 使分区返回既有 41，正文相同并不能替代 ID，因为两次合法消息也可能文本相同。

## 从简单方案演进

最简单正确基线是单个会话目录加持久追加表：发送条件去重并分配序号，在线端收到提示后按游标同步。它先守住持久事实，代价是所有流量经过一组节点。当单会话追加队列 `p99` 超过发送 SLO 的 40%，或单群写入达到分区待校准安全吞吐的 60%，把热点会话迁到专属顺序分区并限制机器人；不能用多个无协调写者争抢同一序号。

随后把在线分发拆为网关层和区域扇出树。它降低连接成本，却新增重复帧、乱序到达和旧成员缓存。设备以 `last_seq` 丢弃重复并补洞，广播前复核成员与消息版本。当群扇出 backlog 最老年龄超过实时预算的一半，优先合并提示、限制低优先事件，而不是拖慢权威追加。

离线恢复从逐条主动推送演进为按游标分页拉取。当离线游标滞后 `p99` 达到保留窗的 50%，或单设备一次需补拉超过 10,000 条，就发送摘要并分页；超过保留窗则返回快照边界，而非假装连续。上述 40%、60%、50% 与 10,000 都是待压测和产品校准的初始参数：调低会增加迁移与同步成本，调高会扩大队列、恢复时间和热点风险。

未选择“所有消息建立全球总序”，因为它把无关会话绑到同一故障域。只有法规审计要求跨会话中央事件顺序，且吞吐足够低时才重新考虑。

## 设计决定

本设计让会话分区拥有消息真相，网关只拥有短暂连接。发送先验证当前成员世代，再以客户端 ID 去重、分配 `seq`、持久提交，随后异步扇出；客户端只有收到提交结果或同步到该序号才把本地气泡标为已接受。超时使用同一 ID 重试，设备实时帧丢失则按游标补洞。

故障时间线是：0 ms 提交消息 41；10 ms 给发送者的 ACK 随连接断开而丢失；30 ms 接收者设备 A 收到实时帧、设备 B 离线；3 s 发送者重试并得到既有 41；5 s 设备 B 从游标 40 拉到 41。可精确保证的是消息 41 的会话位置、幂等结果和当前可见版本；不能保证 A 何时展示、B 实时收到或已读及时返回。

撤回和成员移除都增加版本或成员世代，旧推送只提示“状态变化”，不能授予正文读取权。未采用等待所有成员设备 ACK 后才返回的方案；小型控制室若每台受管终端都必须确认且允许阻塞，它才可能变优。

## 运行与演进

核心 SLI 是发送提交成功率与 `p99`、单会话追加队列年龄、实时提示延迟、游标缺口率、重连补拉字节、重复 ID 命中率和撤回后旧版本读取次数。按会话、群规模、区域和设备数分桶，不能用总体平均掩盖热门群。过载时先丢输入状态和已读广播，再合并通知、限制机器人和大群发送，持久同步路径最后降级。

演练上述 ACK 丢失时间线，同时注入旧成员网关延迟帧，验证重试仍返回 41、离线设备能补齐、旧世代无法读取。协议升级先让读端识别新旧消息版本，再灰度新写端；回滚停止产生新格式，不能删除已提交记录。分片迁移使用会话路由世代 fencing，源分区转发或拒绝旧世代写，绝不双写分配序号。

多地域会话按所有者区域提交，远端成员承担额外 RTT；若产品选择本地可写，就必须引入明确的多写者合并语义，而不是仍声称严格会话序。隐私删除需要保留最小墓碑与审计关系来阻止复活，正文与设备游标分开设保留期。成本按持久字节、在线出口和离线回放三项归因。

## 面试考察本质

给定“已接受消息在一个会话内只有一个可恢复顺序”的不变量，因为发送方超时后不知道追加是否发生、离线设备不知道漏过哪些实时帧，候选人应推导出会话提交序列、稳定客户端消息 ID 与游标同步；并依据热点会话串行度、群扇出和恢复窗口决定分区与降级，不能把服务提交、设备送达和用户阅读混成一个 ACK。

优秀信号包括主动限定顺序范围、明确会话分区的写权限、用 `λ×D_on×b` 算扇出、给出 ACK 丢失反例，并说明撤回如何阻止旧广播复活。常见误区是先说消息队列、追求跨会话总序、为每台离线设备永久可靠推送，或用 presence 和已读回执替代消息日志。

二十分钟回答应覆盖接口、状态所有者、不变量和发送重试；四十分钟加入群聊热点、多设备游标、成员世代与故障时间线；六十分钟再讨论跨地域路由、保留窗、协议迁移和隐私删除。追问始终围绕“哪个事实已提交、哪个端点只是可能收到、什么指标触发切换”展开。
