# 实时评论

## 表面题目

设计实时评论系统，为直播间、赛事或热点内容接收评论，审核后让大量在线观众快速看到，并在重连时提供有限历史。成功语义分为评论被权威房间日志接受、边缘实时广播、某位观众屏幕显示三层。前者可持久、可恢复，后两者受慢连接和审核竞态影响，只能给有界 best-effort 体验。

它与群聊的关键差异是观众规模和契约：十万观众不应每人拥有永久可靠队列，直播滚屏也没有必要在十分钟断线后逐条重放。与此同时，审核隐藏和撤回是当前可见性事实，延迟广播不能因为曾经合法就绕过新裁决。

## 反问与边界

先问评论是先审后发还是先发后审，房间内是否要严格序号，历史保留多久，主播/管理员消息是否有优先通道，用户断线后补多少条。容量要确认房间评论率 `λ`、观众数 `V`、平均字节 `b`、热点分布、区域数、慢观众比例、审核吞吐和滚屏延迟预算。

还要明确丢帧策略、抽样是否允许、作者重试如何去重、撤回是否要求已显示客户端移除。安全边界包括封禁世代、内容过滤、租户与房间配额、敏感日志保留。非目标是跨房间全局顺序、让每位观众实时看到每一条、保证已经看过的内容从记忆中消失，或把实时订阅连接当永久消费队列。

## 客观模型

最小接口为 `Submit(room,author,client_comment_id,body)`、`Moderate(comment_id,expected_version,decision)`、`Sync(room,after_seq,limit)` 与 `Subscribe(room,last_seq)`。权威状态是 `room -> {comment_seq,moderation_epoch}` 与 `comment -> {id,seq,author,body,visibility,version}`；房间日志和审核裁决拥有可恢复事实，边缘广播只维护短暂游标。

不变量是：一个房间中已发布评论拥有唯一 `seq`；同一作者与客户端 ID 只占序一次；当前隐藏、撤回或封禁版本不能被旧 fan-out 重新显示；读历史与边缘发送都复核当前可见性；慢观众不能无限占用逐人积压。主播和审核控制消息可用独立优先级，但不能篡改评论序号。

逻辑实时出口近似 `λ×b×V`，分层扇出降低单节点连接数，却不消除总字节。若每观众可靠积压 `q` 条，内存约为 `V×q×b`。100,000 人、每秒 50 条、每条 200 B 时，单个断线十分钟的观看者积压约 6 MB；若一万人同样慢就是约 60 GB，还需逐项处理撤回。

## 必然约束

[DEDUCED:live-comments-durable-room-order-and-live-fanout-have-different-contracts] 房间提交点可给评论分配序号并在保留窗内补拉，但任意多观众的网络与播放器状态不受它控制。断线观众能精确恢复“仍可见的持久评论窗口”，却只能 best-effort 收到实时滚屏；把两者混为可靠逐人队列会让恢复量随断线时长无界。

[DEDUCED:live-comments-slow-viewers-must-not-create-unbounded-per-viewer-backlog] 上述 60 GB 最小反例说明少量慢连接即可耗尽网关内存，并把队尾延迟传给健康观众。边缘应保留短帧窗口，超出即丢旧低优先帧并让客户端从 `last_seq` 按需补拉或直接跳到最新。小型付费讨论若每条必须追责，可以采用可靠队列，但它已经更接近群聊。

[DEDUCED:live-comments-moderation-version-is-part-of-visibility] 评论 C 在 0 ms 以 `seq=900` 发布，40 ms 被隐藏，100 ms 旧边缘才准备发送。若边缘只看“曾经发布”，C 会穿透当前裁决；发送前复核 moderation epoch 或收到版本化撤回才可阻止。已经显示的用户无法被保证从未见过，因此精确承诺止于未来读取和仍可控的缓冲。

## 从简单方案演进

最简单正确基线是单房间追加日志加一个广播进程：作者 ID 去重，审核状态随记录保存，客户端用序号补洞。它适合小房间。当 `λ×V` 达到区域广播池待校准安全下行的 70%，或房间 fan-out `p99` 超过滚屏预算的 50%，引入按地区分层扇出，主播与审核消息走优先通道。

分层树解决连接集中，却新增边缘陈旧、重复帧和撤回竞态。边缘携带 `seq` 与 moderation epoch，短窗口外不缓存逐人历史。当广播 backlog 超过 2 秒，丢弃低优先评论帧并发送最新游标提示；健康客户端不等待慢节点。重连按窗口分页，不把十分钟前的滚屏强行补齐。

当审核队列最老年龄超过 5 秒且举报率达到房间基线 3 倍，收紧作者速率、切到先审后发或只展示规则通过的高置信评论。70%、50%、2 秒、5 秒和 3 倍均是待压测、演练及产品校准的初始参数：调低牺牲完整度换尾延迟与安全，调高提高可见量但扩大资源和审核暴露风险。

未选择“每观众一个永久可靠队列”；只有参与者少、有身份且每条意见必须追责的讨论房间，它才重新变优。

## 设计决定

本设计由房间分区分配 `seq`，提交时验证作者世代与客户端 ID，按产品策略完成初审后写入可见状态。边缘订阅房间流，按序号去重并在发送前应用最新 moderation epoch；慢连接只保留有界字节和时间窗口，超过后跳到最新或提示按需拉取。

故障时间线是：0 ms C 以 900 发布；20 ms 部分边缘收到；40 ms 审核隐藏并推进 epoch；100 ms 慢边缘处理旧 C 时复核失败而丢弃；已显示客户端收到撤回标记但可能已经看见。可精确保证保留窗内当前可见历史按房间序补拉、旧版本不能用于未来授权；不能保证每个观众实时看到、跨房间顺序或抹除既有认知。

未采用所有评论都同步等待全量人工审核，因为它无法满足普通直播互动延迟。高风险未成年人房间、法规敏感场景或低评论率活动中，先审后发可能重新变优。

## 运行与演进

核心 SLI 是房间提交延迟、实时 fan-out `p50/p99`、按观众丢帧率、游标缺口、边缘 backlog 字节与年龄、审核队列年龄、撤回传播和旧版本发送次数。按房间规模、地区与优先级拆分。过载时先丢表情/低优先帧、采样普通评论，再限制新评论；主播公告、审核控制与持久历史查询保留。

演练让部分边缘延迟并在其间隐藏评论，验证发送前复核和撤回；再注入一万慢连接，确认每连接内存有界且健康观众尾延迟稳定。协议升级先让边缘识别新旧 epoch 字段，再灰度房间写入；回滚继续保留新版本可见性，绝不把隐藏评论复活。

多地域按房间所有者提交，区域边缘只复制；热点房间可迁移但用路由世代拒绝双写序号。正文、举报和观众连接日志分别设置隐私期限。成本按权威写入、区域复制、总出口与审核计算归因，不能用减少存储掩盖巨大实时出口。

## 面试考察本质

给定“房间内获准评论可按序恢复且当前审核状态不被旧广播穿透”的不变量，因为系统无法用有限资源让任意多慢观众实时接收所有帧，候选人应把持久房间日志与可丢失互动扇出拆开；再依据 `λ×V`、广播 backlog 和审核风险选择分层、采样或先审后发。

优秀信号包括主动与群聊区分、算逐观众积压、定义 moderation epoch、说明已经显示无法撤销认知。常见误区是为十万观众排永久队列、只看平均评论率、旧边缘无需复核，或用更多重试改善直播实时性。

二十分钟回答覆盖房间序号、审核和有界广播；四十分钟加入分层扇出、慢观众反例与降级；六十分钟再讨论热点迁移、审核攻击、协议灰度和隐私。追问持续要求说明“什么需要持久、什么允许丢、哪两个指标触发体验降级”。
