# 社交图谱

## 表面题目

系统保存关注、好友、拉黑等关系，并回答邻居、共同好友、二度关系或推荐候选。成功不是“用图数据库”，而是每条关系在并发修改下有可解释状态，高阶遍历在有界成本内完成，撤边和隐私变更及时阻止继续暴露。题名掩盖了边语义差异：关注是单向，好友常需双方接受，拉黑则会覆盖其他可见关系。

## 反问与边界

先问关系类型、谁能创建/接受/撤销，是否保留历史；查询是直接邻居、共同好友、最短路径还是批量分析。确认边写入可见时限、撤权上限、遍历 p99、最大深度、超级节点规模、读写比和地域。结果需精确还是可截断，排序依据是什么，用户删除如何传播到反向索引与推荐缓存。安全上身份、租户与隐私策略必须由可信服务给出。本题不承诺任意深度在线图算法，也不将离线推荐候选当作当前可见事实。

权威源按边 owner 划分：例如 `(from_user, relation_type)` 分片拥有单向边版本；双向好友由请求状态机导出两条可见边。正向、反向邻接表、共同好友缓存都是派生索引，必须带 edge/policy generation。

## 客观模型

命令为 `MutateEdge(edge_id, expected_version, action)`，查询为 `Neighbors(user, relation, snapshot)` 和 `Traverse(seed, depth, budget)`。边状态含 owner、target、type、version、visibility 和 tombstone。状态机对好友可为 requested、accepted、revoked；拉黑优先级高于 accepted。唯一边键与期望版本提供幂等 CAS。

若平均度为 d，朴素 k 阶遍历上界约 `1+d+...+d^k`；d=500、深度 3 已超过一亿两千五百万次展开。共同好友只需交两个有序邻接表，成本 `O(deg(a)+deg(b))`，应从较小集合哈希。超级节点使均值无意义，必须观察 degree p99/max 和展开截断率。

## 必然约束

[DEDUCED:social-graph-edge-must-have-one-write-authority] 一条关系边必须有唯一写权威和单调版本，否则双向副本在并发关注、拉黑与撤边时会产生不可判定冲突。最小反例是 A 区接受好友、B 区同时拉黑，两个“本地成功”若无统一版本就可能各自传播相反可见性。

[DEDUCED:social-graph-traversal-must-have-explicit-budget] 高阶遍历的访问量随度数乘法增长，必须显式限制深度、访问节点数与超级节点展开数。仅设置请求超时会把已消耗 CPU 和 fanout 留给系统，且返回质量不可解释；预算耗尽应携带 coverage/truncated。

[DEDUCED:social-graph-revocation-needs-generation-fence] 隐私撤边必须推进可在读路径检查的代际栅栏，仅异步删除反向索引会在传播窗口继续暴露关系。旧推荐缓存命中 A-B 后，还需当前 block/privacy generation 最终过滤；物理索引清理不能作为安全线性化点。

## 从简单方案演进

单表按 `(owner,target,type)` 存边并以 owner 查询，是小规模正确起点。反向查询扫描全表 p99 超预算后增加 target 反向索引；它带来双写与短时不一致，所以以 owner 记录为权威、异步投影携带版本。反向 lag 超撤权 SLO 时，不是盲目同步双写，而是增加读时 generation fence。

当邻接表超过单行或单分片吞吐，按 owner 与 bucket 分段；超级节点单独分层。若 degree p99 超单页上限、一个 owner 占分片 QPS 百分之二十，切分邻接段并给遍历预算。共同好友热点可预计算，但边更新放大约为受影响二跳数；只有查询/更新比和复用率足够高才物化。

更快反向索引刷新需要更多日志分区、写 I/O 与缓存清除。未选“关系两端同步事务双写”，因为跨区扩展和部分失败成本高；若关系量小、同库事务可用且要求即时双向显示，它重新变优。未选通用 BFS 无限制接口，因为资源无法隔离。

## 设计决定

选择 owner 分片的权威边日志、异步正反邻接投影、当前隐私栅栏。写入 CAS 后返回 committed version；投影按 edge_id/version 幂等，旧事件不复活 tombstone。查询固定邻接 generation，先取候选，再批量检查当前 block/privacy generation。高阶遍历维护 visited set、深度、节点与边预算，并对 hub 采样或截断。

owner 分片不可用时边变更失败，不在另一分片制造冲突；读可返回带 age 的旧邻接，但隐私服务不可用时保护型查询 fail-closed。超时后的写重试用 mutation_id 查询结果。未采用图缓存直接作为权威，它只加速且必须接受撤销栅栏覆盖。

## 运行与演进

观察边提交 p99、投影 lag、撤边后曝光数、邻接 degree 分布、visited nodes、hub truncation、超时与每查询 fanout。过载先禁止三度遍历，再缩小推荐候选和共同好友 K，直接邻居与撤权路径保留。按租户隔离遍历预算，避免名人节点拖垮普通用户。

故障时间线：09:00 A-B 为 accepted；09:01 A 拉黑 B，owner 写 version 12 并推进 privacy generation 30；09:01:01 反向索引仍显示 A，但读门按 30 过滤；09:03 投影 tombstone。演练旧 version 11 迟到不得复活。灰度新分桶方案时双写投影、核对邻接摘要后切读，回滚仍保留权威日志。压测切换指标是 visited p95 超预算、hub 分片 CPU 七成；业务校准撤边上限和截断时的产品文案。

## 面试考察本质

本题本质是：在“边状态唯一且隐私撤销立即约束可见性”的不变量下，因为反向邻接与高阶路径只能异步物化且超级节点规模不可预知，候选人必须推导边 owner、读时栅栏和遍历预算，再按查询深度、召回与撤权风险选择投影。

优秀回答会区分边事实和邻接索引，写出 `d^k` 放大，主动处理拉黑覆盖与旧事件。常见误区是只说图数据库、双向边无状态机、超时替代预算、缓存删除等于隐私删除。追问可进入共同好友、名人节点、跨区写与用户擦除。20 分钟讲边模型，40 分钟补索引和隐私，60 分钟讨论图遍历公平、迁移和离线推荐边界。
