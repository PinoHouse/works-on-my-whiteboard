# 直播

## 表面题目

设计直播系统，主流程是主播持续摄取音视频，系统实时编码成多个码率并发布递增分片，观众从接近“直播头”的位置播放，也可在 DVR 窗口内回看。成功不是主播连接建立，而是某个 ingest epoch 的分片被权威接受、进入可读时间轴，并在目标端到端时延内到达观众。直播与点播最大的区别是内容尚未完成，发布、处理和消费同时发生，错过的时间不能靠稍后完整上传自动修复。

题目还要明确互动直播、体育赛事和普通广播是否同一延迟级别。本文处理单主播权威输入、多地域观众、分片播放和有限 DVR；聊天、打赏及多主播混流只通过时间戳关联，不假设与视频原子一致。源切换必须产生新 ingest epoch，旧连接恢复后不能继续写同一频道。

## 反问与边界

先问端到端 glass-to-glass `p95`、可接受卡顿、首帧、并发观众、主播上行质量、地域、DVR 时长、内容审核和灾备目标。低延迟到底是一秒、三秒还是二十秒会改变分片大小、播放器缓冲和协议。还要问主播断线后允许静帧、降码率还是立即结束；故障切备用摄取时允许丢多少毫秒；新观众从关键帧还是任意分片开始；回看是否必须与直播头使用同一 manifest。

设观众数 `V`，平均选中码率 `r`，出口近似 `V×r`；十万观众、3 Mb/s 即约 300 Gb/s。若产生 `k` 个码率，编码计算和短期存储约随 `Σri` 放大。分片时长 `s` 时每观众每秒请求约 `1/s`；从 2 秒片改成 200 毫秒 chunk，请求/发布频率放大十倍，并压缩重试时间。真实阈值必须用目标设备、协议与峰值赛事压测，不能把示例数字冒充测量。

## 客观模型

实体包括 `Channel`、`IngestLease(epoch,owner,expires)`、`MediaChunk(epoch,sequence,part,digest,timestamp)`、`LiveManifest`、`DVRWindow` 和 `ViewerSession`。频道控制面通过线性化 CAS 发放递增 epoch 的摄取租约；媒体面每次提交携带 epoch，只有当前 epoch 可推进发布游标。chunk 不可变，manifest 是追加式快照并显式标出 discontinuity；新 epoch 可以从新关键帧开始，但绝不与旧 epoch 争写同一事实。

端到端时延近似 `capture + encode + package + publish + network + player_buffer`。把 player buffer 从 3 秒降到 500 毫秒最多节省 2.5 秒，却也把可吸收的编码/网络抖动减少 2.5 秒。若某一分钟的可用吞吐积分低于播放所需字节，卡顿不可避免。DVR 保留 `W` 秒时，每频道每 rendition 热存储约 `W×ri/8`，超过窗口的片可归档或删除；恢复只能从尚在窗口内且摘要完整的片重建。

不变量是一个频道任一时刻只有一个有效 ingest epoch；同一 `(epoch,sequence,part)` 只有一个 digest；发布游标不越过缺失必需 part；播放器按 manifest 的 discontinuity 重置解码。观看会话绑定策略 generation，公开视频可缓存，但紧急封禁必须在边缘请求路径上可见，不能等 DVR 自然过期。

## 必然约束

[DEDUCED:live-streaming-ingest-epoch-must-fence-a-recovered-old-publisher] 最小反例：节点 A 持 epoch 41，网络分区后控制面判租约失效并让 B 取得 epoch 42。B 发布 sequence 900；A 恢复后若只凭频道名也提交 900，边缘可能分别缓存两个字节版本，观众时间轴分叉。单靠“最后到达”无法说明哪个摄取合法。每个提交必须验证当前 epoch，42 生效后 A 的 41 即使迟到也拒绝，必要时 manifest 插入明确 discontinuity。

[DEDUCED:live-streaming-lower-latency-removes-buffer-failure-margin] 编码偶发 800 毫秒抖动时，三秒缓冲可继续播放；若为追求 700 毫秒端到端把缓冲降到 300 毫秒，同一抖动必然耗尽缓冲。降低 chunk 大小可以更早发布，却增加请求、加密和调度开销。故“低延迟且同等稳定”不是免费属性，需要更稳定的摄取/编码、冗余路径或接受更多卡顿。

[DEDUCED:live-streaming-dvr-window-bounds-replay-and-recovery] DVR 只保留过去 30 分钟，控制面宕机 40 分钟后不能承诺从头恢复这场直播；即使元数据日志存在，前十分钟媒体字节已删除。窗口也界定新观众可 seek 的范围和备用清单可重建的上限。若法规要求完整留档，应写入独立归档存储及其确认语义，不能把热 DVR 当永久证据。

## 从简单方案演进

最简单基线是一台摄取/编码服务器把单码率流转发给少量观众，延迟低且调试直接。并发使单机出口超过 70% 或最远地域 RTT 消耗延迟预算 30% 时，引入分片、边缘 fan-out 和不可变缓存；代价是发布清单、回源风暴和片序管理。网络分布造成卡顿率超过待校准 1% 时加入对齐多码率，换取编码与存储放大。

当单摄取节点故障导致频道恢复超过 RTO，加入双路上行或可迁移租约及 epoch fencing。双路不意味着双写：备用可接收但只有当前 epoch 可发布。若赛事频道的编码队列年龄超过一个分片时长，先保留低码率和音频、暂停最高档；若持续五分钟，则按频道价值和公平份额拒绝新开播，而非让所有频道同时越积越慢。这些百分比和分钟数是待演练策略。

第二组切换指标是：glass-to-glass `p95` 连续超过目标 20%，同时网络下载只占预算小部分，才缩短编码/分片路径；若 rebuffer 随后超过体验阈值，则回增缓冲或启用冗余。未选逐帧中心转发，因为普通广播的成本和连接状态过高；当场景是小规模强互动、必须亚秒反馈且可接受更低并发时，它重新变优。也未选无限 DVR，只有合规归档价值高于存储与删除成本时才独立建设。

## 设计决定

主播先向控制面申请短租约，取得 `channel_id,epoch`；摄取边缘持续续租并给每个 chunk 标 epoch、序号、时间戳和摘要。编码器输出对齐 rendition，发布器只推进连续完整前缀，manifest 指明 epoch 切换。观众会话从当前关键帧或 DVR 指定点开始，经边缘读取不可变 chunk。重复提交同一 digest 幂等，不同 digest 的同身份提交告警并拒绝。

过载降级从最小到最大用户可见：停止 DVR 归档复制和低价值预热；关闭最高码率；加大播放器目标缓冲、牺牲少量实时性；只保音频与最低视频；限制新观众，最后拒绝新主播。已发布连续前缀不因后台拥塞回撤。反选以墙钟和到达时间判主播新旧，因为分区节点的钟和网络顺序都不能证明权限；只有单进程、无故障切换的临时活动才足够。

## 运行与演进

关键 SLI 是 glass-to-glass 延迟、首帧、rebuffer、直播头落后、chunk 发布缺口、epoch fence 拒绝、编码队列年龄、各档命中、源站出口和 DVR 可恢复窗口。待压测的两组阈值：chunk 请求频率上升到何值令边缘 CPU 而非带宽饱和；备用摄取接管在多少秒内能产生新关键帧且观众卡顿低于目标。另需按主播分位观察上行丢包，平均值会隐藏一个超级赛事。

故障时间线：0 秒 A 持 epoch 11；5 秒 A 与控制面分区；租约到期后 8 秒 B 得 epoch 12 并从关键帧发布；10 秒 A 恢复提交 epoch 11 的迟到片，被 fence；观众 manifest 在 12 处标 discontinuity，不拼接同序号。再令发布器丢失一次确认，重试相同 digest 应幂等。DVR 演练删除窗口外对象，确认 API 对超界 seek 明确失败而非返回错位内容。

升级编码器先对影子频道双产出，比较时间戳、关键帧和字节成本，再灰度少量频道；回滚切回旧 pipeline 时提升发布 generation，保留 discontinuity。紧急下架提高 deny generation 并要求边缘确认，已经下载到设备的短缓冲是不可逆暴露。租户隔离按主播摄取、编码和观众出口分别限额，避免一个热点频道拖垮长尾直播。

## 面试考察本质

本题考察：给定“同一频道在任一时刻只有一个可发布 ingest epoch，时间轴必须连续或显式断开”，因为故障节点是否仍写入、未来 chunk 何时产生和网络抖动都不可立即获知，候选人应推导 fencing、追加发布和有限缓冲，并在低时延、卡顿容忍、计算与带宽之间取舍。

优秀回答会明确区分摄取租约与媒体字节，算 `V×r` 和 `1/s` 放大，画出 A 分区、B 接管、A 迟到的序列，并解释 DVR 不是永久归档。常见误区是把直播当“不断追加的点播文件”、只说 CDN、用墙钟选新主播、追求亚秒却不减少抖动，或在过载时随机丢片破坏连续前缀。追问可围绕关键帧、双路摄取、体育尖峰、内容审核和互动同步。

二十分钟完成延迟预算、epoch 状态机和 fan-out 估算；四十分钟加入多码率、边缘、DVR 与降级；六十分钟再谈跨区接管、审核撤权、编码迁移和成本。真正的分水岭不是知道某种直播协议，而是能否说清旧发布者何时失去写权限、观众能看到哪个连续前缀，以及每减少一秒延迟具体放弃了多少容错余量。
