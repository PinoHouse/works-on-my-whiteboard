# 通知投递

## 表面题目

设计通知投递系统，业务事件经过用户偏好、优先级、静默时间、频控和过期规则，选择 in-app、push、短信或邮件等渠道。表面成功像是“把一条通知发给用户”，实际至少分为创建有效通知意图、渠道提供方接受一次尝试、终端展示以及用户注意或处理。系统只能准确记录前两类可验证事实，不能从一次 HTTP 成功推导注意力。

这道题的核心不是给同一正文做可靠队列，而是按业务语义决定是否打扰、何时打扰、通过哪些已获同意的渠道打扰。订单失败与营销折扣即使文本相似，也有完全不同的过期和升级边界；同一订单换了文案，仍不应产生两个有效提醒。

## 反问与边界

先问哪些通知允许丢失、哪些受法规要求必须尝试，渠道优先级是用户选择还是产品策略，静默期如何处理紧急事件，去重窗口、过期时间和升级条件是什么。容量要区分业务事件率 `λ`、平均收件人数、每意图尝试次数、用户设备数和各渠道配额，并询问高优先队列的年龄 SLO、成本预算与退订风险。

还需定义“送达率”的分母与证据：提供方接受、终端回执、in-app 已打开和业务对象已处理不能互换。安全边界包括退订立即生效、敏感正文最小化、渠道凭据隔离、租户配额和可审计策略版本。非目标是保证通知按跨渠道顺序展示、证明用户阅读，以及在用户没有授权时自动增加备用渠道。

## 客观模型

最小接口为 `CreateIntent(recipient,type,business_key,priority,expiry)`、`Evaluate(intent_id,policy_version)`、`DispatchAttempt(intent_id,channel,consent_generation)`、`RecordAttempt(intent_id,channel,attempt_id,status)` 与 `MarkHandled(intent_id,source)`。权威状态包括 `notification_intent -> {recipient,type,dedupe_key,priority,expiry,historical_decision,state}` 和 `recipient_channel -> {current_consent_generation,deny}`；渠道尝试与可验证回执是外部效果记录。历史裁决回答当时为什么选择渠道；发送闸门在实际调用提供方的同一临界区用当前 consent generation 决定现在是否仍可外发，而不是把可延后使用的离线许可交还 worker。

不变量包括：同一用户、业务对象、提醒类型和策略窗口至多一个有效意图；每次外发必须携带当前 consent generation 并在发送闸门复核 `deny=false`，退订或授权撤销提交后旧 generation 不得越过闸门；过期意图不能因迟到重试再次唤醒用户；一次渠道未知结果不能被假写成失败。用户处理业务对象后，尚未发出的升级应被抑制；已经越过闸门进入外部平台的调用和终端已经显示的内容无法保证撤回。

若意图允许渠道集合 `C`，每渠道单次被下游接受概率为 `p_c`，在独立假设下至少一个被接受的上界模型为 `1-Π(1-p_c)`；打扰与成本却近似 `Σ attempts_c`。总尝试负载约为 `λ×recipients×avg_attempts`，并被少数群发事件与渠道故障放大。该模型只估算下游接受，不能推出用户注意。

## 必然约束

[DEDUCED:notification-delivery-policy-selection-precedes-channel-attempts] 偏好、频控、优先级和过期共同拥有“是否允许打扰”的业务语义，执行 worker 只看一次失败无法重建完整上下文。历史裁决必须持久化以说明当时为何选 push，却不能授权 60 秒后的短信：延迟 worker 发前必须以当前 consent generation 经过发送闸门复核。退订或授权撤销事务先写 `deny=true`、提升 generation，并对所有尚未越过外部调用边界的旧 generation 立即 fencing；只有已跨过该边界的尝试无法撤回。

[DEDUCED:notification-delivery-provider-acceptance-is-not-user-attention] 平台接受 push 只说明它接管一次尝试，设备可能离线、权限可能稍后撤销、横幅可能被系统折叠。即使终端回执存在，也无法证明人看见或理解。因此断连后可精确追踪意图、裁决、尝试和可信回执，只能 best-effort 追求展示与注意；把接受率命名为阅读率会直接误导业务决策。

[DEDUCED:notification-delivery-deduplication-is-scoped-by-business-semantics] 订单 O 的“付款失败”在 09:00 选择 push，平台实际接收但响应丢失；worker 若按正文去重，文案变化会再次打扰，另一个订单的相同正文又可能被误抑制。去重键必须包含业务对象、提醒类型和有效窗口。超时尝试可以重复，但不能因此创建新的有效意图。

## 从简单方案演进

最简单正确基线是意图表加单渠道 worker：条件创建业务去重键，记录历史裁决，并在每次外发前读取当前 consent generation。它容易审计，却在多渠道、静默期和群发事件下出现分支膨胀。某渠道 `UNKNOWN + permanent_failure` 连续 10 分钟超过待校准的 2%，且高优先意图最老年龄超过 SLO 的 30% 时，才按已提交升级策略灰度考虑备用渠道；真正外发前仍复核当前授权，而不是每次超时立即广播。

第二步引入策略编译与延迟调度，将偏好版本、过期、频控令牌和升级链固化到意图。它解决跨 worker 决策不一致，却新增旧策略迁移和定时队列热点。每用户每日通知 `p99` 达到频控预算的 80%，或七日静默/退订率相对基线升高 50% 时，扩大低优先聚合窗口并只保留 in-app 摘要，不能增加重试美化“送达率”。

第三步按租户、优先级和渠道隔离队列，避免营销洪峰阻塞安全提醒。高优先队列年龄达到预算一半时暂停低优先新尝试；渠道恢复后按意图过期和用户是否已处理重新裁决，不机械重放。2%、30%、80% 和 50% 都是待压测、演练与产品校准的策略参数：调低降低漏报风险但提高成本和打扰，调高则相反。

反选“所有事件立即向所有渠道广播”只在紧急告警、用户明确同意且漏报远比重复昂贵时合理；日常通知中它破坏偏好并放大疲劳。

## 设计决定

本设计让意图、历史策略裁决与当前发送授权分别成为权威：先条件创建 `(recipient,business_key,type,window)`，记录本次策略版本、优先级、有效期和候选渠道；worker 为每次外部调用生成 attempt ID，再把当前 consent generation 交给发送闸门。闸门不返回可缓存许可，而是在与退订/撤权按收件人渠道串行化的临界区内复核并跨越提供方调用边界：若 dispatch 先赢，尝试已进入外部平台；若撤权先赢，撤权提交时闸门已拒绝旧 generation。超时记录 `UNKNOWN`，查询或等待策略决定后续，不伪装为确定失败。in-app inbox 从当前仍允许读取的未过期意图生成，不依赖 push 历史。

故障时间线是：0 秒创建有效期 10 分钟的高优先意图 I；1 秒策略版本 12 选择 push，consent generation 7 的调用超时且平台可能已接受；10 秒状态记为 `UNKNOWN`；30 秒用户退订，授权事务提交 `deny=true,generation=8` 并 fencing 旧许可；60 秒延迟 worker 即使仍持有版本 12，也因 generation 7 复核失败而不能尝试短信。能保证的是 I、历史裁决、授权世代与尝试可恢复，且撤权提交后尚未跨越发送闸门的旧任务不再外发；不能撤回 30 秒前已进入平台的 push，也不能保证展示顺序、任一终端收到或用户注意。

未选择由渠道 worker 自行重试和升级，因为它们无法拥有全局频控与同意状态。若产品只有一个受控终端、没有用户偏好且通道返回可事务确认，简化的单 worker 才重新变优。

## 运行与演进

核心 SLI 是有效意图创建成功率、策略评估延迟、高优先队列年龄、各渠道 `accepted/unknown/permanent_failure` 比例、过期前至少一次可信接受率、每用户打扰次数、静默与退订变化。指标必须按优先级、租户、渠道和提供方拆分；HTTP 成功不能取代终端回执，更不能命名成用户阅读。

演练在平台已接收后丢弃响应，验证状态保持 `UNKNOWN`；再让延迟升级任务读取旧裁决后提交退订，验证 generation 提升、发送闸门 ACK 后该任务被 fencing，且不会调用短信提供方。过载顺序是先延后营销、聚合低优先、仅保留当前授权的 in-app，再限制新批量事件；法规必达意图也不能绕过适用的授权边界。恢复时先处理将过期的高优先意图，并对每个任务重新检查当前授权和用户是否已处理。

策略迁移先让评估器双读旧新版本，对小比例意图记录影子裁决差异，再启用新版本；回滚只影响未裁决意图，已外发尝试必须保持可审计。渠道密钥与正文分域加密，租户不能借共享配额淹没他人。成本按尝试、短信费用和用户疲劳共同归因，不以低基础设施成本掩盖退订。

## 面试考察本质

给定“同一业务提醒至多形成一个有效意图，且退订/撤权提交后旧 consent generation 不能越过发送闸门”的不变量，因为历史裁决不等于当前同意、外部渠道接受也不透露终端展示或人的注意力，候选人应把历史策略裁决、发送时授权、渠道尝试、可信回执和用户处理拆开；再依据漏报损失、打扰成本、过期边界与渠道健康选择冗余，不能照搬持久消息的已读语义。

优秀信号包括先问通知类型与同意、用业务键而非正文去重、诚实记录未知结果、按优先级隔离，并同时观察接受率和疲劳。常见误区是全渠道广播、把 HTTP 200 当阅读、失败就无限重试，或让渠道 worker 越过退订。

二十分钟回答应给出意图、策略、尝试三层状态和单渠道基线；四十分钟加入升级链、频控、过期与超时反例；六十分钟再讨论多租户配额、策略灰度、法规边界和用户疲劳。追问应持续要求说明“谁有权决定打扰、目前究竟知道哪一层成功、哪两个指标触发降级”。
