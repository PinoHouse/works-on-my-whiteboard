# 文本分享服务

## 表面题目

设计文本分享服务，表面流程是用户提交一段文字，得到 `paste_id`，再把读取地址分享给别人。真正的状态变化是“某个标识从未占用变为指向一个已发布的完整字节版本”，成功读取必须返回该版本的全部字节，或明确返回不存在、已过期、已删除，不能把截断正文和旧正文伪装成成功。热门公开文本、带密码的私密文本以及限时分享看似共用一个接口，撤销和缓存语义却完全不同。

题名容易掩盖两个边界。第一，正文创建后是否允许原地编辑；第二，所有者点击删除后，产品承诺多久停止从全球边缘读取。语法高亮、全文搜索和协作编辑不是核心目标，滥用举报也不能只做成异步旁路：一旦内容被判定必须撤销，读取边界就要执行该状态。本设计让已发布正文不可变，编辑产生新版本并原子切换入口，从而把字节完整性和入口可变性分开。

## 反问与边界

首先确认创建与读取契约：最大正文大小是多少，创建超时重试是否复用同一 `paste_id`，匿名与登录用户的保留期分别多长，过期后是立即不可读还是仅不再列出，删除能否恢复，私密内容是否允许 CDN 缓存。还要问读取 `p99`、撤销生效 SLO、峰值创建率 `W`、读取率 `R`、大小分布、最热对象占比和地域范围；日均流量不能描述一个地址突然在社交平台传播的扇出。

安全边界包括单租户字节配额、创建与枚举速率限制、恶意或非法内容扫描、加密与日志脱敏。正文可能携带密钥，访问日志不记录内容，私密 paste 的缓存键必须包含授权上下文或直接绕过共享缓存。未知生产数字只作为待校准假设：本设计暂定公开不可变正文可边缘分发，私密正文每次经过授权服务；异步清理只回收空间，不定义过期正确性。非目标包括多人实时编辑、把此服务当永久对象备份，以及保证外部举报系统永不误判。

## 客观模型

最小接口为 `Create(bytes, expiry, visibility, idempotency_key)`、`Read(paste_id, credential)`、`PublishRevision(paste_id, expected_version, bytes)` 与 `Revoke(paste_id, expected_version)`。权威元数据记录为 `(paste_id, active_version, expiry, visibility, deleted, revocation_version, owner)`，正文对象按 `(paste_id, content_version)` 不可变存储。元数据分片拥有入口、过期和撤销状态的写权限；正文对象只有在校验和匹配且元数据条件提交成功后才成为可见版本。边缘只持有带 `content_version`、`revocation_version` 和到期边界的副本。

核心不变量是：成功读取返回同一已发布版本的完整字节序列；已到期或已撤销入口不能在承诺边界之后被任何缓存重新暴露；新版本发布失败不得留下指向半写对象的入口；删除墓碑不能被延迟创建重试复活。设对象大小为 `S`，热门窗口请求数为 `R`，边缘命中率为 `h`，则源站出网字节约为 `E_origin = R × S × (1-h)`。一个十兆字节正文被读取十万次，即使 `h=0.99`，源站仍需发送约十吉字节；若每次更新影响 `K` 个已缓存副本，主动失效控制面工作量下界为 `Ω(K)`。

偏斜来自热对象和大对象的乘积，不能只看请求数。创建容量还要计入内容扫描、对象写放大和墓碑保留；删除不立即物理擦除是为了阻止旧入口复活，但会提高保留成本。状态所有权因此分为两层：对象存储证明某版本字节完整，元数据分片最终裁定哪个版本可读。

## 必然约束

[DEDUCED:pastebin-cache-without-revocation-version-can-serve-deleted-content] 假设边缘只按 `paste_id` 和十分钟 TTL 缓存正文。12:00 它填充 `p1 -> secret`，12:01 权威元数据提交删除，12:02 新读仍命中旧条目并返回 `secret`。边缘没有撤销世代，也没在读时比较权威状态，因此“数据库删除成功”无法推出“内容已不可读”，而失效消息丢失时甚至没有比 TTL 更强的可声明上界。结论是撤销必须成为版本化读契约；若业务只承诺 TTL 结束后生效，可以接受到期等待，但必须如实暴露该窗口。

[DEDUCED:pastebin-expiry-must-be-enforced-at-authoritative-read-boundary] 到期时间若只由后台扫描器物理删除，就存在扫描延迟和任务故障窗口。最小反例是入口在 13:00 到期，清理器因积压到 13:07 才运行，13:03 的权威回源仍从对象存储拿到正文。清理速度不是访问授权，因此元数据读路径必须以权威时间或明确误差界判断 `now >= expiry` 并拒绝；扫描器只负责降低存储成本。若产品定义“到期表示进入异步回收队列”而非不可读，才可放宽该结论。

[DEDUCED:pastebin-immutable-content-decouples-read-scaling-from-write-coordination] 已发布正文不可变时，缓存命中只需证明版本仍获准读取，同一字节可以独立复制；允许原地修改则旧字节和新字节在传播期同时存在，必须用新对象版本加元数据指针的条件切换决定哪个版本有效。不可变性由此把热门读取变为字节分发问题，而编辑能力必然付出入口协调、旧版本保留和失效成本。

## 从简单方案演进

最简单正确基线是一张元数据表加一个对象存储：先写临时对象并校验摘要，再条件占用 `paste_id`、提交入口；读取先校验元数据状态再取对象。它适合低流量，并把到期和删除放在一个权威边界。当连续十五分钟源站读 `p99 > 80 ms`，或 `E_origin` 超过该租户出网预算的 70% 时，为公开不可变正文引入区域缓存和 CDN；私密正文仍经授权。缓存降低回源，却新增热键、缓存穿透、撤销传播和大对象占用。

第二步让缓存条目携带内容版本、撤销版本和绝对到期边界，发布新版本时先写完整对象，再条件切换指针并发失效。若撤销传播 `p99 > 30 s`，而产品承诺十秒内停止读取，就停止把 TTL 当撤销机制，改为主动失效加边缘版本探测；超过承诺时宁可拒绝回源失败，也不返回已知旧版本。这里的 `80 ms/70%` 与 `30 s/10 s` 是两个待压测和故障演练校准的切换指标：调高前者能少用边缘资源但增加源站尾延迟和出网，调低后者能更早保护撤销 SLO 但增加控制面与版本探测 QPS。

第三步在对象总字节或元数据写入逼近单分片安全容量前按 `paste_id` 哈希分片。创建幂等键必须与最终 ID 一起原子保存；迁移期间旧分片只转发或拒绝过期纪元写，不能让两个分片分别发布同一 ID。滥用扫描在发布前给出同步底线，深度扫描异步执行；命中撤销后推进 `revocation_version`，不依赖对象物理删除。

未选择把正文直接放进关系库并让所有全球读取同步访问主库，因为热门大文本会把出网、连接和存储 IO 绑定在单一权威上。当文本始终很小、读取低频、撤销必须瞬时强一致且运维规模有限时，这一简单方案会重新变优。也未选择用正文哈希直接作为公开 ID，因为相同私密内容会泄露等同性，且删除后重新上传的授权语义无法只由内容哈希表达。

## 设计决定

本设计采用不可变正文对象与可变入口元数据分离。正常创建先验证大小、租户配额和幂等键，写入带摘要的临时对象，扫描通过后以唯一条件写提交元数据；超时重试命中同一幂等结果。读取先在授权边界确认可见性、到期和撤销世代，再返回匹配版本的缓存或对象；缓存版本落后时回源，权威元数据不可用时，公开内容只在未越过最大陈旧边界且无撤销迹象时降级，私密内容快速失败。

编辑不覆盖旧对象，而是创建 `v+1`，用 `expected_version` 原子切换 `active_version`。删除提交墓碑并增加撤销世代，失效事件可重复投递，旧事件不能覆盖更高世代。对象异步擦除只影响成本与隐私擦除完成时间，不影响读路径拒绝。对巨大正文实行分块校验与最大响应字节限制，但只有所有块摘要匹配时才发布。

反选方案是“边缘完全信任固定短 TTL，不维护版本”。它实现简单且控制面便宜，适合只读公开内容、没有紧急撤销承诺且 TTL 本身被产品接受的场景；一旦承诺删除或权限变化在 TTL 之前生效，该方案就不再正确。

## 运行与演进

核心 SLI 是有效读取成功率、读取 `p99`、源站出网字节、缓存命中率、撤销传播分位数、到期后成功读取事件数、对象摘要错误数和每租户未回收字节。过载时先停用预览、语法高亮和低优先分析，再限制匿名创建与超大正文；已发布正文读取优先。明确撤销和私密鉴权不因过载 fail-open，回源失败时也不能用无世代旧缓存冒充成功。

故障演练时间线：12:00:00 边缘 A、B 都缓存 `p1/v7`；12:00:05 元数据提交 `deleted=true, revocation_version=8`；12:00:06 失效发布节点宕机，A 已收事件而 B 未收。A 立即拒绝，B 在版本探测周期前最多按已声明旧窗口服务，窗口到达后必须探测或失败，不能继续返回 `v7`。12:01 发布器恢复并重放世代八，B 丢弃正文；随后延迟到达的世代七事件不得降低状态。演练核对用户可见字节、失效队列年龄和回源峰值。

新元数据格式先让读取兼容旧新世代，再灰度写新字段；回滚停止产生新格式但保留解析能力。跨地域复制分别观测元数据新鲜度和对象可用性，不能用“对象已复制”代替“撤销已传播”。租户删除需要覆盖正文、历史版本、索引和扫描样本，并保留不可读墓碑直到所有可能缓存越过最大生命周期。

## 面试考察本质

这题考察的是：给定“撤销或到期后不能再泄露已分享字节”的不变量，因为边缘缓存无法即时知道删除、权限和入口版本变化，候选人能否把不可变正文、版本化入口与权威到期判断分开，并在热门字节分发成本、缓存命中和撤销时效之间做出可量化取舍。

优秀回答会明确对象存储与元数据各自的写权限，先提交完整对象再发布指针，用 `E_origin = R × S × (1-h)` 解释热对象出网，并用十二点的删除反例击穿“数据库删了就安全”。常见误区是把后台清理当授权、让私密正文进入共享缓存、原地覆盖大对象，或只讨论高可用而不定义撤销传播上界。

二十分钟回答应完成接口、版本记录、不变量和单权威基线；四十分钟加入 CDN、撤销世代、过期判断与出网公式；六十分钟再讨论滥用扫描、隐私擦除、多地域传播和分片迁移。追问应固定在“失效事件丢失后谁还能返回哪个字节”，要求逐事件说明状态所有者、用户结果和切换指标，而不是罗列缓存产品。
