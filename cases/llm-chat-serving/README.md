# 大语言模型聊天服务

## 表面题目

设计大语言模型聊天服务，表面流程是用户提交一段对话并持续收到生成文本。真正的系统状态包含会话版本、输入令牌、模型与提示模板版本、推理队列中的资源预留、逐步产生的输出以及终止原因。成功不只是接口返回二百，而是首个令牌在可接受时间内出现、流式连接中断后不会把两次生成误拼成一次，并且事后能解释这次输出由哪套模型和配置产生。

题名容易把概率模型当作普通无状态 HTTP 处理器。两个字节数相同的请求可能有完全不同的输入令牌数和输出长度，同一个提示也可能因采样或执行内核差异得到不同答案。聊天历史是否由服务端拥有、是否允许工具调用、输出安全策略在哪一步执行，会改变故障和重试语义。本设计聚焦多轮文本生成与流式返回，工具执行、训练和向量检索另设边界。

## 反问与边界

先澄清质量和体验：首令牌延迟、令牌间延迟、端到端完成时间分别有什么 SLO，用户能否接受排队，最大上下文与最大输出是多少，取消后多久必须释放 GPU。吞吐要用每秒输入与输出令牌以及序列长度分布表达，不能只问每秒请求数。还要问租户优先级、每用户并发、突发额度、模型大小、精度格式、GPU 类型与地域，以及质量回归由哪些固定评测集和在线风险指标判断。

语义边界包括：服务端保存完整历史还是客户端每次提交，重试同一请求需要复用原输出、重新采样，还是只保证不会重复计费；温度、随机种子和模型版本能否由调用方指定；内容审核是在输入、增量输出还是完整输出上执行。隐私上需要定义对话保留、训练退出、调试日志脱敏、租户适配器隔离和删除传播。多地域故障时可以切到较小模型吗，若切换必须向用户标记质量等级。非目标是承诺模型事实正确，也不把保存种子等同于跨硬件逐字节复现。

## 客观模型

最小接口为 `CreateTurn(conversation_version, messages, generation_config, idempotency_key)`、`Stream(turn_id, attempt_id, offset)`、`Cancel(turn_id)`。核心 turn 记录保存 `(turn_id, conversation_version, model_digest, tokenizer_version, template_version, safety_version, sampling_config, seed, execution_profile, active_attempt, status, finish_reason)`，attempt 记录保存 `(attempt_id, lease_epoch, status, output_chunks, kv_rng_checkpoint)`。会话存储拥有已提交轮次顺序；调度器拥有待执行与已预留令牌预算；单个解码工作器只在租约内拥有某次 attempt，租约世代只防止同一 attempt 的旧工作器继续提交，不能把没有完整 checkpoint 的重生成伪装成原概率路径的续写。

设输入令牌数为 `I`、允许最大输出为 `O`、批内并发序列数为 `B`。预填充计算大致随输入长度增长，而解码阶段每生成一个令牌都要读取模型权重并维护注意力状态；KV cache 的主要容量可近似写成 `B*(I+O)*L*D*b` 的比例量，其中 `L` 是层数、`D` 是每令牌每层缓存宽度、`b` 是字节系数。即使不依赖精确常数，也能看出一个八千令牌请求和一个一百令牌请求不能按“一次请求”计同样成本。

不变量包括：同一会话版本只能提交一个后继轮次；每个输出分片带 attempt 与单调 offset，断线续传不能跨 attempt 拼接；只有 `OPEN` attempt 的当前租约世代可以提交，关闭后的 attempt 永不重新打开；实际生成不能越过已批准的上下文和输出上限；每次结果保留完整版本谱系。只有同时保存下一 token 所需的完整 KV cache、RNG 状态与确定执行位置时，换租约世代才可在同一 attempt 的下一个 offset 恢复；否则必须关闭旧 attempt 并从 offset 零创建新 attempt。长提示、长输出和少数高优先租户形成偏斜，连续批处理若只追求总吞吐，后来到达的短请求可能长期等在大批次之后。

## 必然约束

[DEDUCED:llm-chat-serving-token-budget-must-govern-admission] 推理成本随输入与输出令牌数显著变化，只按请求数排队无法同时控制显存占用、公平性与尾延迟。最小反例是一块只能安全容纳一万令牌 KV 预算的设备：队列看到十个请求并允许全部进入，其中九个各需一百令牌，一个允许九千五百令牌，总预算达到一万零四百而溢出；若十个都是一百令牌则远未饱和。请求计数相同、资源结果不同，说明准入至少要估计输入令牌、输出上限和当前缓存占用。估计可以保守且在运行时修正，但不能把未知长度当零成本。

令牌预算还不等于简单先到先服务。长输出若一次预留全部上限会造成容量闲置，完全不预留又可能在生成中途因显存耗尽杀死已开始的响应。可采用输入精确计量、输出分段信用和硬上限，调度器在每轮解码后更新实际消耗。业务公平可按租户令牌债务而非完成请求数计算；否则一个租户发送大量短请求会占满批次槽，另一个长请求则因单次权重相同而获得不成比例资源。

[DEDUCED:llm-chat-serving-reproducibility-requires-versioned-provenance] 概率生成的可复现性依赖模型、分词器、提示模板、采样参数与执行环境的联合版本，单独保存用户文本不足以解释输出差异。事件反例是同一文本先由分词器甲切成一百二十个令牌，升级后由乙切成一百二十三个；即便模型权重未变，位置和采样分布已经改变。即使固定种子，批处理顺序、数值精度和非确定内核也可能改变低位概率并在后续令牌放大。因此谱系能支持归因与同环境重放，但只有确定执行配置和受控硬件才可承诺逐字一致。

## 从简单方案演进

最简单基线是一张 GPU 上一次只运行一个请求，完整生成后返回。它易于隔离和重放，却浪费预填充与解码之间的计算空隙，长请求阻塞所有短请求。当 GPU 计算利用率低于百分之四十而队列等待 `p95` 已高于执行时间，先引入静态批处理；它提高吞吐，但必须等待同批请求凑齐，并让短序列被最长序列拖累。

第二步采用连续批处理：序列可在解码轮次边界加入和离开，调度依据当前 KV cache、预填充预算和租户信用。它解决批次空洞，新增调度抖动、缓存碎片和租约恢复。切换条件可以量化为首令牌 `p99` 超过 SLO 的百分之八十，或因预算估计不足导致的准入后驱逐超过万分之一；此时应分离预填充与解码队列、按长度分桶，或为交互流量保留容量，而不是继续堆积请求。

当单模型副本无法承担峰值时复制工作器，路由器按模型摘要、适配器与可用令牌预算选实例。热门模型可多副本，小模型不能与大模型共享一个无约束队列。若模型加载时间 `p95` 超过可接受恢复时间的一半，保持热副本或分层权重缓存；代价是昂贵显存与版本驻留。跨地域故障时只有目标区域已经加载相同模型摘要与安全版本才可无感切换，否则明确降级到另一质量等级。本节的百分之四十利用率、SLO 百分之八十、万分之一驱逐率和恢复预算二分之一都是待压测校准的初始策略参数：百分之四十加等待超执行时间用于识别批处理空洞，百分之八十给突发与网络留尾延迟余量，万分之一把已准入后杀死响应视为高代价事件，半个恢复预算给路由和预热留空间。调高会延后批处理、分队列或热副本投入但扩大排队和中途驱逐，调低会更早保留昂贵容量并可能降低 GPU 利用率。

未选择“所有请求到达即送入任意空闲 GPU”的方案，因为它忽略模型驻留、序列长度和租户公平，容易造成权重反复加载与显存溢出。若模型单一、请求长度上限很窄且负载远低于容量，这个简单路由会重新变优。也未选择为逐字重现永久固定整个硬件栈，因为成本与安全升级受阻；只有审计重放租户才使用隔离的确定执行池。

## 设计决定

本设计先用确定的模型摘要、分词器和模板把请求转换为输入令牌，再由令牌预算调度器准入。交互请求获得有上限的预填充和分段输出信用，连续批处理在每轮解码更新实际 KV 占用；租户采用加权令牌公平并设并发上限。工作器得到某个 attempt 的带世代租约，输出分片以 `(turn_id, attempt_id, lease_epoch, offset)` 条件追加；更高租约世代只能 fence 同一 attempt 的旧工作器。attempt 之间没有可继承的 offset 或概率前缀，turn 只把一个 `OPEN` attempt 标为当前可见。

流式连接断开不自动重启推理，客户端以 `(attempt_id, offset)` 续读已提交分片；只有明确取消才推进状态并回收资源。工作器故障时，若没有包含完整 KV cache、RNG 状态和执行位置的 checkpoint，状态存储以一个原子事务把旧 attempt 从 `OPEN` 改为 `FAILED_CLOSED`、撤销其当前可见资格并创建从 offset 零开始的新 attempt；客户端收到 attempt 变更后必须开启新的输出段，不能把相同文字前缀当作旧路径续写。只有完整 checkpoint 存在时，才可提高同一 attempt 的租约世代并从已提交 offset 后恢复。谱系相同只支持归因，不保证新采样与旧 attempt 逐字相同。幂等键防止网络重试重复计费与重复创建轮次，同一键带不同消息摘要时拒绝。

安全检查先过滤明显非法输入，再对增量输出保留可撤销缓冲窗口；严重风险触发终止原因而不是伪装成正常结束。过载时先拒绝低优先级长上下文、降低新请求最大输出，再把明确同意降级的请求路由到较小模型。反选的“自动把所有失败请求切小模型”虽然可提高成功率，却改变质量和行为边界；只有调用方在契约中允许该等级，路由器才执行。

## 运行与演进

核心 SLI 是首令牌延迟、令牌间延迟、完整生成延迟、每秒输入与输出令牌、排队令牌秒、KV cache 占用、准入后驱逐率、取消资源释放时间和流中断续传成功率。质量侧按模型摘要观察安全拦截、用户重试和固定评测回归，不能用吞吐掩盖输出退化。成本以每千有效输出令牌的 GPU 秒衡量，并区分被取消与被安全拦截的浪费。

故障时间线如下：零毫秒 attempt 七由持有租约世代三的工作器甲生成到 offset 四十，但只持久化到三十二且没有完整 KV/RNG checkpoint；五十毫秒甲失联，六十毫秒客户端重连只能读到 attempt 七的 offset 三十二。租约到期后，状态存储原子关闭 attempt 七并创建 attempt 八；乙取得 attempt 八的租约世代一，从 offset 零重新采样，客户端明确开始一段新输出。甲在网络恢复时提交 attempt 七的 offset 三十三或四十一都因 attempt 已关闭而被拒绝；给 attempt 七提高租约世代也不能把乙的新概率路径接在旧前缀后。对照演练若提供 offset 三十二之后的完整 checkpoint，才允许乙用 attempt 七的世代四从 offset 三十三恢复。演练验证两条路径从不拼接，计费只归属于最终可见 attempt。

模型升级先固定评测，再让新摘要影子运行并比较安全、质量、首令牌与单位令牌成本；小流量灰度时每个 turn 永久绑定一个摘要，不能在会话中途偷偷换权重。回滚保留旧分词器、模板和安全配置读取能力。隐私删除需要覆盖会话、输出分片、调试样本和派生缓存；多租户适配器按授权与显存地址空间隔离，调度指标不得暴露别的租户提示长度细节。

## 面试考察本质

给定“已开始的生成必须在有界资源内形成一条可追踪、不会被旧工作器混写的输出序列”这一不变量，因为输出长度在准入时未知、GPU 显存稀缺且概率执行不能仅凭用户文本解释差异，候选人应推导出令牌预算、带世代的执行权和完整版本谱系，并承认吞吐、首令牌尾延迟、质量与逐字复现不能同时最大化。业务应按交互体验、审计要求和每令牌成本决定取舍。

优秀信号包括把预填充和解码分开估算、用 KV cache 而非请求数做准入、讨论连续批处理中的短长请求公平、为流分片设计 offset 与 attempt，并准确区分“可归因重放”和“逐字确定”。常见误区是只说横向扩 GPU、把温度设零当作跨环境确定性证明、断线后无条件重跑并拼接输出，或在过载时静默换模型。

二十分钟回答应给出 turn 记录、令牌容量公式和单 GPU 基线；四十分钟加入连续批处理、公平调度、流式重连与故障世代；六十分钟再讨论模型驻留、版本灰度、安全缓冲、隐私删除、质量评测与成本。追问可用“一个九千五百令牌请求加九个短请求”检验准入，用“offset 四十处工作器失联”检验输出所有权，再用分词器升级检验候选人是否真正理解概率服务的可复现边界。
