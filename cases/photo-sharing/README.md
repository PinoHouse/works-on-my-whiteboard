# 图片分享

## 表面题目

设计图片分享服务，表面流程是用户上传原图、选择可见范围，其他用户随后在相册或瀑布流中看到缩略图。真正的状态变化是不可变对象字节与可变元数据建立绑定，审核和隐私状态推进版本，再由异步任务生成多尺寸、多格式、OCR、向量和 CDN 副本。上传成功需要明确是“原图已持久保存”还是“某个可展示版本已审核并可读”。

图片流的展示顺序只是派生布局或相关性 rank；上传时间、对象摘要和审核裁决才是权威事实。本设计关注单张媒体的版本、派生与撤回，不把视频长转码、社交关注图扇出或完整推荐模型混作图片存储的核心。

## 反问与边界

先问原图大小、像素分布、格式、每日上传率与热点下载比例，再问首张安全缩略图、完整变体和审核完成各自的 SLO。覆盖上传是否创建新 asset，私密切公开或公开撤回多久必须停止读取，旧分享链接能否失效，用户删除是逻辑隐藏还是要求对象和备份按期清除。排序可以陈旧，但审核与隐私不能被排序分抵消。

还要确认所需尺寸与编码集合、区域副本、OCR/向量是否允许延迟、审核失败是否保留原图供申诉。多租户配额按原始字节、派生字节和出网分别计算；签名 URL 不记录在日志，图像元数据中的位置和设备信息默认剥离。非目标是用 CDN 命中证明授权、为每位关注者复制一份图片字节，以及承诺撤回已经保存到用户设备的内容。

## 客观模型

最小命令为 `CreateAsset(upload_digest, metadata, privacy)`、`Publish(asset, expected_version)`、`ChangePrivacy(asset, expected_version)`、`GetRendition(asset, kind, viewer)`。对象存储拥有不可变 `object_digest -> bytes`；元数据库拥有 `asset -> (owner,current_digest,moderation_state,privacy_version,status)`；派生服务只拥有 `(digest,privacy_version,recipe_version) -> rendition`。相册成员序可由相册 owner 权威追加，瀑布流布局、视觉簇和热度都是派生 rank。

不变量是每个可见派生物都能追到当前获准源摘要；审核拒绝或隐私撤回后不能由旧缩略图、长 TTL URL 或视觉索引继续授予访问；迟到任务不得覆盖新版本；展示排序不改变上传时间和审核裁决。物化发生在上传后异步管线，而非按关注者扇出。

若变体 `i` 的字节为 `s_i`、重试与区域副本因子为 `r_i`、索引字节为 `B_index`，单图派生写量近似 `B_derive=Σ(s_i*r_i)+B_index`；上传率 `λ` 需要处理带宽约 `λ*B_derive`。热点照片的读出网约为请求字节乘缓存未命中率，和上传派生是两条不同放大链。

## 必然约束

[DEDUCED:photo-sharing-rendered-bytes-must-bind-to-an-immutable-source-digest] 若输出 key 只含 `asset_id` 和尺寸，旧转码任务无法知道源已被替换。最小反例是 asset 7 的 D1 在 10:00 入队生成 200px 图，10:01 因隐私问题撤回并重传 D2、版本推进到 4，10:02 D1 慢任务才写入 `asset/7/200.jpg`。它会覆盖 D2 或重新进入 CDN。把输出绑定 `(D1,v3,recipe)` 并在发布指针时校验当前 `(D2,v4)`，旧任务只能留下不可见垃圾。若产品禁止同一 asset 换源，也仍需摘要验证损坏与重试。

[DEDUCED:photo-sharing-thumbnail-cache-is-not-an-authorization-authority] 缓存只知道某 URL 曾对应字节，不知道 viewer 当前权限、审核撤回或分享成员变化。一张公开图片在版本 3 发出长 TTL bearer URL，随后版本 4 改为私密；即使新 URL 改用 v4 命名，持有 v3 URL 的人仍能直接读取旧对象。版本化命名只能防止新旧字节共用缓存键，不能单独撤销已签发 URL。用户私密媒体必须把它与短 TTL、每次在线鉴权/版本探测、可撤销令牌或主动 purge 组合，撤回承诺由这些机制的最慢边界决定；无限缓存和立即撤回不能同时拥有。公开且明确不可撤回的静态素材才可放宽检查。

[DEDUCED:photo-sharing-derivative-count-not-follower-count-dominates-upload-amplification] 一次上传即使没有关注者，也可能生成五种尺寸、三种编码、两区域副本、OCR 和向量索引；其写量由这些分支相加。具体反例是一张 20MB 原图产生 15 个各 1MB 的变体并复制两区，仅变体就写 30MB，再加索引；关注者为零并未令成本为零。反之一千万关注者读取同一公开缩略图可由 CDN 共享，不需要一千万份写入。因此容量首先按派生 recipe 和字节规划，而不是套用社交 feed 的 fan-out。

## 从简单方案演进

最简单正确基线是保存原图与元数据，读取时同步生成一个固定缩略图并短期缓存。它适合低上传量，却让大图解码进入用户关键路径。当最小可展示缩略图延迟接近上传体验预算时，将安全低清版本内联生成，其余尺寸、编码和索引异步物化；新增任务幂等、版本 fence、积压和中间对象清理。

第一个待压测校准指标是 `PENDING` 媒体的最小可展示缩略图 `p95` 超过 `3 s`，且审核策略允许受限预览：内联一个低清版本并为剩余 recipe 排优先级；若审核 backlog 超过 `10 min` 则仍不公开。第二个待故障演练校准指标是撤回后 CDN 仍命中比例超过 `0.01%`，或吊销传播 `p99` 超过 `30 s`：缩短并版本化签名 URL、增加在线版本探测和主动 purge。两组数字只是初始策略；调低会增加回源、签名与计算成本，调高会扩大等待或隐私暴露窗口。

图片量增长后按 digest 分片对象、按 asset owner 分片元数据，派生队列按像素工作量和租户信用调度，不能让巨幅图片饿死普通上传。recipe 升级以新 key 并行生成，读指针灰度切换；原结果在引用和回滚期限结束后再回收。

## 设计决定

本设计先把原图以内容摘要不可变写入，校验完整性后原子创建 asset 元数据；审核通过才发布可见指针。每个任务携带源摘要、隐私版本和 recipe 版本，条件发布失败时不覆盖当前结果。读取先检查 viewer 与元数据当前版本，再签发短期对象令牌；CDN 缓存字节但无权扩展令牌范围。版本化对象名只隔离缓存键，撤回还必须停止签发旧版本令牌，并依赖令牌短 TTL/在线版本探测与主动 purge 收敛既有访问。

转码超时可按同一任务键重试，输出内容寻址使重复结果可合并。权威元数据不可达时，私密或审核状态未知的读取失败关闭；公开且契约允许短陈旧的版本可在明确时间界内服务。下架优先推进版本与停止新令牌，purge 只是缩短残余窗口。

反选“上传时同步生成全部尺寸和索引”，在尺寸固定、上传极低且创建响应必须保证全部变体立即可用时会重新变优；通常它把慢 recipe 和审核故障绑到上传成功上。也不选择仅靠永久公开对象 URL，因为它无法为撤回提供可声明边界。

## 运行与演进

SLI 包括原图持久化成功率、首个安全缩略图延迟、各 recipe backlog 年龄、源摘要不匹配拒绝数、撤回后新令牌签发数、CDN 旧版本命中、出网字节和每租户派生放大。过载时先延后相似搜索与高成本格式，再减少非必要尺寸；审核、隐私 fence 和最小安全预览优先。

故障演练时间线：0 秒 D1 元数据为 `PENDING/v3`，5 秒低清生成，8 秒审核拒绝 D1 并推进 `REJECTED/v4`，9 秒用户用 D2 覆盖上传并条件推进为 `PENDING/v5`，10 秒旧转码器才完成 D1，12 秒 D2 审核通过并推进 `PUBLISHED/v6`。D1 任务因 `(D1,v3)` 不等于当前 `(D2,v5)` 而不能发布；恢复只重放 D2 的当前版本任务，发布时再次以 v6 条件校验。对象网关停止签发 D1/v3 令牌，已签发 bearer URL 即使带 v3 名称也只能按短 TTL/吊销与 purge 边界收敛；演练核对 CDN 中 D1 不能经新鉴权请求访问。

recipe 与审核模型升级先影子比较字节摘要、质量、安全和单位像素成本，再灰度读指针；回滚保留旧 recipe 读取。隐私删除覆盖元数据、派生物、视觉索引和调试样本，备份按声明期限清除。不同租户的私密对象、签名密钥和配额隔离，热点公开图可共享缓存但不能共享授权上下文。

## 面试考察本质

给定“任何可见派生字节都必须属于当前获准源图片版本”这一不变量，因为转码器和 CDN 不会立即知道审核、隐私或重传变化，候选人应推导出不可变对象、可变元数据与带版本派生物的分层，并在首图延迟、派生写放大、缓存命中和撤回时效之间取舍。

优秀回答会写出 `Σ(s_i*r_i)`，把状态 owner 放在对象与元数据而不是 CDN，用 D1 慢任务覆盖 D2 的最小反例说明摘要 fence，并解释瀑布流 rank 可重算而审核裁决不可。常见误区是按关注者复制字节、用长 TTL URL 做私密授权，或让转码成功反向改变审核状态。

二十分钟完成对象/元数据/派生三层和上传流程；四十分钟加入像素成本、异步任务、撤回时间线与签名 URL；六十分钟再讨论 recipe 迁移、索引、热点出网、隐私删除和租户公平。追问始终落在“此字节由哪个摘要生成、哪个版本允许它公开、旧任务为何不能覆盖”。
