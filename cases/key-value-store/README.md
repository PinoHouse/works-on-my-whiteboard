# 键值存储

## 表面题目

设计键值存储，表面上只有按键写值、读值和删除三个动作。真正要交付的是一套可声明的状态语义：一次写入何时算成功，成功后遭遇哪些节点或机房故障仍必须找回该版本，并发写入如何排序，分区迁移时谁有权接受新版本。调用方依赖的不是“有若干副本”，而是确认响应与后续可见性之间的明确承诺。

题目容易把缓存和持久存储混在一起。若数据丢失可以由上游重建，淘汰与短时旧读是容量策略；若确认写入代表用户资产已经保存，副本落盘、恢复点和删除墓碑都进入正确性边界。本设计以持久键值存储为目标，支持单键条件写与删除，不承诺跨键事务、任意查询或把二级索引隐藏进基础接口。

## 反问与边界

先问值大小分布、键数量、读写比例、热点键比例、峰值吞吐和数据增长，再问真正的故障声明：确认后允许丢一个进程、一台机器、一个可用区还是整个地域；恢复点目标和恢复时间目标分别是多少。读取需要线性一致、读己之写、单调读还是最终一致，并发写是条件版本、最后写胜还是保留兄弟版本。删除是否必须阻止离线副本复活旧值，快照和备份保存多久，这些都比“复制三份”更能决定架构。

地域与成本边界也需显式。同步跨区往返是否落入写延迟预算，磁盘介质是否保证写入完成才确认，压缩与合并可使用多少后台带宽，单租户能否制造超大值或热点。键和值可能包含敏感信息，鉴权必须绑定命名空间，传输和静态存储分别加密，日志只记录长度、哈希与版本元数据。未知生产数字以 `Qw`、`Qr` 和分位数表示；非目标是跨键外键、全文检索以及在网络分区中同时承诺零写失败和单一最新值。

## 客观模型

最小接口为 `Get(key, consistency)`、`Put(key, value, expected_version, idempotency_key)`、`Delete(key, expected_version)`。逻辑记录是 `(namespace, key, version, value_or_tombstone, checksum)`，版本由当前分区所有者按提交顺序分配。每个分区的共识日志记录键范围、复制组配置和单调纪元 `epoch`，元数据服务只发布已提交配置的路由。复制组领导者拥有当前已提交配置内的写序；纪元只是配置版本标签，只有共识提交规则才能赋予或撤销确认权。

核心不变量是：同一键已提交版本形成可比较前缀；条件写只在预期版本匹配时成功；一次确认达到所声明的持久副本范围；墓碑在所有可能携带旧值的副本完成修复前不能回收；任何确认写的 quorum 都与上一份有效配置的提交 quorum 相交。若复制因子为 `N`，写确认需要 `W` 个副本、读查询 `R` 个副本，仅凭 `W+R>N` 可以保证单一静态配置内集合相交，却不能自动证明返回最新值或配置变更安全，还需要版本比较、已提交边界与联合共识。

容量由用户字节和放大共同决定。设逻辑写入速率为 `B` 字节每秒、复制因子为 `N`、日志与压缩写放大为 `A`，底层持续写带宽至少约为 `B*N*A`。一个一兆字节热点值每秒更新一千次，即使平均键很小，也会对同一复制组制造约一吉字节每秒的逻辑入口并让副本追赶排队；按键数均匀分片无法消除单键串行化下界。

## 必然约束

[DEDUCED:key-value-store-acknowledgement-bounds-declared-durability] 假设接口向调用方声明“任意一台机器故障后已确认写仍存在”，那么返回前必须至少有两个故障独立的持久副本掌握该提交。最小事件序列是：领导者只写入内存便在十毫秒确认版本五，二十毫秒进程崩溃，其他副本仍停在版本四；客户端已经无法区分“写成功后丢失”和“从未成功”。增加异步复制速度不能消除确认与复制之间的窗口。若契约只承诺进程存活期间的缓存写，早确认可以成立，但不得把它描述成持久写。

确认数量还必须映射到故障域，而非只数进程。三个副本若位于同一块掉电域，并不支持可用区故障承诺；一次快照成功也不等于日志已复制到恢复点。由此写延迟的物理下界至少包含最慢必需故障域的持久化往返。业务可选择较弱确认以换延迟，但响应必须携带或固定对应的一致性等级，不能在过载时悄悄把同步持久化改成异步。

[DEDUCED:key-value-store-partition-epoch-fences-stale-owners] 分区从复制组甲迁到乙时，网络分区可能让甲看不见新配置、乙看不见甲的最后写。若两组都凭“我曾是所有者”接受请求，则甲确认版本十一写入红，乙也确认版本十一写入蓝；恢复后没有顺序能让两个客户端都观察到自己所承诺的单一历史。本设计采用联合共识完成撤权：先在旧配置 `C_old` 的日志中提交联合配置 `C_joint`，联合阶段的每个提交都必须分别得到 `C_old` 与 `C_new` 的多数确认；目标成员追平已提交前缀后，才提交最终配置 `C_new` 并开放新组确认。两个旧配置多数必然与提交 `C_joint` 的旧多数相交，因此已接受联合配置的成员不会再帮助孤立旧领导者提交。若拿不到 `C_old` 多数，就无法证明旧 quorum 已失去确认权，迁移必须停住，新组不得凭更大的 epoch 开始确认。若应用接受多值并由业务合并，双写可以被建模，但那已不是本题声明的单值条件写。

## 从简单方案演进

最简单正确基线是一台机器的追加日志加内存索引：写先追加并按契约刷盘，再更新索引；恢复重放日志。它容易给出单键顺序，但容量、吞吐和机器故障都受单点限制。当磁盘占用超过可管理容量的百分之七十，或恢复重放时间超过恢复时间目标的一半时，先做快照与日志分段；当任何单机故障都不能满足可用性目标时，再引入同分区复制，而不是为了“分布式”提前叠加协议。

复制组将领导者的有序日志复制到多个故障域。写提交延迟、未提交日志长度和副本追赶时间是切换指标：若领导者磁盘利用率持续高于百分之七十，或热点之外的整体写入已达到单组安全吞吐的百分之八十，就按键范围或哈希分区。分区解决总容量与并行吞吐，新增元数据路由、跨分区均衡和联合配置迁移；分区数增加还会放大连接数、后台压缩和故障恢复并发。本节的两个百分之七十、恢复时间目标的一半和百分之八十都是待压测校准的初始策略参数：容量百分之七十为快照临时空间留余量，半个 RTO 避免恢复重放独占全部预算，磁盘百分之七十与安全吞吐百分之八十为副本追赶和领导者切换保留余量。调高会提高资源利用率但缩小恢复、追赶与突发空间，调低会更早快照或分片并增加存储、连接和迁移成本。

读取可从领导者获得最新已提交值，也可从追随者换低延迟和读扩展。只有当追随者应用延迟 `p99` 小于声明的陈旧界，且读流量使领导者利用率越过阈值时才开放追随者读；否则“读副本”只是把未知旧值暴露给调用方。若跨地域写 `p99` 超过业务预算而法规允许每区独立写，可改成多值冲突合并模型，但必须重新定义版本偏序和业务合并，不能继续声称线性单值。

未选择无领导的任意读写 quorum 作为默认方案。它能分散写入口，却把读修复、并发版本、提示移交和配置变化的交互暴露给所有请求；在写可用性高于单键线性顺序、值具备可靠合并函数时它会重新变优。当前条件写和严格确认更适合每分区一个明确提交权威。

## 设计决定

本设计让分区共识日志拥有带纪元的成员配置，元数据服务只缓存和发布已提交的路由；每个分区由跨故障域复制组持有，当前领导者排序单键写。正常写路径先校验命名空间、大小、幂等键和 `expected_version`，再追加当前配置日志；达到该配置要求的持久 quorum 后标记提交并响应。迁移严格执行 `C_old -> C_joint -> C_new`，联合阶段同时取得新旧多数，在最终配置提交前新组不对客户端确认。超时后的重试用幂等键返回原提交结果，版本冲突返回当前版本，不把冲突当网络错误无限重试。

读默认从当前领导者读取已提交前缀；明确接受陈旧的调用方可读取满足最大应用延迟的追随者。领导者失联时停止强一致写，只有当前已提交配置能选出掌握提交前缀的新领导者后才继续；它不会让孤立旧领导者或尚未完成联合变更的新组为了可用性确认分叉。删除写入墓碑版本，反熵确认所有副本越过墓碑且超过离线副本最长保留期后才能压缩。

一个被拒绝的方案是“领导者本地写成功即响应，后台尽快复制”。它适合可重建缓存，却与本题机器故障后保留确认值的契约冲突。若产品后来按命名空间提供 `ephemeral` 等级，该路径可以作为明确的新写模式，但指标、响应和恢复说明必须与 durable 等级分开，不能用同一个成功码掩盖两种含义。

## 运行与演进

运行 SLI 不只看请求成功率，还包括已确认写丢失计数、版本回退计数、条件写冲突率、提交延迟、追随者应用延迟、未提交日志字节、分区热点、压缩债务和按声明故障域计算的健康副本数。过载时先暂停低优先级扫描和反熵加速，限制大值与后台迁移，再对写入做租户公平排队；不得通过减少持久确认数维持表面成功率。读可按契约降级到明确陈旧模式，强一致调用方则快速失败。

故障演练时间线如下：零毫秒时 `C_old` 的组甲提交版本二十；一百毫秒领导者提出包含组乙的 `C_joint`，但在取得甲的多数确认前网络隔离甲；一百二十毫秒乙即使拥有更大 epoch 也不得确认版本二十一，甲的少数分区同样不能确认。网络恢复后，`C_joint` 先由甲、乙各自多数提交，乙追平版本二十，再提交 `C_new`；路由刷新后同一幂等写才由乙确认版本二十一。演练还要模拟联合配置已提交后孤立旧领导者恢复，验证它无法获得旧配置多数、只能作为数据源比较校验和，并断电一台已确认副本验证剩余故障域仍能读到版本二十一。

协议升级先让节点理解新旧记录格式，再写双读兼容的版本，按分区灰度并比较校验和与提交索引；回滚只停止产生新格式，必须保留读取能力。分区迁移要求旧组在线到足以提交 `C_joint`，并在联合阶段为每个提交同时等待新旧多数，因此会增加一次配置往返、跨组复制带宽和旧组保留时间；异常发生在 `C_new` 提交前可继续按联合规则恢复，提交后则只能以新配置前进，不能把路由 epoch 倒回旧组。若旧 quorum 永久丢失，必须走独立灾难恢复流程并接受契约定义的数据或可用性后果，不能伪装成普通迁移。备份恢复需定期在隔离环境验证恢复点，而非把“对象存在”当成可恢复证据。

## 面试考察本质

给定“成功响应必须对应一个在声明故障范围内仍存在、且单键历史不分叉的版本”这一不变量，因为节点在分区时无法立即知道自己是否仍是所有者、客户端也无法从成功码看出副本是否落盘，候选人应推导出确认延迟、故障独立副本数与可用性不可同时无限优化，并用业务的丢失成本和陈旧容忍度选择确认与读取等级。

优秀回答会把 durability、consistency 和 availability 分开定义，指出 `W+R>N` 只是静态配置内的相交条件，画出确认前后崩溃反例，并用联合共识说明分区所有权如何真正撤权。它还会计算复制与压缩放大，承认热点单键不能靠普通分片并行。常见误区是说“三副本所以安全”却不说明落盘和故障域，把最后写胜交给未同步墙钟，或只递增路由 epoch 就让源和目标先后可写。

二十分钟应完成接口、记录、确认语义和单机日志基线；四十分钟加入复制组、读等级、墓碑与故障恢复；六十分钟再讨论纪元迁移、热点、压缩债务、跨地域和备份验证。追问应始终落到事件顺序：响应在哪一步发出，哪一个版本已提交，旧所有者凭什么失去写权限，恢复后哪个状态可以覆盖哪个状态。能够回答这些，才说明候选人在设计存储而不是罗列存储组件。
