# 复式记账账本与钱包

## 表面题目

设计复式记账账本与钱包，表面上是查询余额、充值、转账和消费。核心是每次资金变化都由一组不可变、零和的 posting 解释，任一时刻展示的可用余额不能承诺超过已入账资金减去有效 hold。钱包的“余额”只是读模型，不是可以随意覆盖的真相；外部支付成功也不是账本两边已经完整提交的证明。

本设计覆盖同一账本内的分录、账户余额投影、资金 hold、冲正和审计。跨行最终性由转账案例讨论，汇率撮合和会计报表不是主流程。目标是崩溃后仍能从 journal 重建并解释每一分钱，而不是只让某个缓存数字看起来正确。

## 反问与边界

先确认账户类型、币种和精度，是否允许负余额、透支或信用额度，posted、pending、held 和 available 各向谁承诺；分录是一笔业务事件一组还是允许拆批，纠错使用冲正还是调整，账本关闭期能否回写。充值与提现的外部最终性、手续费和退款决定需要哪些中间账户。

规模要看业务事件率、每事件 posting 数、热账户偏斜、余额查询率、历史保留和重放时间，而非只看钱包数。延迟目标应分原子入账和投影可见。租户、法人实体、币种和账本日构成隔离边界；访问控制要防止操作员直接改余额。非目标是用数据库约束自动证明所有业务科目选择正确。

## 客观模型

最小命令为 `Post(event_id, postings[])`、`PlaceHold(account, amount, version)`、`ReleaseHold`、`CaptureHold(journal_entry)` 和 `GetBalance(account, watermark)`。journal service 拥有不可变 entry 与 posting，账户序列拥有可用余额条件，projection service 只拥有派生读取进度，外部渠道拥有充值或提现事实。分录状态从 PREPARED 到 POSTED；错误通过 REVERSAL 或 ADJUSTMENT 引用原分录。

对任一 entry 和币种 `c`，必须满足 `Σ signed_amount(p,c)=0`。若事件率为 `λ`、每事件平均 `m` 条 posting、保留期 `T`、每条平均字节数 `b`，原始日志量为 `V=λ×m×T×b`；索引、余额投影和副本还会乘上写放大。热点通常是平台清算账户或大商户，不由总账户数均匀摊薄。

不变量还包括同一 event_id 至多提交一组语义相同的分录，available 等于 posted credit 减 posted debit、有效 hold 与必要保留，capture 只能消费仍有效 hold。journal 提交是资金事实线性化点；余额缓存带 watermark，不能接受写入。

## 必然约束

[DEDUCED:double-entry-ledger-wallet-atomic-balanced-postings-are-the-source-of-money-conservation] 借贷两边必须作为一组提交，才能从任一已提交前缀证明零和。最小反例是钱包 A 有 100 元，事务 X 单独写 `A:-70` 后进程在写 `platform:+70` 前崩溃；投影显示 A 为 30，而总账凭空少 70。重试完整转账又可能让 A 变成负 40。只有原子接受平衡 posting，或完全不接受，恢复才无需猜哪一边有效。

[DEDUCED:double-entry-ledger-wallet-balance-is-a-derived-view-not-an-editable-authority] 余额是所有已提交分录按账户折叠的结果。直接把 30 改回 100 虽修正页面，却没有业务原因、对手科目和可重放证据，也会在下一次重建时消失。纠错必须提交引用原 entry 的反向或调整分录，让历史和当前值同时成立。只有非资金的临时展示缓存可以丢弃重算。

[DEDUCED:double-entry-ledger-wallet-available-balance-requires-hold-ordering] 若两个 70 元支出并发读取 100 元，各自在不同服务写 hold，就会共同承诺 140 元。hold 与新支出必须经过同一账户版本或预先切割的额度 token；之后的复式入账解决账本守恒，但不能挽回已经对外授权的超支。幂等只去掉同一事件重放，不仲裁两个不同支出。

## 从简单方案演进

最简单正确基线是单数据库 journal transaction：先验证每币种 posting 和为零、event_id 唯一，再原子追加 entry 与 posting；关键余额可在同一事务更新带版本的快照。低规模时按账户读取 journal 即可验证。分两次写借贷或直接 `UPDATE balance` 是错误基线，因为崩溃与纠错不可解释。

第一个待校准指标是单账户可用余额条件冲突超过 2%，或其串行提交队列 `p99` 超过 50 毫秒。达到后把该账户固定路由到一个账本分区，以短批次追加并在批内保持顺序。2% 与 50 毫秒需按授权 SLO 压测；调低会增加路由成本，调高会让热账户重试放大。平台总账户可分片，但单账户不做多主余额合并。

第二个待校准指标是余额 projection watermark 落后 journal 超过五秒，而产品承诺一秒可见。此时支付前读取同步快照或直接读 owner，并在普通页面显示 pending；五秒和一秒是初始产品边界，需结合重放吞吐调整。新增成本是关键读压到写分区，恢复后才切回异步投影。

没有选择可变余额作为唯一表，因为审计和重建下界要求保留原因。也没有选择每个 posting 独立事件后异步补平；只有统计计数而非资金守恒场景才可接受短暂不平衡。

## 设计决定

本设计以 journal entry 为最小提交单位，命令包含 event_id、账本、币种、有效日和至少两条 posting。服务端校验零和、精度、科目权限与业务版本，在一个事务追加；event_id 重试返回原 entry，参数不同则拒绝。账户 hold 在相同 owner 的版本序列中保留 available，CaptureHold 原子释放 hold 并发布平衡分录。

余额投影按 journal offset 消费，响应同时给 `balance` 与 watermark；对资金授权必须读取满足最低 watermark 的 owner 快照。外部充值先记在渠道应收或在途账户，收到确定证据后再转入可用钱包；提现同理。重复回调只命中原业务 event，不直接修改余额。

反选方案是事件分别写入各账户分区，再靠对账补齐对手方，它水平扩展容易，却允许跨账户前缀不守恒。只有各分区先接受同一原子提交证明，或业务允许未提交 posting 完全不可见时，才可采用分布式存储实现。

## 运行与演进

关键 SLI 包括非零和分录拒绝、重复 event 参数冲突、journal 提交延迟、热账户队列、hold 老化、projection lag、重建校验差异和未解释在途金额。过载时先降级历史检索与报表，再延迟普通余额刷新；资金授权、原子 Post 和 hold 释放优先，绝不能通过跳过零和校验提高吞吐。

故障演练：10:00:00 event E9 的 A:-70 与平台:+70 在事务中提交，响应丢失；10:00:01 投影只消费到前一 offset，页面仍显示 100；10:00:02 客户端重试 E9。journal 返回既有 entry，不追加第二组；关键读取等待 watermark 后显示 30。再注入重复 projection 消费，按 entry ID 折叠后余额仍为 30，证明 journal 与读模型的故障边界。

科目或 posting schema 升级先双读影子投影并逐分区重放，回滚仍解析新版本。分区迁移用 epoch 禁止同账户双写，校验迁移前后 journal hash 与余额。访问审计记录谁提交调整分录，敏感说明加密；不同租户不得共享可写科目，即使共用物理 journal。

## 面试考察本质

这题考察的是：给定“每笔资金变化都能由不可变零和分录证明、可用余额不能超支”的不变量，因为余额缓存不知道另一边 posting 和 hold 是否已提交，候选人应推导出原子 journal、带顺序的可用余额与可重建投影，并在审计正确性、热账户串行度和读新鲜度间取舍。

优秀回答会先写 `Σ postings=0`，区分 journal 和 balance，给出单边崩溃反例，让 hold 参与账户顺序，并用冲正而非删除纠错。常见误区是只做两张余额表、认为最终一致会自动守恒、把外部支付成功直接加余额，或用幂等键替代不同支出的额度竞争。

二十分钟应讲清 entry/posting、零和与 event_id；四十分钟加入 hold、projection watermark、充值提现在途；六十分钟再讨论热账户、分区迁移、重放审计与多币种。追问可要求修复一笔错误分录，观察候选人是否保留原始证据和对手科目。
