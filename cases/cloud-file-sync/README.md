# 云文件同步

## 表面题目

设计云文件同步，表面上是让多台设备看到同一目录。真正问题是离线多写者如何在重连、重试、改名和路径复用后收敛，同时不静默丢失任何已确认内容。一次本地“保存”只可由设备文件系统声明已持久化；一次“同步到云端”由云端版本图接受带父版本的操作声明；“其他设备已可见”还需各设备游标推进，三者不是同一成功状态。云端版本图与目录映射拥有跨设备收敛历史，设备只拥有离线副本和待提交操作。

主流程包括本地事件发现、稳定 file_id 识别、内容分块、上传 generation、条件提交目录操作、拉取远端日志和冲突呈现。题名容易让人只讨论块去重和传输速度，却忽略文件身份、并发历史与冲突。设计支持离线编辑、重命名和显式冲突副本，不承诺任意二进制文件都能自动语义合并，也不把“云端已收”冒充“所有设备已下载”。

## 反问与边界

先问文件数量和大小分布、设备数、离线时长、变更比例、共享目录热点、同步新鲜度以及本地保存、云端接受、全端可见分别需要什么提示。文件能否在不同设备同时编辑，文本可否三方合并，二进制冲突如何呈现，删除和改名是否可撤销，路径大小写和 Unicode 规范是否跨平台一致。还要问文件系统事件是否可靠、全盘扫描成本和断点续传要求。

安全边界包括租户与共享权限、端到端加密是否允许服务端合并、被撤权设备的离线密钥、保留版本和合规删除。网络和电量预算会限制移动端拉取频率。后文百分比、分钟数和操作数均为待校准策略。非目标是用最后到达覆盖解决所有冲突、让路径充当永久身份，或在设备离线时承诺它已经看见云端新版本。

## 客观模型

设备保存 `{device_id, local_cursor, pending_operations, local_file_id_map}`；云端保存 `{file_id, content_generations, parent_edges, directory_entries, operation_log, conflict_branches}`。稳定 `file_id` 标识逻辑文件，path 是带版本的目录 entry；content generation 不可变，并携带父 generation。云端操作日志拥有跨设备接受顺序，但不能抹去客户端未观察到的并发父关系。接口可表示为 `UploadContent`、`CommitFile(file_id,parent,new_generation,op_id)`、`Rename(file_id,entry_version,new_path)` 和 `Pull(cursor)`。

不变量是：已云端接受的 generation 不被无共同祖先关系的后来上传静默覆盖；同一路径最终指向明确一个 file_id 或显式冲突 entry；重复 op_id 只应用一次；本地保存、云端接受与设备可见分别可查询。并发分支可由用户、确定合并函数或保留双方来收敛，但任何合并都必须引用输入父版本。

若文件大小为 `B`、块大小 `c`、变化比例 `δ`、活跃设备 `D`，内容传播字节下界约为 `B×δ×(D-1)`，块数为 `ceil(B/c)`，尚未计元数据与加密。若每 `T` 秒扫描 `F` 个文件，每设备检查率为 `F/T`；小文件多且实际变更少时，元数据扫描会压过有效传输。热点共享目录的冲突和日志 fan-out 不能用平均每用户设备数解释。

## 必然约束

[DEDUCED:cloud-file-sync-save-and-sync-acknowledgements-name-different-durability-domains] 设备 A 在断网时 fsync 新内容，本地保存成功成立，但云端没有该 generation；云端恢复后接受操作，也不代表关机的设备 B 已拉取。若 UI 只显示一个“已保存”，断网丢机后用户无法知道内容在哪个故障域。故三个确认必须有独立状态与图标或 API。产品可以选择只向用户突出一个，但恢复说明和自动化不能混淆其含义。

[DEDUCED:cloud-file-sync-path-is-not-a-stable-file-identity-under-rename] 最小反例：初始 `file_id=17` 位于 `/team/plan.md`、generation 5。A 离线编辑它；B 把 17 改名为 `/team/plan-2026.md`，并将新文件 `file_id=23` 放入旧路径。A 恢复后若只按 path 上传，就会覆盖 23，误把“编辑旧文件”变成“替换新对象”。携带 `file_id=17,parent=5` 后，云端可把内容附到正确身份并根据 B 的 generation 判断冲突。

[DEDUCED:cloud-file-sync-convergence-requires-preserving-concurrent-history] A、B 都基于 generation 8 离线修改，彼此不可见，因而不存在可证明的先后。若服务端采用最后到达覆盖，较差网络的设备反而获胜，另一个已经云端接受的内容消失。版本图保留两个父为 8 的分支，文本可三方合并，二进制产生冲突副本；只有合并 generation 同时引用两分支后，所有设备才有可解释的收敛点。墙钟更大也不能证明因果顺序。

## 从简单方案演进

最简单正确基线是单设备文件夹加周期全盘扫描：本地保存可靠，云端只接受带内容哈希的整文件上传，不涉及并发。第二台设备加入后，引入云端操作日志和游标，避免每次下载全目录；离线写出现后，再增加稳定 file_id、父 generation 和冲突分支。块级去重只在大文件变化少、重传成本成为主瓶颈时加入，而不是正确性的前提。

第一个**待校准**切换指标是：完整目录扫描耗时超过同步新鲜度目标的百分之五十，或端点扫描 I/O 连续十五分钟超过本地预算百分之五时，切到文件系统事件加云端日志 cursor，并保留低频补扫检测漏事件。百分之五十、百分之五和补扫周期要按设备电量、事件可靠性与目录规模校准；事件化降低扫描，却新增游标断档和平台差异。

第二个**待校准**切换指标是：某共享目录七天条件提交冲突率超过百分之一，或离线待同步操作 `p95` 超过一万条，停止 last-writer-wins 自动覆盖，保留并展示分支，为文本启用合并助手，并限制高冲突目录的离线批量改名。门槛应按人工处理容量和文件价值校准；自动合并减少打扰，但错误合并的损失可能高于显式冲突。

未选择以中心锁禁止离线并发编辑，因为设备离线时无法续租，锁要么阻塞可用性，要么过期后仍产生冲突。若场景是短时在线协同、文件价值极高且所有客户端可持续连网，独占租约会重新变优。也未选择按服务器到达时间覆盖，它只简化 UI，却不能满足不静默丢历史的不变量。

## 设计决定

本设计让设备首次发现文件时分配稳定 file_id，并把本地路径作为可版本化目录 entry。内容分块上传形成不可变 generation；云端提交必须携带 `op_id,file_id,parent_generation,entry_version`。父与当前头相同则前进，父为祖先可按规则快进或合并，无因果关系则保留冲突分支。云端接受后返回 server cursor；其他设备拉取并应用到本地后分别上报自己的可见 cursor。

上传 part 与操作都幂等，超时重试同一身份。平台事件乱序或丢失由周期补扫和内容哈希修正。删除是版本化 tombstone，离线设备上传旧内容时形成恢复分支而非无条件复活路径。过载时先延迟缩略图和低优先级预取，限制大文件并发，再按目录公平同步元数据；不能丢弃冲突操作来让队列看似变短。

反选方案是把目录建模为单个全局强事务树。它能简化在线改名原子性，却让离线设备无法提交并使大目录成为协调热点。若企业工作区规模小、所有设备在线且目录权限事务比离线可用性更重要，它会重新变优；当前采用每 entry 版本与操作日志，跨目录批量动作暴露进行中和补偿状态。

## 运行与演进

SLI 按确认域分别统计本地待上传年龄、云端条件提交成功、设备 cursor 落后、端到端可见延迟、冲突分支数、静默覆盖检测、扫描 I/O、块去重率和租户日志积压。用户可看到“仅本机”“已到云端”“设备 B 尚未同步”。过载时优先传播删除、权限和目录元数据，再传小文件与活跃内容，最后传历史大文件；公平调度防止一个大视频阻塞整个目录。

故障演练时间线：零毫秒 A、B 均有 generation 12；A 离线编辑为 13A 并本地保存；B 在线改名且提交 13B；一小时后 A 恢复，上传 13A 并提交 `parent=12`。服务发现当前头 13B 与 13A 并发，保留两分支并返回冲突，而不是按到达覆盖。两设备随后拉到同一版本图，用户合并为 generation 14，引用 13A 和 13B，目录最终收敛。演练同时拔掉 A 磁盘，验证只有标为云端成功的内容可恢复。

协议升级先让客户端透传未知操作和 generation 字段，再让新客户端产生版本图，最后停止旧 path-only 提交；旧客户端请求通过网关转换但遇到歧义必须冲突，不能猜测身份。回滚保留读取新分支。端到端加密时服务端只能比较密文身份和父关系，内容合并下放客户端。撤权先阻止新操作，再轮换共享密钥；历史设备副本的删除属于独立安全契约，不能由同步成功自动证明。

## 面试考察本质

这题考察的是：给定“不静默丢失已保存内容、路径改名后仍识别同一文件”的不变量，因为离线设备不知道其他设备的并发历史，候选人能否区分本地保存、云端接受和多端可见，用稳定身份、父 generation 与冲突分支交换自动化程度、离线可用性和用户处理成本。

优秀回答会先画三种确认状态，使用 file_id 而非 path，给出改名加路径复用的最小反例，并说明版本图如何保留并发。常见误区是只谈分块去重、以服务器时间最后写胜、把文件系统事件当绝不丢失，或认为所有设备最终下载等同于云端已经接受。

二十分钟完成设备、云端状态和保存/同步语义；四十分钟加入稳定身份、游标、版本图与冲突；六十分钟再讨论块传播、跨平台路径、加密、撤权和升级。追问应落在具体设备离线期间：它知道哪些父版本，重连操作绑定哪个 file_id，云端为何有权接受或分叉，以及用户在哪个确认状态下可以安全丢弃本机。
