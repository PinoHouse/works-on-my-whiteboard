# 分布式缓存

## 表面题目

设计分布式缓存，表面上是把热门数据放到内存，以更低延迟承接读流量。真正的问题是：读接口承诺的是“某个仍在 TTL 内的副本”、有界陈旧版本，还是主存当前版本；业务写返回成功后，同一用户能否立即读到自己的新值。主流程包括查缓存、未命中回源、填充，以及主存写入后的失效或更新。成功写只能由业务主存的条件提交点声明，缓存节点拥有的是可丢弃副本，不能因为命中快就取得业务事实的所有权。本题不把缓存当持久数据库，也不承诺任意 key 的全局事务。

题名还掩盖两类不同风险：普通 key 的旧值多存几秒可能只是体验偏差，价格、权限或配额的旧值却可能破坏业务不变量；冷 key 未命中只增加一次读取，热 key 同时过期则能把保护层变成回源风暴。设计必须分别给出新鲜度边界和主存保护边界，而不是以一个“命中率”概括成功。

## 反问与边界

先问缓存对象来自哪个权威主存，值是否携带单调 `source_version`，读者可容忍的 `max_staleness`、读己之写和权限撤销时限分别是多少。还要问读写比、峰值请求率、单 key 偏斜、值大小、TTL 分布、负缓存时长、主存安全 QPS 与连接数，以及一次主存查询的尾延迟。若客户端只要明确标记的旧数据，故障时可用性优先；若读结果参与扣款或鉴权，则必须绕过缓存或校验版本。

地域、租户和隐私也进入边界。缓存副本能否跨地域，敏感值是否允许驻留共享内存，租户 key 是否有不可混淆的命名空间，删除和权限撤销是否要求在固定时间内清除。未知生产数字只写成符号；后文出现的百分比和分钟数均是待校准策略参数，不是测量结论。非目标包括让缓存承担最终写入确认、以 TTL 代替法定删除证明，以及在网络分区中同时承诺所有读都可用且永远最新。

## 客观模型

核心状态为 `key -> {value, source_version, expiry, generation, negative}`。主存拥有业务事实和单调版本；缓存分片只拥有带来源版本、可随时淘汰的派生副本；失效控制面拥有每个命名空间或 key 的 generation 传播进度。最小读接口可表示为 `Get(key, freshness_requirement)`，返回值、版本与 `freshness_evidence={verified,unverified}`；写接口仍落在主存，成功后执行 `Invalidate(key, committed_version)` 或为本会话记录绕过屏障。若产品承诺写后读最新，写成功至少表示主存已提交，且随后读路径不会接受低于已提交版本的副本。

不变量是：缓存值不得冒充主存当前值；版本低于调用方最小版本时不能作为新鲜读返回；负缓存必须受更短失效边界约束；同一回源结果可以共享，但失败不能无限并发重试。设热 key 到达率为 `λ`，从发现 miss 到新值可见的重建窗口为 `w`，无请求合并时一次过期的期望并发回源近似 `A_miss ≈ λw`；single-flight 让主存读取接近一次，其余请求等待或取得明确标记的旧值。向 `K` 个副本广播精确失效的控制面工作至少为 `Ω(K)`，所以副本增加会扩大失效扇出。

倾斜决定系统而非平均命中率。若总命中率为百分之九十九，但一个每秒十万请求的权限 key 同时过期，`w=0.2s` 就可能产生约两万次在途回源。随机 TTL 只能打散不同 key，无法打散同一 key 的同一过期时刻；因此热 key 身份、在途回源数、版本落后和主存排队必须是一等状态。

## 必然约束

[DEDUCED:distributed-cache-hit-does-not-establish-freshness] 缓存命中只证明某副本未被本地淘汰。最小反例是 `price:sku-1` 在缓存中为版本七、TTL 六十秒；十二点零分十秒主存条件写入版本八并成功，失效消息丢失；十二点零分十一秒缓存仍命中版本七。没有读取主存当前版本、调用方携带 `min_version=8`，或节点已经观察到不低于八的 generation，缓存无法区分“版本七仍是最新”和“失效消息在路上”。因此普通命中的新鲜度是未知，而不是已证明“新鲜”或“陈旧”；只有契约明确允许返回潜在陈旧值时，这个结果才可接受。

[DEDUCED:distributed-cache-write-success-must-name-the-visibility-boundary] 若写接口的成功只表示主存提交，而产品同时声称同会话随后缓存读到最新值，两条承诺之间缺少状态转换。可行边界是同步等到失效 generation 可见、写后会话携带 `min_version=8` 并绕过旧副本，或明确声明有界陈旧。缓存不可用时，主存提交仍可成功；但响应必须说明后续读采用哪条规则，不能让一个成功码暗含尚未发生的全缓存传播。

[DEDUCED:distributed-cache-hot-key-expiry-amplifies-origin-load] 对同一热 key，所有 miss 共享同一个缺失事实。若每个请求独立回源，`λw` 个请求会重复执行等价读取；增加缓存节点只会让更多节点同时看见过期，无法降低这一下界。最小故障时间线是：零毫秒 TTL 到期；十毫秒一千个请求 miss；二十毫秒主存变慢；一百毫秒请求超时并各自重试；两百毫秒连接池耗尽，其他冷 key 也失败。请求合并、重试预算和可控旧值服务必须在风暴前配置，而不是等主存超时后才启用。

## 从简单方案演进

最简单正确基线是进程内按需缓存：固定 TTL、容量上限、未命中直接查主存，并且 API 只承诺 TTL 内有界陈旧。它没有跨节点失效成本，适合少量实例和可容忍旧值的数据。实例数上升导致重复内存与低命中，才引入一致哈希的共享缓存分片；这增加网络跳转、分片热点、成员变更和租户隔离问题，但仍不改变主存是事实源。

第一个切换指标标为**待校准**：当某 key 的 miss 后同时在途回源数连续五分钟超过二十，或该 key 回源流量超过主存安全读容量的百分之十，启用 single-flight、软 TTL、随机抖动与受控 `stale-while-revalidate`。二十和百分之十只是初始保护门槛；压测后应按主存排队曲线、连接池余量和旧值风险调整。切换解决回源风暴，却新增 leader 超时、等待队列和旧值可见，需要以最大等待时间与重试总预算封顶。

第二个切换指标也标为**待校准**：当带版本采样得到的陈旧读比例连续三十分钟超过百分之零点零五，且该数据类要求读己之写，就改为写后会话携带最小版本、同步等待 generation 或直接绕过缓存；只有失效传播 `p99` 低于两秒并稳定一个观察窗后才恢复普通读。这个门槛交换主存压力和新鲜度预算，不应机械用于允许分钟级陈旧的推荐结果。

再下一步才按数据风险分层：权限和价格采用主动失效加版本屏障，内容页采用软过期，昂贵空结果采用短负缓存。未选择“每次命中都向主存校验版本”，因为它让缓存失去隔离主存读流量的价值；若只有极少数高风险 key 且主存提供廉价版本目录，这个方案会重新变优。也未选择双写缓存和主存作为两个权威，因部分失败会制造无法裁定的分叉。

## 设计决定

本设计采用 cache-aside 加带版本值。正常读先检查 key、租户、expiry 与调用方 `min_version`；版本满足调用方已知下界时可返回相对该要求的 `verified`，没有版本下界且尚未观察到可作为证据的 generation 时，普通命中只能按契约返回 `unverified` 的潜在陈旧值。软过期可由一个 leader 回源，其他请求在等待上限内排队或取得明确标记为超出软 TTL 的旧值。miss 回源结果只有在其 `source_version` 不低于当前缓存版本时才能填充，避免慢请求覆盖新值。主存条件写是业务写线性化点；成功后发布 `{key, committed_version, generation}`，并让写会话在看到相应缓存版本前绕过。

超时和重试按层限额：缓存超时快速转入受配额的回源，主存请求由 single-flight 共享且只由 leader 重试，调用方取消不必取消仍有等待者的共享请求。主存过载时，允许陈旧的数据先延长软 TTL，再拒绝低优先级回源；强新鲜数据快速失败而不返回未标记旧值。删除与权限撤销使用更严格的版本屏障和短本地最大驻留期，不能仅依赖长 TTL。

反选方案是写穿缓存：先由缓存层同步写主存再更新所有副本。它能集中写路径，却让缓存成为写可用性依赖，并仍需处理更新副本部分失败。若未来所有写本来就通过一个受控代理、key 集很小且失效传播必须同步审计，写穿会重新变优；当前多入口主存和大规模读缓存更适合以主存提交为唯一成功语义。

## 运行与演进

运行 SLI 包括按数据风险分层的陈旧读率、写后读违约数、缓存命中延迟、miss 回源并发、single-flight 等待人数、主存因缓存产生的 QPS、失效传播延迟、负缓存命中和租户热点。不能只看全局命中率。过载顺序是暂停预热和低优先级刷新、限制单租户回源、为允许旧值的数据延长软 TTL，最后拒绝强新鲜读；不得为了成功率把权限旧值当最新返回。

具体故障演练：零毫秒主存把版本四十一提交并响应；十毫秒失效总线丢包；二十毫秒同会话携带 `min_version=41` 读取，缓存只有四十，因版本屏障绕过并拿到四十一；五十毫秒普通读没有该下界，只能按“允许潜在陈旧”的契约返回四十并标为 `unverified`，不能断言它已陈旧；一秒后节点观察到 generation 四十一，版本四十才可判定为 stale 并被条件更新替换。随后让热 key 在主存延迟升高时过期，验证只有一个回源 leader、等待者不自行重试、旧值超过硬截止后被拒绝。

协议迁移先让所有读路径理解版本与 `freshness_evidence`，区分 `unverified`、超出软 TTL 和已由版本证据判定的 stale，再双写版本元数据，最后启用最小版本屏障；回滚可停止强制屏障，但必须保留解析字段。预热只按真实热点小流量灰度，比较主存 QPS 和陈旧率，避免发布本身制造风暴。敏感数据按租户加密或不入共享缓存，key 日志散列化；成本审视同时计入缓存内存、跨区失效流量和主存被保护的容量，不能把更高命中率自动视为更低总成本。

## 面试考察本质

这题考察的是：给定“成功写后用户能读取到哪一个版本”这一不变量，因为缓存节点在失效可能丢失时不能从普通命中判断当前版本、热 key 请求又共享同一个缺失事实，候选人能否把主存提交、带证据的新鲜度、缓存可见性和回源放大分开，并按旧值风险与主存容量选择版本屏障、失效和请求合并。

优秀回答会先定义事实源、写成功和新鲜度等级，再用版本七到八的最小反例证明命中不等于最新，用 `λw` 量化风暴，并说明 TTL 抖动为何不能解决单 key 同时过期。常见误区是只谈命中率、把缓存双写当原子操作、在过载时静默返回旧权限，或认为多加缓存节点自然保护主存。

二十分钟应讲清 cache-aside、版本值和成功语义；四十分钟加入 single-flight、软硬 TTL、负缓存与租户公平；六十分钟再讨论失效 generation、写后会话、故障演练和迁移。追问始终落在可观察状态：谁提交了版本，哪条读路径接受多旧的值，过期瞬间有多少请求能访问主存，以及降级是否仍满足该数据类的契约。
