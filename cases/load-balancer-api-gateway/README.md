# 负载均衡器与 API 网关

## 表面题目

设计负载均衡器与 API 网关，表面动作是接收一个请求并选后端转发。真正的状态变化是入口规范化 method/path，固定一份不可变策略快照并据此确定 route classification，再执行该路由声明的认证、授权、配额与准入，最后才选择 endpoint 并在有限重试预算内产生后端尝试；对写请求而言，网关超时并不能说明后端没有提交。成功必须同时满足“请求按同一策略快照的正确路由进入了允许的后端”和“响应语义没有因代理重试制造额外业务副作用”。

题名容易把四层连接分发、七层路由、认证网关和业务编排混在一起。本设计聚焦 L7 API 入口：终止 TLS、认证、限额、版本化路由、过载保护和安全重试。后端业务状态仍由服务自身拥有，网关不把订单或付款复制成自己的事实源；服务发现只提供候选集合，持久 key 的数据归属也不由本题决定。

## 反问与边界

先问协议、连接寿命、峰值请求率 `λ`、请求和响应大小、路由热点、端到端 `p99`、可接受排队时间及后端服务率 `μ`。语义上要区分安全方法与非幂等写，客户端是否提供幂等键，超时后能否查询原结果，流式请求是否可重放。策略边界包括 method/path 如何规范化、同一快照如何按规范键唯一分类路由、每类路由声明的认证来源与租户配额、灰度规则、请求改写以及策略变更多久生效；若分类有歧义，入口必须在认证和转发前拒绝。

故障取舍要问清：后端拒绝、连接失败、响应超时分别能否重试，是否跨区域，何时 shed load，健康端点是否可能已无队列容量。安全上，认证必须先于会消耗高成本的业务转发，外部头部不能伪造内部身份，策略日志记录版本与决策理由而非敏感令牌。非目标包括由网关承诺跨服务事务、把每个后端的内部数据库一致性隐藏起来，以及用被动重试代替业务幂等。

## 客观模型

最小入口状态为 `(route_policy_version, canonical_method_path, route_classification, authenticated_principal, request_id, idempotency_key, endpoint_health, endpoint_load, retry_budget)`。控制面拥有不可变策略快照与发布世代；网关数据面拥有某次请求的规范键、路由分类、认证、准入和尝试预算；后端拥有业务提交结果。正常顺序是协议解析与大小限制、规范化 method/path、固定一份完整策略快照、以规范键确定 route classification、按该路由执行认证与授权、租户配额和容量准入、再选择后端 endpoint、转发与有条件重试。路由分类包含认证方案、授权规则、配额桶、准入类和候选后端池，但不是具体 endpoint。顺序本身是正确性边界：先用未经规范化的路径鉴权会产生解析差异，先选 endpoint 再分类会让策略和后端池来自不同决策，先无限排队后限额会让恶意租户耗尽内存。

不变量包括：每个进入后端的请求都由规范化 method/path 在一份固定策略快照中唯一分类，并完成该 route 的认证、授权、配额和准入；endpoint 选择不得改变 route classification 或策略版本；外部身份只能被可信认证步骤写入；一次请求的总尝试数不越过预算；非幂等写在结果未知时不由网关盲目重试；过载不能让一个租户占满共享连接与队列。若外部到达率为 `λ`，每个请求平均额外尝试 `r` 次，则后端到达率为 `λ_backend = λ × (1+r)`。当 `λ_backend ≥ μ` 时队列趋于增长，重试会降低而不是提高成功率。

主导放大来自 TLS、鉴权、策略求值、镜像和重试按请求重复。健康探针只给出过去时刻的可达性，实时队列与下游依赖仍会变化；因此候选端点集合与某一请求是否应被准入是两个状态问题。

## 必然约束

[DEDUCED:load-balancer-api-gateway-retry-without-idempotency-key-amplifies-ambiguous-writes] 客户端发出无幂等键的 `POST /orders`，后端在 80 ms 提交订单，但响应包丢失；网关在 100 ms 超时后无法区分“未执行”和“已提交未响应”，若自动重试，第二次又创建订单。超时只是一条观测边界，不是业务回滚证明。结论是非幂等写默认不重试，或由后端以幂等键保存第一次结果；只有连接在任何请求字节发出前明确失败时，才可安全换端点。

[DEDUCED:load-balancer-api-gateway-health-based-selection-does-not-prove-request-capacity] 端点 A 的探针在零毫秒成功，只说明当时探针路径可达；十毫秒后它的工作队列可能已满，数据库连接也可能耗尽。若网关继续把新请求按“健康”均匀送入并在超时后重试，A 的排队会扩散到其他端点。选择算法必须纳入并发、队列、被动错误和隔离上限，且在无容量时拒绝或排队；健康不是容量证明。

[DEDUCED:load-balancer-api-gateway-policy-version-must-travel-with-routing-decision] 假设原始路径 `/a/../admin` 在鉴权器看来属于公开前缀，而后端规范化为 `/admin`；或网关先从新端点表选择 admin 后端，再用旧策略快照分类，便可能绕过预期规则。正确路径必须先规范化 method/path，从同一不可变快照唯一确定 route classification，再按该 route 执行认证、授权、配额和准入，最后才从分类给出的后端池选择 endpoint。策略版本、规范键、route ID 与决策同行，才能审计请求为何被允许、拒绝或进入灰度组；这不要求每个请求同步查询控制面。

## 从简单方案演进

最简单正确基线是单个反向代理，加载一份静态路由与认证策略，把请求转发给固定后端。低流量时它提供清楚顺序，但自身 CPU、连接与队列形成单点。当网关 CPU 或连接池利用率持续超过 `70%`，或入口队列 `p99` 已消耗端到端延迟预算的 `25%`，水平扩展无会话数据面，并把策略作为版本化只读快照分发。新成本是策略世代偏斜、负载反馈延迟和发布回滚。

第二步给每条路由设并发上限、租户公平队列和 retry budget。只有安全方法或带后端幂等契约的写才重试，并在总超时内使用指数退避。第二个待校准切换指标是某路由的 5xx/timeout 超过 `1%` 且 `retry-attempts/request > 1.1`：立即把自动重试收紧到安全幂等方法，启用每租户并发上限或直接丢载；恢复同时观察成功率与后端队列。`70%/25%` 与 `1%/1.1` 是两组待压测和故障演练校准值，调高会提高利用率但放大队列与故障切换风险，调低会更早扩容或拒绝并增加闲置容量。

第三步将控制面与数据面分离。控制面验证规范化后的 method/path 不会命中歧义 route，签名后发布包含路由、认证、配额、准入类与候选池引用的不可变版本；数据面只在完整快照就绪后原子切换，不逐条覆盖规则。每个请求先固定其中一个版本并完成 route classification。端点选择随后在该 route 的候选池内采用带并发反馈的最少负载或加权选择，并为异常端点熔断；熔断恢复用少量探测流量，不能所有实例同时全开。

未选择让网关持久化并编排所有业务工作流，因为它会把入口故障域扩大为业务事实源，并使服务间状态难以归责。当场景确实是短小、可补偿且需要统一编排的边缘聚合 API 时，有限编排层会重新变优。也未选择对所有 5xx 自动重试；只有后端明确声明幂等和剩余预算充足时，这才可取。

## 设计决定

本设计使用无状态数据面集群和版本化控制面。每次请求先执行协议与大小限制并规范化 method/path，再固定一个 `route_policy_version`，用规范键从该快照唯一确定 `route_id`；随后按 route 声明执行认证、授权、租户配额和容量准入，只有全部完成才在 route 的后端池内选择 endpoint 并占用连接。网关生成稳定 `request_id`，保留调用方幂等键但不伪造业务幂等；后端响应和重试尝试都关联同一请求追踪，重试不得重新分类到另一策略版本。

端点选择只在已准入 route 的候选池内读取本地健康与负载快照。连接前失败可以在同一池和同一策略版本内换 endpoint；请求已发送后的超时视为结果未知，安全读取可在剩余预算内重试，非幂等写返回可查询的模糊结果或由后端幂等记录解答。过载时按已确定的准入类先拒绝低优先租户与昂贵路由，限制队列长度，再 shed load；不通过重新分类、改换策略版本或无限重试制造表面接收。

反选方案是基于源地址的永久粘性路由。它实现简单并支持内存会话，却会造成 NAT 热点、实例下线重映射和状态难迁移。只有协议无法外置会话、会话量可控且实例故障后允许重建时，粘性才重新变优；默认会话状态应由后端共享存储或令牌承载。

## 运行与演进

关键 SLI 包括入口成功率与 `p99`、认证拒绝原因、策略版本分布、队列等待、每租户并发、后端尝试数、重试放大、结果未知写入数和端点被动失败。看板按路由、策略世代和端点分解，不能用整体 2xx 隐藏一个支付路由的重复写。过载降级顺序是关闭镜像与非必要变换、收紧长响应和低优先请求、限制重试，最后明确拒绝新流量；认证与租户隔离不 fail-open。

故障时间线：0 ms 网关规范化 `POST /orders`，在策略 v42 分类为 route `orders-write`，完成该 route 的认证、配额与准入后选择 endpoint A；80 ms 后端提交订单 O1；90 ms 响应链路断开；100 ms 网关超时。正确结果是保留 v42 与 `orders-write`，记录“提交未知”并返回可查询错误，不发第二次 POST。若故障注入开启盲重试，120 ms 后端创建 O2，即复现重复副作用。另一次演练令 endpoint A 探针持续成功但队列填满，验证 route 准入先于 endpoint 选择，且被动负载反馈、并发上限与熔断能阻止重试向健康端点集扩散。

策略发布先用规范化 method/path 在影子数据面比较 v42/v43 的 route classification、允许、拒绝和候选池差异，再按实例灰度；回滚只切回已知完整快照，请求日志仍保留原版本、规范键和 route ID。证书、认证密钥与策略版本分别轮换，避免一次回滚恢复过期密钥。跨地域灾备必须独立定义幂等记录可见性，否则把模糊写跨区重试只会扩大重复风险。

## 面试考察本质

这题考察的是：给定“规范请求必须由同一不可变策略快照分类并完成该 route 的安全与准入后才能选择 endpoint，非幂等副作用不能因入口猜测而重复”的不变量，因为网关从原始路径无法假定上下游解析一致、从超时无法知道后端是否提交、从健康探针也无法知道下一请求容量，候选人能否把规范化、路由分类、鉴权配额、准入背压、端点选择和重试预算组成一个过载时不自我放大的入口决策。

优秀回答会先画出“规范化键 → 固定快照 → route classification → 认证/授权/配额/准入 → endpoint”的执行顺序，指出后端而非网关拥有业务结果，用 `λ × (1+r)` 解释重试放大，并以 80 ms 提交、100 ms 超时的 POST 反例说明模糊结果。常见误区是先选 endpoint 再决定策略、把所有 5xx 视为可重试、把健康等同空闲、让控制面逐条热改规则，或用无版本日志审计授权。

二十分钟回答应完成入口职责、策略版本和单代理基线；四十分钟加入幂等重试、负载反馈、队列与公式；六十分钟再讨论策略灰度、租户公平、跨地域模糊写和安全密钥轮换。追问应固定在“响应丢失时网关知道什么”和“探针成功时端点还有什么未知”，让候选人逐步说明状态所有者与降级顺序。
