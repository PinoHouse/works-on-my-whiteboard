# 分布式限流器

## 表面题目

设计一个分布式限流器，表面上像是在多个服务节点前回答“这次请求是否还能通过”。真正的起点却是服务目标：每个租户在给定窗口内可以消耗多少预算，突发最多有多大，被拒请求需要得到怎样的重试语义，控制面故障时又要优先保护可用性还是配额。系统还必须给出可核查的决定原因，让调用方、值班人员和审计流程能够区分正常耗尽、输入错误与协调器不可用。

题目中的“分布式”意味着同一租户的请求会同时落到不同节点。单机算法正确并不自动推出全局行为正确；节点扩缩、请求倾斜和网络分区都可能改变实际放行总量。因此答案首先要声明全局配额不变量和允许的误差界，再讨论状态放在哪里、多久同步一次，以及故障期间是否接受有边界的超发或可用性下降。

## 反问与边界

在选机制之前要澄清配额的作用域：预算按租户、用户、接口还是区域统计，多个维度冲突时采用最严格者还是分层扣减；请求金额是否恒为一个令牌，批量请求能否原子扣减；突发容量、长期速率和公平性分别怎样定义。还要问清 SLO：正常路径的延迟预算、允许拒绝率、协调器可用性目标，以及超过配额时返回立即拒绝还是携带精确的重试时间。

容量与风险边界同样重要。节点数是固定还是弹性变化，流量是否高度偏斜，单个热点租户会不会占满共享协调资源，跨区域是否允许短时不同步。安全边界要求租户标识来自可信认证结果，不能由调用者随意覆盖；空白标识必须拒绝，合法标识则保持字节级精确，避免大小写或 Unicode 归一化把两个租户意外合并。还要明确故障语义：只有可识别的协调器不可用错误才进入降级策略，取消、非法金额、时间回退和内部错误不能伪装成一次成功放行。

本切片把范围收紧到单租户、单位请求、冻结逻辑时钟和无补充的短突发，用它隔离全局预算与故障策略。它不测跨区域复制、真实网络延迟、持久化恢复或恢复后的第一次请求，所以这些行为不能从实验结果外推。

## 客观模型

把租户的名义容量记为 `C`，补充间隔记为 `T`，当前可用令牌记为 `B(t)`。令牌桶要求 `0 ≤ B(t) ≤ C`；正常协调路径上的一次金额 `a` 请求只有在 `B(t) ≥ a` 时才能放行，并把状态原子地变为 `B(t)-a`。随着逻辑时间推进，令牌按固定速率补充但不超过容量。来源记录 `source-rfc-3290-token-bucket` 给出了这种容量、可用令牌和补充速率的标准模型，本案例只转述其定义并把它用于可重复实验。

共享方案为每个字节级租户键维护一个桶，所有节点请求都在同一线性化扣减点竞争；本地方案则以精确的 `(tenant,node)` 二元组维护独立桶。二元组必须是结构化键，不能把字符串直接拼接，因为 `(a,bc)` 与 `(ab,c)` 会产生歧义。若每个节点都持有完整容量 `C`，在补充尚未发生的冻结突发中，`N` 个节点最多共同放行 `N*C`，而业务声明的名义预算仍只有 `C`。

故障模型把协调器状态看作可用或不可用。fail-closed 在不可用区间拒绝请求，保住预算上界但牺牲成功率；fail-open 允许请求继续并明确标记降级，保住请求可用性却失去精确扣减。实验记录总请求、允许、拒绝、故障期分流、降级决定、错误数、名义容量和超发量；其中冻结单位请求下的超发量严格定义为 `max(0, allowed-C)`。

## 必然约束

[DEDUCED:distributed-rate-limiter-per-node-multiplies-global-quota] 若租户完整容量 `C` 被复制到 `N` 个彼此不协调的节点，每个节点都能独立证明自己的前 `C` 个单位请求合法，因此全局最多放行 `N*C`。这不是实现偶然，而是局部状态无法表达全局已消耗预算的必然结果。两节点、容量四的冻结突发给出最小反例：共享桶只能允许四次，本地桶却允许八次，名义配额超出四次。

全局守恒要求针对同一租户的成功扣减具有单一可比较顺序，或者每个节点只花费由全局分配且总和不超过 `C` 的子预算。前者可以由共享协调器完成，后者可以由租约或分层配额完成；仅靠异步汇报本地完整桶无法在汇报间隔内建立硬上界。租户隔离还要求一个租户的桶锁、热点和错误不改变另一个租户的令牌，节点隔离测试则必须证明结构化二元组不会串桶。

时间也是状态的一部分。一次操作只能读取一次注入的逻辑时钟，创建桶与扣减必须使用同一个时刻；时间回退或持续时间溢出必须作为可识别错误向上传播。取消需要在任何状态变更前检查，不能先花掉令牌再返回取消。健康决定必须原样携带剩余令牌和重试时间；降级决定则要清空这些看似精确但实际上未知的数值，并给出固定原因。

## 从简单方案演进

最简单的单节点令牌桶适合只有一个执行点的系统：它没有网络往返，容易给出精确剩余量和重试时间。当服务横向扩展后，直接在每个节点复制同样的桶虽然延迟最低，却把上界扩成 `N*C`，而且节点扩容会在没有业务配置变化时凭空增加预算。它只适合配额本来就是“每节点容量”，或者业务明确接受随节点数增长的松弛上界。

第二步是共享协调：所有节点对租户桶执行原子扣减。它恢复严格的全局守恒，适合高价值写入、滥用防护和合规边界，但增加一次协调延迟，并把热点租户集中到状态分片。可以按租户哈希分片、批量处理或就近路由来扩容，但任何缓存的可花费预算都必须计入总量。若正常路径 `p99` 延迟预算小于一次协调往返，或协调吞吐低于峰值请求率，共享方案就需要继续演进。

第三步是分层分配：全局层给节点发放额度为 `q_i` 的租约，并维持 `Σq_i ≤ C+ε`，节点只在本地消费自己的份额。切换条件可以量化：当共享协调器利用率长期超过百分之七十、其 `p99` 往返消耗端到端延迟预算的四分之一以上，且业务允许超发界 `ε` 时，分层方案更合适。租约批次越大，协调频率越低但流量倾斜时闲置越多；批次越小，利用率更好但协调开销升高。节点故障时未消费租约必须等待到期或经过安全回收，不能立即重复发放。

跨区域还可以为每区分配静态或动态子配额。若法规或成本要求区域自治，可以接受区域间暂时不能借用额度；若全局利用率更重要，则需要再平衡与明确的传播延迟。无论选择哪一步，机制都要把允许的误差、收敛时间和故障策略写进契约，而不是把“不常发生”当作正确性证明。

## 设计决定

本切片用共享桶作为全局配额基线，用按 `(tenant,node)` 分桶的实现暴露局部预算乘法。共享协调器按字节级租户键持有互斥保护的桶；本地实现按结构化二元组持有独立桶。两者只接收注入时钟和令牌桶配置，不读取墙钟、不休眠，也不依赖 goroutine 获胜顺序。并发测试只断言总放行数等于容量，不把具体赢家误当稳定契约。

[DEDUCED:distributed-rate-limiter-outage-policy-trades-availability-for-quota] 当协调器不可用且系统无法知道剩余令牌时，fail-closed 通过拒绝所有故障期请求维持名义上界，但会直接降低可用性；fail-open 通过允许这些请求维持服务成功率，却可能让超发量随故障期流量增长。两者不能同时保证精确配额与持续放行，因此策略必须由业务风险决定，并在每个决定上标记降级原因。

只有 `coordinator-unavailable` 这一可识别故障触发策略。任意内部错误、上下文取消、非法金额、时间回退与溢出都返回零决定和可追踪错误，防止错误分类扩大授权。不可用包装器在故障期完全不调用底层协调器，恢复可用也不重置原桶。实验在九百毫秒只恢复状态并观察指标，没有发送恢复请求；因此它证明的是故障区间取舍，而不是恢复后行为。

## 运行与演进

全局配额运行在 Unix 纪元、固定种子和两秒截止时间下执行。容量为四、补充间隔十秒，节点 `a` 与 `b` 在同一逻辑时刻按 `a1,b1,a2,b2,a3,b3,a4,b4` 的稳定顺序提交单位请求。共享实现应得到四次允许和零超发，本地实现应得到八次允许和四次超发。故障运行先用两个健康请求耗尽容量二的共享桶，在一百毫秒先施加故障再处理同刻请求，随后在二百、三百、四百毫秒继续请求，并在九百毫秒恢复后只观察。

上线时要同时观察允许率、拒绝率、故障期允许与拒绝、降级比例、内部错误、协调延迟、热点租户和配额超发。告警应区分“业务自然耗尽”和“协调器不可用导致策略生效”，否则 fail-closed 会被误判为正常限流，fail-open 又会被成功率掩盖。日志与追踪只记录经过授权的稳定租户标识或不可逆摘要，不能泄露令牌、用户凭据和原始敏感键。

发布演进采用影子计算与分阶段切换：新策略先只计算不执行，对比旧策略的决定分歧和预测超发；随后按租户小流量启用，并设置超发、错误率和延迟回滚阈值。调整容量、租约批次或故障策略都属于行为变更，需要版本化配置和审计记录。恢复语义、跨区域再平衡和租约回收要由后续独立实验验证，不能用本切片的通过结果替代证据。

## 面试考察本质

这道题考察的核心不是背出一种组件，而是能否从 SLO 推导不变量，再从信息边界识别不可能同时满足的目标。优秀回答会先写出全局预算公式，主动构造 `N*C` 反例，说明单机正确为何不能推出分布式正确；也会区分硬配额、允许误差的配额和仅用于平滑流量的软限制，并据此选择共享协调、本地预算或分层租约。

深入追问可以围绕线性化点、热点租户分片、节点扩缩时额度回收、时钟回退、批量金额原子性和跨区域分区展开。候选人还应明确 fail-open 与 fail-closed 的风险所有者：支付或安全边界可能优先守住配额，低风险读取可能优先可用性；任何降级都必须可观测、可审计并有持续时间上界。若提出分层方案，应能解释 `Σq_i` 的约束、租约到期以及流量偏斜带来的闲置。

最后要看证据意识。固定事件相位证明同刻故障先于请求，所有零指标仍要显式输出，单位属于测量契约而非下游猜测。测试应覆盖租户与节点隔离、歧义键、预取消、任意错误绕过策略、并发总预算和构建器深拷贝。对于没有发送恢复请求的实验，正确结论是“恢复后未探测”，而不是从协调器重新可用推断令牌状态一定正确。
