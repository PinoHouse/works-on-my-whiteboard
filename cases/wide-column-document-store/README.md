# 宽列与文档存储

## 表面题目

设计宽列或文档存储，表面上是保存字段可变的 JSON 文档或稀疏列，并按 key 扩展。真正问题是为已知查询形状选择分区，使一次条件更新只由一个权威副本集仲裁，同时承担 LSM 写入、压实、墓碑和索引的物理成本。成功写必须明确：同一分区的条件版本检查成立，更新已到达所选故障域的提交边界；它不自动表示所有二级索引、跨分区视图或全文检索已经同步可见。

题名把两种表面数据模型放在一起，但共同核心不是“无 schema”。文档仍有字段类型、版本和迁移规则，宽列仍有行键、聚簇顺序与墓碑语义。分区副本集拥有行或文档版本，schema/index generation 由元数据控制面拥有；客户端不能用灵活字段绕过身份、大小和查询边界。本设计不提供任意 join 或跨分区可串行事务。

## 反问与边界

先列出必须在线满足的查询：是否总能给出 partition key，分区内按什么 clustering key 排序，要返回多少行或文档，是否需要字段过滤、分页、聚合或二级索引。再问单分区大小和 QPS 分布、文档大小、字段更新粒度、TTL 与删除率、写读比例、离线副本最长时间、条件更新频率和允许的一致性等级。只有从查询反推数据布局，才能判断复制与压实预算。

还要明确 schema 演进：新旧客户端能否同时读，未知字段是保留还是丢弃，数组和嵌套字段如何合并，索引何时算可用。敏感字段是否单独加密，租户能否选择任意 partition key 制造热点，数据驻留和备份保留多久。后文数值是待校准初始阈值。非目标是把无 schema 理解为无契约、让跨分区扫描伪装成低延迟点查，或用墙钟最后写胜把不同 generation 的字段拼成从未存在的文档。

## 客观模型

核心状态为 `{partition_key -> ordered rows/documents, versions, tombstones, schema_generation, index_generation}`。按分区键选择的副本集拥有该分区权威版本；同分区 leader 或 quorum 是条件写仲裁点；schema 注册表拥有字段解释和兼容策略；二级索引是可重建派生状态。接口包括按 key 的 `Get`、携带 `expected_version` 的 `Put/Patch/Delete`，以及只针对声明查询形状的分区内 `Query(partition_key, clustering_range, limit)`。

不变量是条件更新只对匹配版本应用一次；一个读取结果必须对应可解释的文档 generation，不能逐字段拿“最新”再拼装；墓碑在所有可能回归的旧副本越过删除版本前不可回收；同步索引只有与主记录处于同一声明提交边界时才能参与唯一性约束。灵活 schema 表示新增字段不必全表重写，不表示消费者可任意改变字段含义。

设用户逻辑写入 `B` 字节，复制因子 `R`，平均经过 `c` 次压实重写，维护 `I` 个同步索引且索引记录平均为原记录比例 `α`，物理写近似 `W_phys ≈ B×R×(1+c+Iα)`。若查询未给 partition key，需要触达 `P` 个分区并合并，网络和尾延迟至少随候选分区扇出增长；因此“可按任意字段查询”不是 schema 灵活性的免费结果。大分区和低选择性索引会同时扩大读、写和恢复。

## 必然约束

[DEDUCED:wide-column-document-store-partition-key-bounds-write-coordination] 条件更新需要一个能观察旧版本并排定新版本的权威边界。若余额与配额分处两个分区，任一分区的 quorum 都不知道另一边是否成功，无法仅靠单分区 CAS 提供跨分区原子性。最小反例是两个客户端同时读取 `version=4`，分别在不同分区修改文档的额度和状态；若服务把字段级结果拼接，可返回一个从未由任一写者提交的组合。把需共同约束的字段放入同一分区可由一次版本转换仲裁；跨分区需求则必须显式工作流或事务层。

[DEDUCED:wide-column-document-store-tombstone-retention-protects-against-stale-replica-resurrection] 初始 A、B、C 都有 `profile:7.email=old@example`。C 离线时，A、B 写删除墓碑；若 gc grace 在 C 回归前清除了墓碑，C 的旧值只有一个看似有效的版本，读修复已无删除证据压过它，邮箱便复活。删除成功不仅是当前读看不到，还要求墓碑保留覆盖离线副本、修复和备份恢复窗口；若业务允许旧副本永不重入而只能全量重建，保留期才可相应缩短。

[DEDUCED:wide-column-document-store-indexes-turn-write-cost-into-query-latency-budget] 每个同步二级索引都把一次主记录写扩展为索引删除、插入、复制与压实。假设 `R=3,c=4,I=3,α=0.2`，公式给出约 `B×3×5.6=16.8B` 的物理写，不含 WAL 和读放大。索引能降低某个查询延迟，但会占用所有写的容量；低选择性索引仍返回大量候选，可能两边都不划算。因此索引选择必须绑定真实查询 SLO 和写放大预算。

## 从简单方案演进

最简单正确基线是单分区键值加整文档替换：按稳定身份点查，条件版本防丢更新，WAL 后刷入不可变表。查询形状出现分区内时间范围时，再增加 clustering key；字段变化频繁时采用向后兼容 schema generation；只有明确的在线过滤 SLO 无法靠主键模型满足，才增加物化查询表或二级索引。

第一个**待校准**切换指标是：任一分区写 QPS 连续十分钟超过单分区压测安全能力的百分之七十，或分区大小 `p99` 超过一百 GiB，改用时间或散列 bucket、拆大文档，或重新定义访问边界。百分之七十和一百 GiB 需依据介质、恢复时间和查询扇出校准。拆分降低热点，却让跨 bucket 查询增加合并和分页状态，不能只看写吞吐。

第二个**待校准**切换指标是：`W_phys/W_logical` 连续一小时超过十二，且 compaction backlog 预计清空时间超过三十分钟时，暂停新增低选择性索引，调整压实资源、TTL 和墓碑策略，或把索引转为异步物化视图。十二倍和三十分钟应由磁盘余量与故障恢复预算校准；过早切换浪费资源，过晚会让后台债务侵入前台 SLO。

未选择让所有查询自动 scatter-gather。它保持写模型简单，却使分区数增长直接进入每次查询尾延迟并放大隔离困难；若是低频离线分析、结果允许分钟级延迟，导出到分析系统后会重新变优。也未选择字段级无条件 last-write-wins，因为它依赖不可靠墙钟并可拼成虚构 generation；只有字段真正独立且合并函数满足交换、结合和幂等时才考虑。

## 设计决定

本设计以查询驱动的复合 partition key 和 clustering key 建模，每个分区由一个跨故障域复制组持有。同分区条件写在 leader/quorum 比较 `expected_version`，将整文档 generation 或明确字段操作写入 WAL 与 memtable，达到声明持久级别后响应。读取选择强、quorum 或有界陈旧等级，并返回版本；它不会从多个副本逐字段拼接。删除写墓碑，只有修复覆盖最长离线窗口后才压实。

索引分为同步约束索引和异步查询索引：前者必须与主记录在同一可证明提交边界，数量严格受限；后者携带主记录版本，查询可发现落后并回表验证。超时重试使用操作身份与预期版本，冲突返回当前版本而非盲重试。过载时先停后台低优先级扫描和新索引回填，限制大分区写，再按租户拒绝；不得缩短墓碑保留来临时腾空间。

反选方案是把所有文档字段正规化进分布式 SQL。它提供更强跨记录约束，却让本题的高写入、已知分区查询承担不必要的跨 range 事务和 schema 变更成本。若业务后来出现多实体唯一约束、复杂 join 和强事务占主导，SQL 会重新变优；当前边界强调同分区原子性和可预测查询形状。

## 运行与演进

SLI 包括按分区分桶的读写尾延迟、条件冲突率、最大分区、热点份额、物理/逻辑写放大、压实债务、墓碑扫描数、旧副本修复年龄、索引落后版本和 scatter 查询扇出。运行看斜率而非瞬时 CPU：backlog 持续增长意味着系统无法偿还写入。租户按分区大小、写字节、索引数量和查询扇出限额，避免一个灵活 schema 租户拖垮共享压实。

故障时间线：零毫秒 C 副本离线；一秒 A、B 提交删除版本九并确认；一天后压实仍保留墓碑；第三天 C 回归，反熵看到九高于旧值八，先应用墓碑再参与读取；所有副本和备份恢复点越过九后才允许回收。演练反向设置过短 gc grace，应在预生产稳定复现数据复活并由校验器报警，证明监控覆盖正确风险。

schema 迁移先注册可兼容的新 generation，读者支持新旧字段，再双写或后台回填，最后停止旧字段；回滚保持新字段透传，防止旧客户端覆盖未知内容。索引创建先快照构建，再追增量日志，在校验版本一致后原子发布 generation；删除索引也先停止路由再回收数据。备份恢复需带墓碑和 schema generation。敏感字段的加密与擦除不能依赖 SSTable 立即物理删除，应结合密钥销毁、压实证明和保留契约。

## 面试考察本质

这题考察的是：给定“条件写只对同一权威分区的一次版本状态转换生效、删除不会被旧副本复活”的不变量，因为系统无法无代价知道跨分区冲突且 LSM 必须保留删除证据，候选人能否从查询形状推导分区、墓碑与索引边界，并用写放大和扇出预算决定 schema 灵活性的实际范围。

优秀回答会先列查询而非先选数据库，说明 schema flexibility 不等于任意查询，画出离线副本复活时间线，并用 `B×R×(1+c+Iα)` 估算索引和压实成本。常见误区是把跨分区批量写叫事务、看到三副本就提前清墓碑、给每个字段建索引，或按平均文档大小忽略一个巨型热分区。

二十分钟完成实体、partition/clustering key、条件版本和查询形状；四十分钟加入 LSM、墓碑、索引与一致性级别；六十分钟再讨论 schema/index generation、恢复、多租户和重分区。追问应落在某次写确认后哪个副本集拥有版本、某个查询会触达几个分区、旧副本凭什么不能带回删除值，以及新增索引会把每秒物理写提高多少。
