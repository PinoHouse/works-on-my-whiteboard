# 分布式 ID 生成器

## 表面题目

设计分布式 ID 生成器，表面动作是调用 `NextID()` 并拿到一个整数或字符串。真正交付的状态语义是：任意两个成功响应不能相等，超时重试不能让调用方误把两个不同业务对象当成同一个，而“数值更大”是否代表“现实时间更晚”必须单独声明。高吞吐来自让生成器在本地推进序列，但全局唯一的前提包含 worker 身份、时间片和序列都不被复用。

题名经常把唯一、趋势递增、全局严格有序和不可枚举混成一个要求。它们需要不同协调成本：唯一不等于连续，按时间大致排序不等于能裁决业务先后，随机 UUID 也不自动满足数据库索引局部性。本设计要求全局唯一、单个 worker 的成功 ID 按逻辑时间不回退，并允许从 ID 提取近似生成时间；不承诺跨 worker 的严格连续序号，也不把 ID 当授权凭证。

## 反问与边界

先问峰值和突发发号率、活跃 worker 数、ID 位宽、寿命、是否必须十进制、是否暴露给外部、能否接受可枚举，以及数据库索引更看重局部性还是随机分散。语义上要确认调用超时后是否由业务幂等键复用原 ID，跨区域能否各自发号，排序是只为分页还是用于资金或消息先后裁决。若业务真正需要“无空洞且全局严格连续”，每个成功号都必须经过一个序列权威，不能再把本地发号宣传成同一问题。

时间边界包括 NTP 回退上限、虚拟机暂停和恢复、worker 最长失联时间、租约服务不可用时是否宁可停止发号。安全上，时间和 worker 位可能泄露规模与创建时刻；对外订单号可再加不可预测包装，但内部主键仍守住唯一。未知生产数字使用 `Q`、`Δt` 与位宽变量表达。非目标包括用 ID 证明业务提交顺序、在没有业务幂等记录时实现“重试只创建一次”，以及从机器墙钟推出法律时间。

## 客观模型

最小接口为 `AcquireWorker(instance, ttl)`、`NextID(worker_lease, idempotency_key?)` 与 `ReleaseWorker(fencing_token)`。每个生成器本地拥有 `(epoch, worker_id, logical_timestamp, sequence, fencing_token)` 的原子状态，租约协调器拥有 `worker_id -> current_fencing_token, expiry` 的唯一授予记录。数据写入方只有在生成器持有当前 token 时才接受它产生的号段或批次；仅把租约过期时间存在生成器内不足以 fence 暂停后复活的旧进程。

若 `b` 位 ID 分配 `t` 位时间、`w` 位 worker、`s` 位序列，则每 worker 每时间片上限为 `2^s / Δt`，活跃 worker 不超过 `2^w` 时理论总上限为 `2^(w+s) / Δt`。例如 `s=12` 且时间片一毫秒时，单 worker 单毫秒最多发四千零九十六个；第 4097 个不能靠更多空闲机器补救，除非路由把突发拆给其他身份。时间字段寿命约为 `2^t × Δt`，自定义 epoch 必须在耗尽前升级格式。

不变量包括：成功 ID 全局不重复；同一 worker 与逻辑时间片内序列不回退或复用；同一 worker 身份任一时刻只有一个有效发号者；时钟回退时不产生已使用过的 `(timestamp, worker, sequence)`；格式版本可被解码且升级不与旧空间重叠。主导放大不是普通读流量，而是身份双主或时钟回退把一个局部错误持续放大为碰撞。

## 必然约束

[DEDUCED:distributed-id-worker-identity-must-be-fenced-before-local-generation] 本地原子递增只能证明一个进程内不重复，不能证明 `worker_id` 未被另一进程同时使用。最小反例：A 持有 worker 7 后发生长暂停，租约过期；协调器把 worker 7 授给 B。A 恢复但未被 fence，二者在逻辑毫秒 1000 都从 `seq=0` 发出 `(1000,7,0)`。租约记录正确也无法撤回已脱离协调器的 A，因此新 token 必须进入后续可验证的写边界，或旧进程在无法续租时在足够早的安全窗口自停。若 worker ID 永不复用且空间足够大，可省租约，但要承担永久分配和格式耗尽。

[DEDUCED:distributed-id-clock-regression-cannot-preserve-availability-and-monotonic-time-encoding] 假设 ID 的时间字段必须在同一 worker 上单调，物理时钟却从 1005 ms 回到 995 ms。立即使用新墙钟会让时间字段倒退；重复使用 995 的序列空间可能碰撞；等待到 1005 或拒绝会损失可用性；推进逻辑时间则不再声称字段等于真实墙钟。系统只能明确选择等待或拒绝、逻辑时间、允许倒退中的语义组合，无法同时保留真实时间、单调性和无停顿。

[DEDUCED:distributed-id-sequence-width-bounds-per-worker-burst-rate] 固定 `s` 位序列为每个 worker、每时间片提供恰好 `2^s` 个组合。即使全局还有大量 worker，一个热点调用方被固定路由到 worker 3 时，第 `2^s+1` 个请求仍必须等下个时间片、换身份或失败。由此位宽预算是硬容量，不是平均性能调优；盲目回绕序列会直接破坏唯一性。

## 从简单方案演进

最简单正确基线是数据库中的单行序列，每次条件递增并返回新值。它天然给出全局严格顺序，适合低吞吐审计编号。当五分钟窗口内中心序列器利用率持续超过 70%，或发号 `p99 > 5 ms` 且业务只要求唯一而非无空洞连续时，切换到租约 worker 加本地时间序列。它移除每个 ID 的网络协调，却新增身份租约、fencing、时钟异常和位宽规划。

本地生成器在每个时间片递增 sequence，墙钟未前进时继续用上一逻辑时间；检测到回退则按明确策略等待或进入逻辑时间模式。第二个切换指标是任一 worker 的 sequence 使用率超过 80%，或时钟回退超过 2 ms：停止正常快速发号，先等待到安全时间；等待超过 100 ms 就拒绝并告警，或切换到单独编码的逻辑 epoch。这里的 `70%/5 ms` 与 `80%/2 ms/100 ms` 是两个待压测与时钟故障演练校准的策略组，不是测量事实；调高会提高资源利用率但缩小故障与突发余量，调低则更早协调、拒绝或扩位宽。

第三步才按地域预分配互不重叠的 worker/epoch 空间，使常态发号不依赖跨地域往返。租约服务不可用时，已有 worker 只在本地安全余量内继续；接近租约截止前停止，而不是 fail-open 永久发号。格式升级先让消费者理解版本位，再给新 worker 发新格式，不能重写已暴露 ID。

未选择纯随机 UUID 作为默认内部主键，因为本题还要求近似时间排序和紧凑位宽；随机方案消除时钟与 worker 租约，却带来更大索引、无趋势顺序和概率碰撞模型。当调用方只需要不可预测唯一值、索引写放大可接受且不需要解码时间时，它会重新变优。也未选择为每个 ID 走跨地域共识，因为只有严格全局顺序的业务才值得承担该延迟。

## 设计决定

本设计将格式版本、逻辑时间、worker ID 和 sequence 编码进固定宽度 ID。租约协调器以单调 fencing token 授予 worker，生成器只在续租安全窗口内发号；下游批量登记当前 token，拒绝旧 token 生成的批次，从而让暂停实例真正失去写权限。常态 `NextID` 在进程内以原子状态推进，无锁实现也必须保证时间与 sequence 作为一个整体比较交换。

墙钟前进时切新时间片并清零 sequence；短回退时保持上一逻辑时间继续消耗序列，序列接近耗尽则等待或拒绝。长回退进入显式新 epoch 或停机，绝不悄悄回绕。调用超时后的业务重复创建由上游幂等键处理：发号器可以产生多个不同 ID 而都保持唯一，但不能据此证明只创建一个订单。

反选方案是中心号段服务一次分配大块连续区间，本地只递增。它避免逐毫秒时钟回退并降低协调频率，但实例崩溃产生大空洞，号段租约与回收仍需 fencing。当业务接受空洞、需要数值紧凑且每个实例的突发远高于时间序列位宽时，号段模式重新变优。

## 运行与演进

关键 SLI 包括发号成功率与 `p99`、每 worker 的序列高水位、时钟回退幅度、租约续期余量、旧 fencing token 拒绝数、worker 双重授予数和抽样重复检测。监控按 worker 与调用方分解，不能让全局低利用掩盖单 worker 突发。过载时先拒绝无配额调用方，再扩大路由分散；不通过序列回绕或忽略租约维持表面可用。

故障时间线：00:00:00 A 持有 worker 7、token 41；00:00:01 A 暂停，租约到期；00:00:06 B 获得 worker 7、token 42 并登记到下游；00:00:07 A 恢复，尝试发布 token 41 的新号段，被下游拒绝且生成器自停。对照故障注入关闭 fencing 校验，让 A、B 在同一逻辑毫秒从 seq 0 开始，应能复现碰撞反例。恢复过程只重新申请新身份，不能把 A 的本地 sequence 猜测性合并到 B。

格式演进先部署双解码，再灰度新版本 worker，并核对新旧空间无交集；回滚停止分配新格式但保留读取能力直到所有存量 ID 生命周期结束。审计日志记录格式、worker、token 和回退模式，不记录能关联敏感业务的完整外部 ID。多租户配额按预计令牌率分配，单租户热点不得耗尽其他 worker 的序列空间。

## 面试考察本质

这题考察的是：给定“任意两个成功 ID 不相等”的不变量，因为发号节点仅凭本地状态无法知道 worker 身份是否已被复用、墙钟是否可信，候选人能否用 fencing、逻辑时间和位宽预算换取无逐号协调的吞吐，并明确放弃全局连续、绝对时间或故障期可用性中的部分承诺。

优秀回答会先拆分唯一、排序、连续与不可枚举，写出 `2^s / Δt` 的单 worker 上限，用暂停实例 A 和新实例 B 的 `(1000,7,0)` 撞号说明本地自增不够，并给出时钟回退的三选边界。常见误区是说“时间戳不会重复”、只让租约表递增却没有下游 fencing，或用扩机器解决单 worker 序列耗尽。

二十分钟回答应完成位布局、唯一不变量和单序列器基线；四十分钟加入 worker 租约、本地状态、回退策略与容量公式；六十分钟再讨论地域预分配、格式升级、外部不可预测包装和故障演练。追问可以固定在 GC 暂停六秒与时钟回退十毫秒，要求候选人逐步指出谁仍有发号权、哪个字段可能复用以及何时必须拒绝。
