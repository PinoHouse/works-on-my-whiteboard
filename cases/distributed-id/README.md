# 分布式 ID 生成器

## 表面题目

设计分布式 ID 生成器，表面动作是调用 `NextID()` 并拿到一个整数或字符串。真正交付的状态语义是：任意两个成功响应不能相等，超时重试不能让调用方误把两个不同业务对象当成同一个，而“数值更大”是否代表“现实时间更晚”必须单独声明。高吞吐来自让生成器在本地推进序列；全局唯一则要求每次 worker 授予拥有不复用的 `lease_epoch`，并让它与 worker、时间片和序列一起进入 ID。

题名经常把唯一、趋势递增、全局严格有序和不可枚举混成一个要求。它们需要不同协调成本：唯一不等于连续，按时间大致排序不等于能裁决业务先后，随机 UUID 也不自动满足数据库索引局部性。本设计要求全局唯一、单个 worker 的成功 ID 按逻辑时间不回退，并允许从 ID 提取近似生成时间；不承诺跨 worker 的严格连续序号，也不把 ID 当授权凭证。

## 反问与边界

先问峰值和突发发号率、活跃 worker 数、ID 位宽、寿命、是否必须十进制、是否暴露给外部、能否接受可枚举，以及数据库索引更看重局部性还是随机分散。语义上要确认调用超时后是否由业务幂等键复用原 ID，跨区域能否各自发号，排序是只为分页还是用于资金或消息先后裁决。若业务真正需要“无空洞且全局严格连续”，每个成功号都必须经过一个序列权威，不能再把本地发号宣传成同一问题。

时间边界包括 NTP 回退上限、虚拟机暂停和恢复、worker 最长失联时间、租约服务不可用时是否宁可停止发号，以及协调器还能分配多少个 `lease_epoch`。安全上，时间、worker 与 epoch 位可能泄露规模、创建时刻和租约更替频率；对外订单号可再加不可预测包装，但内部主键仍守住唯一。未知生产数字使用 `Q`、`Δt` 与位宽变量表达。非目标包括用 ID 证明业务提交顺序、在没有业务幂等记录时实现“重试只创建一次”，以及从机器墙钟推出法律时间。

## 客观模型

最小接口为 `AcquireWorker(instance, ttl)`、`NextID(worker_lease, idempotency_key?)` 与 `ReleaseWorker(lease_epoch)`。协调器为每次成功授予分配全局单调且不复用的 `lease_epoch`，记录 `worker_id -> current_lease_epoch, expiry`；生成器原子维护 `(lease_epoch, worker_id, logical_timestamp, sequence)`。`lease_epoch` 同时是下游可校验的 fencing token，但它首先是 ID 编码的一部分：旧实例即使在暂停恢复后继续返回 ID，也只能使用旧 epoch 空间，不会与复用同一 worker 的新实例碰撞。进程重启或本地状态丢失不得沿用原 epoch，必须重新申请；一个 epoch 永不转授给另一进程。

本题采用 128 位格式：`4` 位格式版本、`48` 位 `lease_epoch`、`44` 位逻辑毫秒、`16` 位 worker 和 `16` 位 sequence。一般地，若 `s` 位序列与 `w` 位 worker 使用时间片 `Δt`，则每 worker 上限为 `2^s / Δt`，活跃 worker 不超过 `2^w` 时理论总上限为 `2^(w+s) / Δt`；本布局每 worker 每毫秒最多六万五千五百三十六个。位宽代价是 ID 比常见 64 位方案更大；`48` 位全局 epoch 最多允许 `2^48` 次授予，`44` 位毫秒约覆盖五百五十七年，任一字段接近耗尽都必须先升级格式版本并划出不重叠空间。协调器不可用时不能授予新 epoch 或复用 worker，现有进程只在租约安全窗口内继续服务，因此该协议用更大 ID、有限授予次数和故障期可用性换取无需逐号协调的唯一性。

不变量包括：成功 ID 全局不重复；每次授予的 `lease_epoch` 不复用且编码进 ID；同一 epoch、worker 与逻辑时间片内 sequence 不回退或复用；时钟回退时不产生已使用过的 `(lease_epoch, timestamp, worker, sequence)`；格式版本可被解码且升级不与旧空间重叠。同一 worker 可有旧、新两个仍运行的实例，但它们的 epoch 必须不同；“只有一个当前租约持有者”只约束下游写权限，不再被误当作 `NextID` 唯一性的唯一防线。主导放大不是普通读流量，而是 epoch 复用或本地时间状态回退把一个局部错误持续放大为碰撞。

## 必然约束

[DEDUCED:distributed-id-lease-epoch-must-be-encoded-before-worker-id-reuse] 本地原子递增只能证明一个进程内不重复，下游 fencing 也只能拒绝业务写入，不能撤回已经返回给调用方的 `NextID()`。最小反例：A 以 worker 7、epoch 41 发到逻辑毫秒 1000 后长暂停；租约过期，B 复用 worker 7。若 token 不进 ID，B 从 `seq=0` 返回 `(1000,7,0)`，随后恢复的 A 可以先把同一值返回给另一个调用方，之后再被下游拒绝也为时已晚。协议因此要求 B 获得不复用的 epoch 42，并分别编码 `(41,1000,7,0)` 与 `(42,1000,7,0)`；旧 A 即使继续发号也碰不到新 epoch 空间。若 worker ID 永不复用且空间足够大，可省 lease epoch，但要承担永久分配和 worker 位耗尽。

[DEDUCED:distributed-id-clock-regression-cannot-preserve-availability-and-monotonic-time-encoding] 假设 ID 的时间字段必须在同一 worker 上单调，物理时钟却从 1005 ms 回到 995 ms。立即使用新墙钟会让时间字段倒退；重复使用 995 的序列空间可能碰撞；等待到 1005 或拒绝会损失可用性；推进逻辑时间则不再声称字段等于真实墙钟。系统只能明确选择等待或拒绝、逻辑时间、允许倒退中的语义组合，无法同时保留真实时间、单调性和无停顿。

[DEDUCED:distributed-id-sequence-width-bounds-per-worker-burst-rate] 固定 `s` 位序列为每个 worker、每时间片提供恰好 `2^s` 个组合。即使全局还有大量 worker，一个热点调用方被固定路由到 worker 3 时，第 `2^s+1` 个请求仍必须等下个时间片、换身份或失败。由此位宽预算是硬容量，不是平均性能调优；盲目回绕序列会直接破坏唯一性。

## 从简单方案演进

最简单正确基线是数据库中的单行序列，每次条件递增并返回新值。它天然给出全局严格顺序，适合低吞吐审计编号。当五分钟窗口内中心序列器利用率持续超过 70%，或发号 `p99 > 5 ms` 且业务只要求唯一而非无空洞连续时，切换到协调器授予 `lease_epoch`、worker 加本地时间序列。它移除每个 ID 的网络协调，却新增 epoch 分配可用性、字段耗尽、时钟异常和位宽规划。

本地生成器在每个时间片递增 sequence，墙钟未前进时继续用上一逻辑时间；检测到回退则按明确策略等待或进入逻辑时间模式。第二个切换指标是任一 worker 的 sequence 使用率超过 80%，或时钟回退超过 2 ms：停止正常快速发号，先等待到安全时间；等待超过 100 ms 就拒绝并告警，或在同一 `lease_epoch` 内推进明确标记的逻辑时间。这里的 `70%/5 ms` 与 `80%/2 ms/100 ms` 是两个待压测与时钟故障演练校准的策略组，不是测量事实；调高会提高资源利用率但缩小故障与突发余量，调低则更早协调、拒绝或扩位宽。

第三步才把租约协调器部署为跨地域一致的 epoch 权威；`AcquireWorker` 需要跨地域确认，常态 `NextID` 仍不需要网络往返。若要把授予也降为区域本地操作，就必须另加 region 位并预留互不重叠的 epoch 空间，不能继续声称单一全局单调序列。协调器不可用时，已有 worker 只在本地安全余量内继续，接近租约截止前停止；这项停止保护当前写权限和运维边界，而即使旧实例错误地继续发号，epoch 编码仍保护它不碰新授予空间。格式升级先让消费者理解版本位，再给新 worker 发新格式，不能重写已暴露 ID。

未选择纯随机 UUID 作为默认内部主键，因为本题还要求近似时间排序和紧凑位宽；随机方案消除时钟与 worker 租约，却带来更大索引、无趋势顺序和概率碰撞模型。当调用方只需要不可预测唯一值、索引写放大可接受且不需要解码时间时，它会重新变优。也未选择为每个 ID 走跨地域共识，因为只有严格全局顺序的业务才值得承担该延迟。

## 设计决定

本设计将格式版本、协调器单调分配的 `lease_epoch`、逻辑时间、worker ID 和 sequence 编码进 128 位 ID。协调器授予 worker 时必须持久化“epoch 已消费”再返回，失败重试只能得到同一授予结果或新的更大 epoch，绝不回收旧值。下游把 epoch 当 fencing token 拒绝旧实例的业务写入；这是写权限保障，而 ID 唯一性直接来自编码空间隔离。常态 `NextID` 在进程内原子推进时间与 sequence，不访问协调器；同一进程崩溃后不能以旧 epoch 重启。

墙钟前进时切新时间片并清零 sequence；短回退时保持上一逻辑时间继续消耗序列，序列接近耗尽则等待或拒绝。长回退必须停机并向协调器申请新 `lease_epoch`，绝不在旧 epoch 内悄悄回绕。调用超时后的业务重复创建由上游幂等键处理：发号器可以产生多个不同 ID 而都保持唯一，但不能据此证明只创建一个订单。

反选方案是中心号段服务一次分配大块连续区间，本地只递增。它避免逐毫秒时钟回退并降低协调频率，但实例崩溃产生大空洞，号段租约与回收仍需 fencing。当业务接受空洞、需要数值紧凑且每个实例的突发远高于时间序列位宽时，号段模式重新变优。

## 运行与演进

关键 SLI 包括发号成功率与 `p99`、每 worker 的序列高水位、时钟回退幅度、租约续期余量、epoch 分配速率与剩余空间、旧 epoch 写入拒绝数、epoch 重复授予数和抽样重复检测。监控按 worker 与调用方分解，不能让全局低利用掩盖单 worker 突发。过载时先拒绝无配额调用方，再扩大路由分散；不通过复用 epoch、序列回绕或忽略格式耗尽维持表面可用。

故障时间线：00:00:00 A 持有 worker 7、`lease_epoch=41` 并返回 `(41,1000,7,0)`；00:00:01 A 暂停，租约到期；00:00:06 B 获得同一 worker 与新 epoch 42，返回 `(42,1000,7,0)` 并登记到下游；00:00:07 A 恢复，即使错误地继续返回 `(41,1000,7,1)`，也不碰 B 的空间，而其 epoch 41 业务写入被下游拒绝。对照故障注入从编码中删除 epoch，让 A、B 在同一逻辑毫秒从 seq 0 开始，应能复现两个 `NextID` 响应碰撞，证明单靠下游 fencing 不够。恢复过程只重新申请新 epoch，不能把 A 的本地 sequence 猜测性合并到 B。

格式演进先部署双解码，再灰度新版本 worker，并核对新旧空间无交集；回滚停止分配新格式但保留读取能力直到所有存量 ID 生命周期结束。审计日志记录格式、worker、`lease_epoch` 和回退模式，不记录能关联敏感业务的完整外部 ID。多租户配额按预计令牌率分配，单租户热点不得耗尽其他 worker 的序列空间。

## 面试考察本质

这题考察的是：给定“任意两个成功 ID 不相等”的不变量，因为发号节点仅凭本地状态无法知道 worker 身份是否已被复用、墙钟是否可信，候选人能否把协调器单调授予且不复用的 `lease_epoch` 编进 ID，再用逻辑时间和位宽预算换取无逐号协调的吞吐，并明确承担更大 ID、epoch 耗尽与协调器不可授予时的可用性代价。

优秀回答会先拆分唯一、排序、连续与不可枚举，写出 `2^s / Δt` 的单 worker 上限，用暂停实例 A 和新实例 B 在缺少 epoch 时的 `(1000,7,0)` 撞号说明本地自增不够，再说明编码 epoch 后旧实例仍发号为何不碰新空间。常见误区是说“时间戳不会重复”、只把 token 交给下游却不编码进 ID，或用扩机器解决单 worker 序列耗尽。

二十分钟回答应完成含 `lease_epoch` 的位布局、唯一不变量和单序列器基线；四十分钟加入 worker 租约、本地状态、回退策略与容量公式；六十分钟再讨论地域 epoch 号段、格式升级、外部不可预测包装和故障演练。追问可以固定在 GC 暂停六秒与时钟回退十毫秒，要求候选人逐步指出谁仍有当前写权限、旧实例为何仍可能返回 ID、哪个编码空间是否复用以及何时必须拒绝。
