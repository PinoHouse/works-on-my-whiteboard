# 在线状态服务

## 表面题目

设计在线状态服务，让好友列表、聊天页或协作房间显示用户在线、离开或最近活跃。表面上像维护布尔值，实际只能把设备最近一次成功续约解释为带过期时间的观测：设备可能断网但租约尚未过期，也可能连接存在却无人看屏幕。系统没有传感器能精确知道人的真实在线状态。

成功语义因此是“在声明的新鲜度窗口内，聚合状态由至少一个当前会话租约支持”，不是“用户此刻可达”。状态广播是让界面更快更新的 best-effort hint；权威事实是租约、会话世代和过期边界。presence 更不能替聊天消息推进已送达或已读。

## 反问与边界

先问“在线”按账户、设备还是应用前台定义，允许多长误在线与误离线窗口，移动端后台是否续约，是否展示精确 last-seen，隐身与隐私策略如何覆盖技术观测。容量要确认活跃设备 `N`、心跳间隔 `h`、好友关系度数、群规模、跨地域会话与订阅刷新 SLO。

还要区分显式正常退出、网络断开和进程被杀：只有第一种可能快速发出关闭事件，任何关闭事件也可能迟到。安全边界包括好友授权、隐身状态、last-seen 保留、租户隔离和防伪造心跳。非目标是保证即时离线、向所有好友可靠投递每次变化、证明用户可收消息或永久审计所有瞬态在线事件。

## 客观模型

最小接口为 `OpenSession(user,device,session_id)`、`Renew(session_id,epoch,expires_at)`、`Close(session_id,epoch)`、`GetPresence(user)` 与 `Subscribe(subjects)`。权威状态是每设备 `lease -> {user,session_id,region,epoch,expires_at,last_activity}`；用户级状态由当前有效 lease 聚合。广播缓存只携带用户、聚合版本和到期提示，不拥有真实状态。

不变量是：只有未过期且属于当前 session epoch 的租约可贡献在线；旧连接的心跳或关闭不能修改新会话；至少一个有效设备租约才显示可能在线；隐身策略可把技术在线映射为离线，但反向不成立。到期由租约所有者的时间边界裁决，客户端时间不能延长服务端租约。

`N` 个设备每 `h` 秒续约，写率约为 `N/h`；若 TTL 为 `k×h+j`，真实断线到显示过期的上界近似 TTL，`j` 吸收调度与网络抖动。缩短 `h` 能降低发现窗口，却线性放大写入。关系广播还近似 `state_changes×subscriber_degree`，超级账号会形成与心跳写完全不同的热点。

## 必然约束

[DEDUCED:presence-service-online-is-a-leased-observation-not-a-delivery-guarantee] 最后一次续约只证明设备在某个过去时刻可与服务通信。随后网络可能断开、应用可能后台、用户可能离开屏幕。服务能精确保证这份观测最迟在 `expires_at` 失效，却只能 best-effort 推断当前可达、界面展示和消息阅读；任何“在线即已送达”的设计都越过信息边界。

[DEDUCED:presence-service-session-epoch-fences-stale-disconnects] TTL 为 30 秒，A 的 epoch 7 在 0 秒断网，8 秒以 epoch 8 重连，旧网关在 20 秒才报告 epoch 7 关闭。没有世代比较时，迟到关闭会删掉新租约；迟到心跳也可能让旧会话续命。条件写只接受当前 epoch，才保证旧连接不能覆盖新连接。

[DEDUCED:presence-service-heartbeat-interval-trades-freshness-for-amplification] 当 `h` 从 30 秒降到 5 秒，同样设备数会把续约写放大六倍，而真实断线仍只能在 TTL 或可靠关闭后判断。更频繁广播每次心跳还会乘上关系度数。结论是新鲜度、写放大和误判率不可同时取优，必须由产品可接受的提示误差选择。

## 从简单方案演进

最简单正确基线是在一个带 TTL 的存储中为设备维护租约，查询时聚合。它不依赖可靠断开事件，却在大规模轮询时增加读延迟。先加入聚合缓存与只在状态边界变化时广播，而不是广播每次续约；缓存到期不能晚于其支持租约的最早有效边界。

当租约续约延迟 `p99` 超过 TTL 的三分之一，或误离线投诉率超过待校准的在线用户 0.1%，增大 TTL 与抖动余量，并把跨地域界面降级为 `last_seen`。这减少误离线，却延长误在线窗口。若业务必须更快，可优先可靠前台关闭信号，但仍保留 TTL 兜底。

当单用户订阅者数达到单次状态变化安全扇出的 20%，或广播 backlog 最老年龄超过 TTL 的一半，停止逐好友推送，改为读取时查询聚合状态、对大群展示粗粒度人数。三分之一、0.1%、20% 与半个 TTL 都是待压测和产品校准的初始参数；调低会增加成本与抖动敏感性，调高会延迟提示变化。

未选择把每次 online/offline 写入全局可靠总序；审计特定受管终端时它可作为辅助日志，但无法让 liveness 观测更真实。

## 设计决定

本设计由会话目录分配单调 epoch，设备以服务端时间续约 lease；聚合器读取所有未过期的当前世代租约，并应用隐身策略。关闭是加速失效的条件写，不是唯一正确性路径。订阅广播携带聚合版本和最大有效期，客户端过期后重新查询，不能永久相信旧“在线”帧。

故障时间线是：0 秒 epoch 7 有效至 30 秒；5 秒网络断开，服务尚未知；8 秒设备以 epoch 8 重连；20 秒旧关闭到达并被拒绝；38 秒旧 lease 已过期，新 lease 继续支持在线。可保证旧会话不篡改新会话与已提交 lease 最迟自然失效；不能保证 5 秒立刻离线、所有好友收到变化、设备可达或用户在看。

未采用每次心跳同步扇出好友图，因为它把租约刷新变成关系图乘法。只有很小的封闭房间、成员数有硬上限且变化必须低延迟时，逐成员推送才重新合适。

## 运行与演进

核心 SLI 是续约成功率与延迟/TTL 比、过期扫描延迟、误在线与误离线反馈、聚合查询 `p99`、状态广播年龄、每用户订阅度数和旧 epoch 拒绝次数。指标按设备平台、区域和关系度数分桶。过载时先停止 typing 等附加提示，再降低广播频率、对大群转为按需查询，最后延长 TTL；认证与隐私策略不降级。

演练设备断网后快速重连，再延迟发送旧关闭和旧心跳，验证 epoch 8 不受影响且 UI 最迟按 TTL 更新。协议迁移先让服务接受新旧续约格式、只向小流量发新 epoch 字段，再要求旧客户端升级；回滚保留已分配世代，不能让计数回退。

跨地域状态可由会话所属区域聚合，远端显示带时间戳的 last-seen；若区域隔离，宁可陈旧并显式标注，也不凭空宣称离线。隐身与好友权限在查询和广播两处执行。成本按续约写、关系扇出和聚合读取三项归因，而不是按“在线用户数”一个指标。

## 面试考察本质

给定“显示在线只能由未过期、未被旧会话覆盖的租约支持”的不变量，因为服务无法立即知道真实断线，也无法从连接推导人的注意力，候选人应把 presence 限定为带 TTL 的 liveness hint，用 session epoch 阻挡迟到事件，并在新鲜度、心跳写入和关系扇出间取舍。

优秀信号包括拒绝布尔真值、写出 `N/h`、解释正常关闭为何不能成为唯一依据、处理多设备聚合与隐身。常见误区是每次心跳广播全好友、旧连接可覆盖新连接、把在线当可达，或复用消息队列的永久交付语义。

二十分钟回答覆盖 lease、TTL、epoch 与聚合；四十分钟加入关系扇出、误判边界和断连时间线；六十分钟再讨论跨地域、隐私、协议升级与成本。追问应要求候选人明确“这个状态是事实还是提示、最迟何时失效、哪两个指标驱动 TTL 或广播切换”。
