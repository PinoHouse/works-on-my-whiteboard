# Top-K 与高频项统计

## 表面题目

设计 Top-K 与高频项统计，表面上是持续接收事件并返回当前最热门的 K 个 key。真正需要定义的是窗口、事件身份、过滤版本、水位线、迟到与重复语义，以及输出能承诺的误差。原始事件和去重规则是权威输入，Top-K 列表只是按某个窗口物化的近似统计，不存在天然的全局精确名次。

题名容易把“高频”混成累计总量、固定窗口、滑动窗口或衰减热度，也容易把估计值写成事实。本设计聚焦高基数流上的窗口 Top-K；不承诺保存全部原始数据，不用近似榜单作为计费或审计依据。

## 反问与边界

先问 K、事件率、不同 key 数、key 偏斜和窗口类型，再问事件时间还是处理时间、允许迟到多久、结果更新周期、误差界与置信度。业务要的是候选发现、趋势告警还是精确结算；第 K 与第 K+1 无法区分时能否返回并列或不确定区间。重放范围必须与 event_id 去重保留期一起声明。

容量还包括分区数、窗口重叠数、每 key 精确状态、跨分区 merge 和 checkpoint 带宽。恶意租户可制造无限新 key，需要长度限制、租户配额和盐化哈希；输出可能泄露小群体行为，应设最小频次。非目标是凭 sketch 恢复每条事件、保证任意过滤版本可回溯，以及把处理水位线伪装成现实时间。

## 客观模型

输入记录为 `(event_id,key,event_time,partition,filter_version)`，窗口定义为 `(start,end,allowed_lateness)`。事件日志拥有分区顺序与原始事实，窗口协调器拥有 watermark 和关闭代次，聚合器拥有可重建 sketch、候选堆与输出版本。不同分区没有事件全序；输出 rank 是估计频次排序。

不变量是结果必须携带窗口、过滤版本、输入 epoch、误差参数与新鲜度；重复投递在声明去重期内最多计一次；晚到与过期按固定规则处理；未观测或未复核 key 不得声称精确第 K。高基数使精确 `key->count` 内存随不同值数增长，滑动窗口又按桶数放大状态。

Count-Min Sketch 取宽度 `w=ceil(e/ε)`、深度 `d=ceil(ln(1/δ))` 时，以至少 `1-δ` 概率满足 `f(x)<=f_hat(x)<=f(x)+εN`。合并相同参数和 hash 版本的分区 sketch 可相加；参数不同则不能无损合并。候选堆只保存可能的重项，并不证明未保存 key 不会越界。

## 必然约束

[DEDUCED:top-k-heavy-hitters-rank-is-an-estimate-with-window-and-error-semantics] 同一 key 在不同窗口和过滤版本下频次不同，sketch 只提供概率上界，因此脱离元数据的“第一名”不可验证。最小反例是 A 在过去一小时有 100 次、过去一天有 100 次，B 在过去一小时 90 次、过去一天 1000 次；不写窗口时 A、B 都可被称第一。输出必须绑定窗口、N、`ε/δ` 与 watermark，事件日志才是权威输入；只有完整保存并精确计数时才能标 exact。

[DEDUCED:top-k-heavy-hitters-approximation-cannot-resolve-near-boundary-order] 当 `f_K-f_(K+1)` 小于估计误差，两个区间重叠，任何只读 sketch 都无法保证严格顺序。具体反例为 `N=1,000,000`、`ε=0.001`，误差上界 1000；A 真值 10,100，B 真值 9,900，仅差 200。若 A 因碰撞高估 500、B 高估 900，B 的估计更高。故当边界间隙不够大，应精确二次计数、返回并列或标不确定，不能把估计第二名写成事实。

[DEDUCED:top-k-heavy-hitters-deduplication-horizon-bounds-replay-correctness] event_id 状态被回收后，系统无法区分首次到达和旧事件重放。事件序列是分区已处理到 offset 9000，checkpoint 损坏后从 8500 重放，但去重表只覆盖最近 300 个 offset，即 8700 到 8999；500 个重放事件中，8500 到 8699 的 200 个已越出去重范围，会再次进入 sketch。即使总量能发现异常，也不知道这 200 个事件分别应从哪个 key 减除。恢复必须重建仍有原始日志的窗口，或把输入 epoch 标为 stale；只有上游保证分区内恰好一次且不可回退时，才能缩短本地去重。

## 从简单方案演进

最简单正确基线是有界窗口内为每个 key 精确计数，关闭时全量排序。它易解释，但内存随基数增长、排序成本约 `O(U log U)`。当唯一 key 状态逼近分区预算时，改为 sketch 加候选堆，定期对边界候选从可保留日志或精确侧表复核；新增碰撞误差、参数版本和候选漏召回。

第一个待回放校准指标是 `εN` 超过第 K 边界候选间隙的 `25%`，或抽样精确核验的 Top-K 差异超过 `1%`：扩大 sketch、精确复核边界并标示不确定。第二个待压测校准指标是精确 key 状态内存超过分区预算 `70%`，或唯一 key 速率超过安全 merge 吞吐：切换 sketch+候选堆、增加预聚合或缩短窗口。两组值是初始策略，不是实测；调低会增加内存与复核成本，调高会提高漏排和崩溃风险。

多分区时本地先按相同 hash/参数物化，再树形 merge，避免每事件跨网；窗口重叠过多则以时间桶相减组合。升级过滤规则创建新 filter version 并行计算，不把旧状态直接解释成新语义。

## 设计决定

本设计以分区事件日志为事实源，按 event_id 在保留期内去重，以事件时间窗口和 watermark 关闭。每分区维护相同参数的 sketch 与有界候选堆，聚合层合并后对第 K 附近候选做精确复核；无法复核时输出估计区间和 provisional 标志。更新在微批边界物化，而非每个查询扫描原始日志。

checkpoint 与输入 offset、窗口、filter/hash version 原子绑定；恢复到较早点时先验证去重范围，超出则重建或标 stale。反选“始终维护精确全量计数”，当 key 基数有硬上限、窗口短且审计要求绝对精确时重新变优。也不选择无界累计 sketch，因为旧热点会永久压制新趋势且无法按窗口解释。

查询超时返回上一份带年龄的快照，不猜测最新名次；merge 参数不匹配直接拒绝。对计费、配额等硬决策，调用方必须读取精确账本，不能把 Top-K 估计转成写权限。

## 运行与演进

SLI 包括输入 watermark 年龄、重复过滤率、迟到丢弃率、checkpoint 恢复时间、状态字节、merge 延迟、边界区间重叠率、抽样精确偏差和 provisional 年龄。过载时先降低发布频率与非关键维度，再缩短非审计窗口；不能隐式调大 `ε` 而保持同一结果标签。

故障演练按 0 分钟处理到 offset 9000、1 分钟 checkpoint 损坏回到 8500、2 分钟发现去重只覆盖 8700 到 8999、无法识别更早的 200 个重放事件，3 分钟拒绝直接 merge、5 分钟从保留日志重建窗口。若日志已过期，输出标 stale 并等待新窗口，不手工从总量减 200。演练核对每个结果携带新 input epoch。

参数升级先双算同一事件样本，比较召回、误差和内存，再切输出版本；回滚保留旧 hash。租户分别限基数和事件率，热门 key 不得让单分区网络饱和；隐私阈值与删除请求在精确事实层处理，派生 sketch 通过窗口过期或整代重建清除。

## 面试考察本质

给定“频次输出必须带可解释窗口、输入范围和误差界”这一不变量，因为处理节点没有保存所有 key 的精确次数，且去重状态过期后无法识别重放，候选人应推导出 Top-K 是派生估计，并在状态预算、边界不确定、更新延迟与重建成本之间选择。

优秀回答会写出 CMS 误差，指出第 K 间隙小于误差时不能定序，区分 event time、watermark 和处理时间，并用 offset 重放反例说明去重范围。常见误区是声称 heap 单独能找到所有重项、合并不同 hash 的 sketch，或把估计榜用于账单。

二十分钟完成窗口、事实源和精确基线；四十分钟加入 sketch、候选复核、重放与 checkpoint；六十分钟再讨论滑动窗口、参数升级、隐私和多租户。追问应要求明确每个数字是 exact、estimate 还是 provisional。
