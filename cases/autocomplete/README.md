# 自动补全

## 表面题目

用户输入短前缀，系统在很小延迟内返回若干可接受的查询或实体候选。成功不是把历史最高频字符串原样列出，而是在当前语言、地域、产品策略和隐私约束下返回稳定且及时的 Top-K。题名隐藏了两个现实：用户每敲一个字符都会发请求，短前缀形成极端热点；热度来自行为流，是滞后的统计估计，不是永恒事实。

## 反问与边界

先问候选来自编辑语料、公开实体还是用户查询日志；是否个性化，多语言如何规范化，拼写纠错是否在范围内。确认首字节 p99、最大 K、空前缀行为、趋势生效时限、删除 SLO、最低匿名阈值和未成年人/敏感词规则。读写比高不代表写便宜，因为一次事件可能更新所有前缀。还需问地域与租户隔离、客户端是否可缓存、结果是否必须固定顺序。本题不声称历史点击等于质量，也不把个性化画像作为默认必要条件。

权威事实源是已批准候选语料、当前资格策略及可删除的聚合事件；trie、prefix Top-K、趋势榜和边缘缓存都是派生物。它们必须报告 corpus_version、policy_generation 和 window_end。更快刷新会增加聚合、构建、发布和缓存失效率。

## 客观模型

接口为 `Suggest(prefix, locale, limit, cursorVersion)` 和 `ApplyCorpus(change, version)`；事件聚合输入为去标识化 `(term_id, event_time, weight)`。候选记录包含 term_id、展示文本、资格代际和多窗口分数。若词条长度为 L，每次事件朴素更新 L 个前缀，写放大 `A = Σ events * min(L,Lmax)`；每秒十万事件、平均 12 字符会产生每秒一百二十万次前缀增量。

分数可写成 `score = α*log(1+count_30d)+β*count_10m+γ*editorial-quality-penalty`，其中窗口终点必须固定。查询选择一个完整发布版本，按规范化前缀读取候选，再用当前策略代际做最终过滤。新鲜度为 `now-window_end`，召回可用离线批准集合上的 `recall@K` 表示，但该值必须实测，正文不虚构数字。

短前缀的请求量并不均匀，可用 `hotspot_ratio = hottest_prefix_qps / total_qps` 描述偏斜；该比例越高，单个 trie 分片和缓存失效风暴越危险。稳定分页通常没有意义，但同一次连续输入会话应固定规范化器和候选版本，避免多字符之间出现由发布切换造成的倒退。

## 必然约束

[DEDUCED:autocomplete-index-must-bind-corpus-and-policy-version] 前缀候选必须绑定语料版本与资格策略版本，否则一次查询会混合已发布词条和已删除或禁止的旧候选。反例是构建到一半时先发布 `ca` 分片、后发布 `cat` 分片，用户继续输入会看到候选凭空回退；原子版本指针或查询固定版本才能避免混代。

[DEDUCED:autocomplete-trend-and-baseline-need-different-windows] 长期流行度与突发趋势的时间尺度不同，单一累计计数无法同时保持稳定相关性与快速响应。一条新事件对十年累计数影响接近零，而只用十分钟窗口会在低流量时剧烈抖动；因此至少拆分基线与短窗，再明确平滑和最小样本。

[DEDUCED:autocomplete-deletion-must-fence-derived-copies] 隐私删除只有在所有可服务的 trie、缓存和回退版本都受删除代际栅栏约束后才真正生效。删除源记录却继续服务旧快照会泄露查询；当前 deny generation 必须在读路径覆盖旧候选，后台物理清理只负责成本回收。

## 从简单方案演进

小语料可按规范化字符串排序，在前缀范围内扫描并按编辑分数排序。候选数使查询 p99 超预算，或短前缀扫描行数 p95 超一万时，预计算每个前缀 Top-K。它把查询降为直接读取，却产生约 `unique prefixes * K` 状态和事件写放大。

第二步离线全量构建不可变 trie，原子切换版本，适合日级变化。当趋势发现延迟超过业务窗口的一半，增加短窗流聚合并以小 delta overlay 覆盖基线；overlay 过大将拉高合并延迟和缓存 miss。当 overlay/基线候选比持续超过百分之十，或合并 CPU 超七成，触发新基线构建。短前缀热点可边缘缓存，但缓存键必须含 locale、版本和策略代际。

更快刷新会让缓存频繁失效并扩大构建并发，因此不是免费提高实时性。未选“每次请求实时扫描原始事件”，因为热点前缀会重复做相同工作；当语料只有数百项且 QPS 很低，它重新成为最简单方案。也不默认用每用户模型；只有个性化收益经业务校准且隐私预算成立才引入。

## 设计决定

选择不可变基线 trie 加短窗趋势 overlay。构建器从固定语料版本和窗口终点生成分片，完成校验后原子发布 manifest；查询固定 manifest，合并两路候选，用当前 deny generation 最终过滤，稳定以 `(score, term_id)` 排序。过滤后不足 K 不回退到旧策略，而从同版本的更长候选表补齐。

事件至少一次到达时用 event_id 去重，迟到事件只进入仍开放窗口；超时返回同版本较小 K，不能混用另一代。趋势管道故障时降级到标记了 age 的基线。对词条删除先推进 deny fence，再异步重建与清缓存。未选择可变共享 trie，因其发布中间态难以回滚；单节点内部工具可以选它。

## 运行与演进

观察端到端 p99、空结果率、prefix 热点偏斜、corpus/window age、版本混用数、删除后命中数、趋势 `recall@K` 和缓存命中。过载先关闭昂贵个性化，再减小 overlay/K，最后只返回编辑候选；敏感词与删除栅栏不可降级。

故障时间线：12:00 删除 term 42 并提交 deny generation 18；12:00:01 边缘仍持基线 17，但读时检查 18，term 42 被滤除；12:05 物理重建发布。若边缘看不到 generation 18，则按删除 SLO 选择停止服务而非猜测允许。灰度新分词器时双构建并比较前缀覆盖和空结果，不就地改旧索引。压测切换指标是热点前缀 p99 超预算、overlay 合并 CPU 连续七成；业务校准指标是趋势发现时限、最低匿名量和删除传播上限。

## 面试考察本质

本题本质是：给定“低延迟返回当前合格且不泄露已删除候选”的不变量，因为热度、趋势和策略只能异步聚合与发布，候选人应推导基线/短窗双时间尺度、版本快照和删除栅栏的不可兼得，再按趋势价值、缓存成本与隐私风险决定刷新频率。

优秀信号包括量化前缀写放大、识别空前缀热点、固定排序与版本、把趋势质量说成待测召回而非事实。常见误区是一个累计计数包打天下、删除只改数据库、缓存键不含版本。追问可进入 Unicode 规范化、低频匿名、事件迟到与热点分片。20 分钟讲 trie 与版本，40 分钟补趋势 overlay 和删除，60 分钟讨论多语言、反作弊和灰度评估。
