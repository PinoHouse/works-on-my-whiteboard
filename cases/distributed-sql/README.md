# 分布式 SQL 数据库

## 表面题目

设计分布式 SQL 数据库并不是把单机 SQL 前面加一层分片路由。用户提交的是一组带谓词、约束和索引副作用的读写，成功意味着整个事务可在声明故障范围恢复，并能与其他已提交事务一起解释为某个串行顺序。主流程包含解析与规划、读取一致快照、在多个 range 写入意图、冲突检查、持久化全局决定和解析参与者。每个 range 的复制组拥有本范围版本，事务记录拥有 commit 或 abort 的全局决定；无状态协调进程不能凭本地内存宣告最终成功。

题目容易混淆“SQL 语法可用”和“事务语义成立”。单范围条件更新、跨范围转账、只读分析和跨地域写的协调下界完全不同。本设计提供 MVCC、可串行化事务和二级索引，不承诺任意跨地域事务都同时具有局域网延迟，也不把分析扫描与在线事务放在同一资源队列中。

## 反问与边界

首先确认事务的最大写集、参与 range 数分布、热点行、索引与外键数量、读写比和冲突率，再问隔离级别究竟需要可串行化、快照隔离还是只读陈旧快照。一次 commit 成功要抵抗机器、可用区还是地域故障；协调器超时后客户端如何查询最终状态；读是否允许看见未解析意图；数据库时钟误差和跨地域 RTT 是否可控。这些答案决定提交记录、时间戳与锁策略。

还需明确延迟和局部性：同区域事务的 `p99`、跨地域事务占比、数据驻留、租户资源隔离、长事务上限、模式变更频率和备份恢复点。未知数字用参与者 `p`、每范围复制延迟 `L_i` 与协调开销 `C(p)` 表示。后文阈值均标为待校准，不是生产测量。非目标是让任意跨 range 查询免费继承单范围延迟、以缓存掩盖提交尚未完成，以及在失去多数时继续接受会分叉的强事务。

## 客观模型

每个 range 保存 `{MVCC versions, intents, leaseholder, replicated log, applied_index}`；事务保存 `{txn_id, read_timestamp, write_set, commit_timestamp, status, epoch}`。range 复制组拥有数据和本地提交序；可恢复事务记录拥有全局状态；时间戳服务或混合逻辑时钟只提供排序候选，最终可见性仍由冲突规则与提交决定共同约束。接口可抽象为 `Begin`、`Read`、`Write`、`Commit` 和 `QueryTxnStatus`。

不变量包括：已提交事务的全部写集原子可见；所有成功事务可串行解释；读取不把部分已提交写与部分旧写拼接；条件约束按该串行点检查；客户端重试同一 `txn_id` 不重复提交副作用。每个 range 的多数派日志提交是该范围耐久边界，跨 range 事务只有在决定记录可恢复且所有必要参与者能够解析时才可向客户端声明成功。

若事务参与 `p` 个 range，第 `i` 个范围完成必要复制的延迟为 `L_i`，协调与决定传播为 `C(p)`，则 `L_commit ≥ max(L_1,...,L_p)+C(p)`。若每个参与者独立以概率 `q` 需要重试，至少一次重试的概率为 `1-(1-q)^p`。索引、外键和热点行会增加有效 `p` 或排队，而平均单范围延迟无法预测事务尾部。容量还要计每个逻辑写在 WAL、MVCC 版本、复制日志和同步索引中的写放大。

## 必然约束

[DEDUCED:distributed-sql-commit-acknowledgement-bounds-transaction-durability] 若转账写集涉及 A、B 两个 range，客户端成功必须意味着两者都已进入可恢复的提交路径。最小反例是 A 的扣款已多数落盘，B 的加款只在协调器内存排队，系统便返回 commit；随后协调器和 B 当前领导者断电，恢复后 A 可见 `-100` 而 B 没有 `+100`。这不是短暂陈旧，而是一个已确认事务永久缺少部分写集。因此确认边界必须覆盖每个必要参与者及声明故障域；只读异步副作用可拆出事务，但响应语义也必须改变。

[DEDUCED:distributed-sql-cross-range-atomicity-requires-a-recoverable-decision-record] 参与者看到 prepare 只知道“可能提交”，看不到其他 range 和协调器的最终决定。事件序列是：协调器让 A、B 写入意图；持久化 commit 决定后，仅向 A 发送解析便崩溃；B 超时。若 B 自行把超时解释为 abort，就与已提交决定冲突；若一律当 commit，而决定其实从未持久化，也会暴露幽灵事务。故必须有参与者都能定位的可恢复记录和确定的状态解析，局部超时只能触发查询或接管，不能创造决定。

[DEDUCED:distributed-sql-participant-count-amplifies-commit-tail-latency] 原子成功至少等待最慢必要 range，故参与者增加不仅线性增加消息，还增加碰到慢副本和冲突的机会。即使每个范围仅有百分之一重试概率，`p=20` 时至少一次重试约为 `1-0.99^20≈18.2%`。把协调器横向扩容不能消除最慢参与者下界；只有减少共同原子写集、改善局部性或弱化契约才会改变它。

## 从简单方案演进

最简单正确基线是单机数据库：WAL 持久化 commit，锁或 MVCC 给出隔离，所有表和索引共享一个事务所有者。机器容量或故障目标越界时，先对同一数据复制为单个共识组；这提高耐久性却仍保持一个提交序。只有总容量或独立热点超过单组安全吞吐，才按主键 range 分区，并优先让常共同访问的表和索引 colocate。

第一个**待校准**切换指标是：跨 range 事务占比超过百分之十五，且 `p95(participant_count)>4` 令 commit `p99` 超过一百五十毫秒时，重新设计分区键、colocate 关联索引，或把可异步通知拆成 outbox 工作流。阈值要用真实 RTT、冲突和业务延迟预算校准；降低参与者会减少尾延迟，但异步拆分新增补偿、对账和中间状态。

第二个**待校准**切换指标是：跨地域严格事务的 commit `p99` 连续十五分钟超过四百毫秒，且网络 RTT 占预算百分之六十以上时，把写归属到单主地域并异步复制，前提是业务明确接受地域灾难的 RPO。四百毫秒与百分之六十不是普适事实；改变它们会交换全球写便利、故障切换数据风险和用户延迟，不能以增加连接池掩盖物理距离。

读取扩展先用 follower 上已关闭时间戳的陈旧快照，分析扫描再进入独立副本或导出流水线。未选择默认无协调的跨分片乐观写，因为唯一约束、外键和二级索引仍要求全局决定，冲突只会推迟到提交尾部；若工作负载几乎都是单 range、冲突极低且业务接受快照隔离，它会重新变优。也未选择全局单领导者，因为其简单性以所有写的跨地域 RTT 和单点吞吐为代价。

## 设计决定

本设计按 range 建立跨故障域共识组，以 MVCC 时间戳读取快照，以写意图隔离未决事务。协调器选取一个可复制的事务记录作为状态所有者；提交时先验证读写冲突与约束，使各参与 range 持久化必要状态，再持久化唯一 commit/abort 决定并解析意图。客户端只有在 commit 决定和各参与者达到声明的可恢复边界后收到成功。协调器崩溃后，任一参与者可按 `txn_id` 查询记录并幂等接管解析。

超时返回“结果未知”而不是擅自 abort，客户端重试同一事务身份或查询状态。读者遇到意图时可等待、推动解析或依据已知提交记录读取正确版本，不能拼接部分新值。死锁和高优先级事务使用有界等待与明确牺牲者；热点行通过排队、公平限流或业务层聚合处理。强一致多数不可用时停止 commit，陈旧只读可在明确时间戳边界内继续。

反选方案是经典中心化两阶段提交协调器仅把决定保存在本地磁盘。它实现简单，却使协调器故障域成为所有参与者的阻塞点。若部署范围单一、协调器磁盘本身已有与数据相同的高可用保证且事务规模很小，该方案会重新变优；在多 range 故障恢复目标下，可复制事务记录更符合成功语义。

## 运行与演进

核心 SLI 包括 commit/abort/unknown 比例、按参与者数分桶的提交延迟、意图年龄、事务记录解析延迟、串行化重试率、热点 range 排队、复制提交索引落后和已确认事务缺写计数。过载时先限制长扫描和低优先级事务，缩短新长事务准入，再按租户公平排队；不可通过降低某个参与 range 的持久确认数维持表面成功率。

故障演练时间线：零毫秒事务 T 在 A、B、C 写入意图；五十毫秒可恢复记录提交 `COMMITTED@ts=90`；六十毫秒 A 完成解析，协调器随后崩溃；B、C 保留意图。读者在七十毫秒推动状态查询，看到同一 commit 决定并读取提交版本；新协调器在一百毫秒幂等解析 B、C。另一轮在决定记录提交前崩溃，应由确定规则 abort。演练验证任何时刻都不会出现 A 新、B 旧却被当成完整事务的结果。

模式变更使用可版本化 descriptor，先让所有节点读新旧格式，再后台回填，最后切换写入约束；索引创建必须计入额外参与者和写放大。升级按 range 灰度，回滚保持读取新格式能力。备份必须在同一时间戳恢复事务一致快照并演练。租户级并发、扫描字节和事务参与者数分别限额；审计记录事务身份与决定而不泄露行值。跨地域拓扑变更先复制并追平，再经共识改变成员，不能让路由先行创造双写。

## 面试考察本质

这题考察的是：给定“一个 SQL 事务成功就表示整组读写可恢复且可串行解释”的不变量，因为单个 range 不知道其他参与者和协调器是否已经决定 commit，候选人能否推导出可恢复决定、参与者持久确认和串行化冲突的必要性，并按事务局部性与跨地域风险交换延迟、可用性和原子范围。

优秀回答会区分 range 共识与跨 range 原子性，画出决定已提交但部分意图未解析的时间线，用 `max(L_i)+C(p)` 解释尾延迟，并说明超时为何是未知而非自动失败。常见误区是把“每个分片都有三副本”当成全局事务完成、让读者跳过未决意图、或用墙钟最后写胜处理约束冲突。

二十分钟完成接口、MVCC、range 与事务状态；四十分钟加入提交协议、冲突、索引和失败接管；六十分钟再讨论地域归属、模式迁移、热点和备份。追问应落到某个 commit 响应发出时：哪些范围已经持久化，唯一决定在哪里，协调器消失后谁能证明结果，以及读取何时能观察整组写。
