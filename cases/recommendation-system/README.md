# 推荐系统

## 表面题目

设计推荐系统，表面流程是用户打开页面，系统从海量 item 中选出若干条并按相关性展示。真正的状态分成权威资格与版本化派生物：目录、库存、隐私、年龄限制和用户屏蔽决定 item 此刻能否出现；反馈日志、特征、embedding、召回集合与模型分只帮助排序。推荐成功不仅是点击率高，还要保证每次曝光可归因且不越过硬资格。

推荐顺序没有权威全序，它在请求时由用户、候选和策略共同物化。训练截止时间、在线特征时间和资格版本是不同新鲜度。本设计聚焦候选召回、在线重排与探索，不把广告预算、搜索关键词或内容审核模型训练本身混入核心。

## 反问与边界

先问业务目标是观看时长、购买、长期留存还是多目标约束，展示 K、候选规模 R、延迟 SLO 和冷启动策略。再问删除、缺货、屏蔽、地区与年龄限制多久必须生效，是否允许旧候选但展示前复核；反馈事件的曝光、点击和转化如何去重与归因，模型版本是否会话固定。

容量按请求率、每请求召回数、特征 join、向量索引副本和重排算力规划，爆款 item 与活跃用户会制造特征热点。需明确探索比例、单用户风险、内容方曝光公平、多租户配额、敏感特征使用和隐私删除。非目标是把模型分当事实真值、承诺同一请求永远同序，以及让点击率覆盖合规、库存或屏蔽。

## 客观模型

最小接口为 `Recommend(user,context,K,request_id)`、`RecordExposure(request,item,rank,versions)`、`RecordFeedback(event_id,request,item,type)` 与 `SetEligibility(item,expected_version,state)`. 目录/库存/隐私服务拥有 eligibility；反馈日志拥有去重事件；模型注册表拥有 `model_digest` 与 training cutoff；特征库、向量索引、召回集和 rank 都是带版本的派生状态。

不变量是任何已删除、无库存、未授权或被用户屏蔽的 item 不得展示；曝光记录携带候选、模型、特征和策略版本；模型分不可跨版本解释为同一真值；探索不得越过用户和租户预算。物化分为离线训练/索引、近线特征与请求时过滤/重排，不采用按关注者写时扇出。

一次请求先召回 `R` 个、资格过滤后剩 `E` 个、展示 `K` 个，在线成本近似 `O(R*feature_cost+E*rank_cost)`。若索引变更滞后 `L`，单位时间下架率为 `q`，旧候选污染随 `q*L` 增长，但展示前权威过滤应把实际违规曝光降为零。长尾 item 增加索引，爆款则增加同一特征键读压。

## 必然约束

[DEDUCED:recommendation-system-eligibility-is-authoritative-while-model-rank-is-derived] 模型只掌握训练或缓存特征，无法授予库存、隐私和屏蔽权限。最小反例是商品 X 10:00 下架，向量索引需 15 分钟更新；10:01 X 被召回并得最高分。若直接展示，用户点击失败且违反资格；在线读取 eligibility v9 后过滤，rank 只能在剩余候选中重算。若目录很小可在召回前同步筛完，仍是资格先于模型，而不是模型分足够高就例外。

[DEDUCED:recommendation-system-offline-model-freshness-does-not-imply-online-feature-freshness] 新发布模型 M17 的训练截止到今天，并不意味着库存、用户屏蔽或近一分钟反馈已进入在线索引。事件序列是 M17/F8 在 0 秒发布，30 秒 X 下架到 eligibility v9，60 秒索引仍为 v8。仅按 model age 监控会宣称系统新鲜，却继续召回 X。模型、特征、索引与资格各自需要版本和 lag；前者可决定相关性质量，后者决定是否允许展示。

[DEDUCED:recommendation-system-exploration-requires-an-explicit-user-and-tenant-budget] 探索把本可给高置信候选的曝光分配给不确定项，因此风险不是总体平均可以抹平的。反例是全站探索 5%，但随机实现连续把某用户 20 次请求中的 8 次给低质量内容，同时小租户库存被大租户实验挤出。总体 CTR 可能仍上升，个体损失与供给公平却失控。应按用户、内容方和租户设日预算、冷却和硬安全过滤；只有无个体风险且曝光同质的离线实验才可只看全局比例。

## 从简单方案演进

最简单正确基线是在合格小目录上按规则或热门度请求时排序，每次同步资格检查。目录增长后全量评分越过延迟预算，先离线生成候选与特征、在线过滤重排；新增索引陈旧、版本 join 和召回漏失。个性化进一步引入 embedding 与反馈，但曝光事件必须携带版本，否则无法判断质量变化来自模型还是数据。

第一个待回放校准指标是召回后 eligibility 过滤率超过 `3%`，或库存/隐私变化到索引可见的 `p99` 超过 `5 min`：为高变 item 做增量索引、缩短候选缓存；展示前在线过滤始终保留。第二个待实验校准指标是灰度模型投诉/隐藏率相对基线上升 `10%`，或单用户探索曝光超过日预算 `5%`：停止放量并回滚。两组值不是生产测量；调低会增加索引写与保守回滚，调高会扩大无效曝光和用户损失。

请求增长时分召回、轻排和重排阶段，对 `R/E/K` 各设预算；过载先缩小低价值召回源和重排特征，而非跳过资格。模型和索引版本并行构建，流量灰度后再回收旧版。

## 设计决定

本设计由多个召回源产生带版本候选，在线先批量读取当前 eligibility 和用户屏蔽做硬过滤，再取近线特征重排，最后执行多样性与受预算探索。曝光日志原子记录 request、最终 rank、资格版本、model/feature/index version；反馈用 event_id 去重并关联曝光，过滤失败不被误标成用户负反馈。

索引或特征超时可用较小的已验证热门候选降级，仍需资格检查。缓存首页在真正展示前重新过滤，不能因 model version 未变跳过。反选“每次请求全量实时计算所有 item”，在目录很小、资格极速变化且延迟预算宽松时重新变优；大目录中它把计算和尾延迟放大到不可控。

也不选择离线一次排好完整用户列表，因为兴趣与库存变化会迅速使它陈旧；对低频邮件摘要且允许长新鲜度时，离线列表可能更经济，但发送时仍做硬过滤。

## 运行与演进

SLI 包括合格候选率、违规曝光数、eligibility/index/feature lag、召回覆盖、重排 `p99`、版本归因完整率、投诉与隐藏、单用户探索预算消耗、内容方曝光偏斜和单位有效曝光成本。过载时先关闭昂贵探索与深层特征，再缩小 R；硬过滤和曝光谱系优先。

故障演练：0 秒 M17/F8 发布，30 秒 X 下架为 v9，60 秒 v8 索引召回 X，70 秒在线过滤移除并记录 `filtered_unavailable`，5 分钟旧首页缓存仍在展示前复核。索引修复只重建受影响分片，训练管线排除被过滤事件作为负反馈。演练验证任一区域违规曝光为零且候选不足有明确降级。

新模型先离线固定集、影子和小流量灰度，比较质量、安全、延迟与公平；回滚按 model digest 且会话可固定版本。隐私删除覆盖反馈、embedding、特征和调试样本，重建索引。租户特征、探索额度和模型适配器隔离，监控不得泄露其他租户人群属性。

## 面试考察本质

给定“推荐绝不越过当前资格且每次曝光可归因”这一不变量，因为离线模型和向量索引不会立即知道下架、屏蔽与个体探索损失，候选人应推导出权威 eligibility、版本化候选和请求时派生 rank 的分层，并在召回成本、特征新鲜度、质量与公平预算之间决策。

优秀回答会明确 `R/E/K`，区分模型与在线特征新鲜度，用 X 下架反例证明展示前过滤，并让探索有用户/租户预算。常见误区是只谈 CTR、把合规做成负权重，或把点击失败当负反馈而不记录过滤原因。

二十分钟完成资格、召回和重排；四十分钟加入版本谱系、缓存、反馈和故障线；六十分钟再讨论探索、公平、隐私与多模型灰度。追问应要求候选人逐项指出哪个状态有写权限、哪个 rank 可重建。
