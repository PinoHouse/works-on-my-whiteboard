# 预约系统

## 表面题目

设计预约系统，表面上是让用户搜索医生、会议室或服务人员，选择时间并收到确认。真正发生的状态变化是某个资源在一段连续时间内的服务能力被分配给一个预约；“预约成功”必须意味着所有与该区间相交的已确认预约总量不超过容量。题名还容易把预约确认、付款授权、实际到场和服务完成混成一个成功状态，但它们分别是未来能力分配、资金承诺和现实履约。

本设计聚焦有明确时长、允许短时 hold、可能取消和爽约的资源预约。日历搜索可以返回近似快照，确认必须精确；跨资源套餐、医疗诊断记录和排班优化不是主流程。系统宁可在资源权威不可用时暂停确认，也不能先答应两个用户同一个不可替代时段。

## 反问与边界

先问资源容量是一个人、一间房还是可并行的多个席位；服务时长固定还是由项目决定，清洁缓冲是否计入相交判断，改期要求新旧时段原子交换还是允许先占后退。还要定义成功语义：搜索有空只代表快照，hold 只代表临时权利，confirmed 才是分配承诺，checked-in 与 completed 才是外部履约事实。付款是预约前授权、到场后捕获，还是无需支付，都会改变补偿顺序。

容量规划不能看全站日均。需要资源数、每个时间桶的峰值确认率、热门资源偏斜、平均服务时长、hold 到期分布、取消与 no-show 比例，以及确认延迟预算。地域上要问资源日历由哪个站点拥有；隐私上搜索响应不能泄露病人或预约原因。候补是否按到达、公平等级或医疗紧急度补位也必须先定规则。非目标是从用户点击确认推断他将来必然到场。

## 客观模型

最小命令是 `Search(resource_type, interval)`、`Hold(resource_id, interval, user, key)`、`Confirm(hold_version)`、`Cancel(booking_version)`、`CheckIn(booking)` 与 `Complete(booking)`。资源日历拥有时间段容量和 allocation version，预约服务拥有确认记录，支付方拥有资金状态，现场系统拥有到诊事实。状态沿 `AVAILABLE -> HELD -> CONFIRMED -> CHECKED_IN -> COMPLETED` 推进，取消和过期是带原因的新事件，不能删除历史。

对容量为 `C`、计划窗口为 `W`、平均占用时长为 `d` 的资源，理论最大服务数为 `N_max=floor(C×W/d)`；若 hold 到达率为 `λ`、平均存活时间为 `h`，结账中暂时不可售的能力近似为 `L_hold=λ×h`。热点来自某位医生或午间会议室，而非资源总数，因此按租户平均会隐藏单一时间桶的争用。

核心不变量是任一资源在任一时刻的有效 hold 与 confirmed 占用不超过容量；确认只引用尚未到期且版本匹配的 hold；同一预约意图不得重复占用；完成只能跟在实际 check-in 后。相交区间的条件更新是授权点，搜索缓存没有写权限。

## 必然约束

[DEDUCED:appointment-booking-overlapping-capacity-needs-one-conditional-owner] 相交时段的容量若由独立预约行各自检查，就没有一条状态能看见所有竞争者。最小反例是诊室 10:00–10:30 容量为一：A 与 B 都在 09:00:00 读到余量一，A 于 09:00:01 插入预约，B 于 09:00:02 插入另一行；两个插入互不冲突却共同超卖。确认必须经过资源时间桶的条件扣减、区间锁或等价的单序列 owner。静态切成互不重叠的房间后可分别授权，但两个 owner 不能同时借用同一份容量。

[DEDUCED:appointment-booking-confirmation-is-not-attendance] 服务端只能证明未来能力已经分配，无法在预约时知道用户是否到场、医生是否完成服务。若 Confirm 同时写成 Completed，用户爽约时系统既无法释放收费规则，也会生成虚假的履约与医疗审计。到诊必须由现场扫码或工作人员动作产生，完成由服务提供方确认；只有业务明确把“购买时段”而非“实际服务”作为商品时，财务上才可在确认时确认收入，但履约状态仍应分开。

[DEDUCED:appointment-booking-holds-trade-conversion-time-for-sellable-capacity] hold 保护慢速填写者不被后来者抢走，却在期限内排除真实买家。若十个名额被十个转化率为零的用户各 hold 十分钟，页面会显示售罄而成交数仍为零。缩短期限提高利用率但增加付款晚到后的退款，延长期限改善转化但放大闲置；幂等键只能去除同一请求重放，不能解决这项业务取舍。

## 从简单方案演进

最简单正确基线是一份按资源时间桶组织的日历，对 `available_version` 做条件更新并在同一事务写 hold。低并发时直接冲突返回并推荐相邻时段即可。若先查缓存再插预约行，即使每行唯一也守不住区间容量；若失败后无限重试，热门桶会把一次竞争放大成多次写。

第一个待校准切换指标是任一 `resource_id + time_bucket` 的条件写冲突率连续五分钟超过 5%，或确认 `p99` 消耗用户等待预算的 40%。达到后将该热点路由到固定分区的短队列，按到达顺序执行条件命令；5% 与 40% 是压测起点，调低会过早排队，调高会让重试和尾延迟先失控。不同资源仍可并行，不引入全局日历锁。

第二个待校准指标是过期 hold 占当日可预约容量超过 8%，或候补补位 `p95` 超过十五分钟。此时缩短 hold、限制续期次数并以取消事件驱动候补；8% 和十五分钟需按转化与通知渠道校准。新增代价是候补公平、通知过期和慢用户被挤出。若持续热点来自固定客户，可再采用每用户配额，而不是扩大资源 owner 数量。

没有选择跨地域各自确认后再合并，因为现实服务无法复制，冲突补偿会撤销已承诺预约。只有资源容量预先切成互不重叠的房间或医生班次、区域不能越界借卖时，边缘自治才重新变优。

## 设计决定

本设计让资源日历成为容量 owner：Hold 携带规范化区间、服务时长、用户与幂等键，在一个条件事务中检查所有覆盖桶并写入带期限和版本的占用。Confirm 只把同一版本从 HELD 转为 CONFIRMED；取消写原因并释放容量。响应丢失时按原键读取既有 hold，不新建另一个时段。日历故障时搜索可显示“需确认”，但新确认失败关闭。

付款授权在 hold 期间发生，不拥有时段；捕获策略依据取消规则在确认或履约后执行。回调晚于到期且时段已给另一人时，不撤销后来的预约，而把原订单送入退款或人工处理。到场与完成来自现场 owner，重复事件以 booking version 收敛。审计同时保存意图、容量授权、资金与履约四条状态链。

反选方案是先完成付款再抢时段，它消除 hold 闲置，却会让热门资源的大量用户先冻结资金再退款。只有支付无冻结成本、资源可自动替换且用户接受替代时段时，该顺序才更优。

## 运行与演进

关键 SLI 包括超容量确认数、条件冲突率、确认延迟、hold 占用分钟、过期回收延迟、候补等待、取消和 no-show 比例，以及 confirmed 后无可解释履约终态的年龄。过载时先降低搜索刷新与推荐，再限制单用户并发 hold，最后对热点资源启用有界候补；已在确认和现场 check-in 的命令优先。

故障演练时间线：09:59:50 A 获得 10:00 时段、期限版本 42；09:59:59 支付方授权但回调丢失；10:00:00 owner 提交到期并把时段补给 B；10:00:02 A 重试 Confirm。系统必须按版本拒绝 A、保留 B，并查询原支付参考号后退款，不能因授权时间早就覆盖当前容量。随后重复 A 请求应返回同一补偿结果。

发布新区间算法先影子比较相交集合，再灰度非热门资源；迁移资源分区时用 epoch fencing，源 owner 停写、目标追平所有 hold 与到期索引后再切路由。日志脱敏预约原因和用户标识，租户配额隔离候补洪峰。hold 时长实验必须同时观察转化、退款和可售能力，不能只优化接口延迟。

## 面试考察本质

这题考察的是：给定“相交预约不能超过资源容量、确认不能冒充现实履约”的不变量，因为客户端只看到旧余量且系统不能预知到场，候选人应推导出每资源时段的条件分配、有限 hold 和独立履约状态，并依据转化损失、资源利用率与公平性决定协调强度。

优秀回答会先说清日历、预约、支付和现场四个 owner，画出 hold 到确认再到到场的状态机，用具体区间反例否定“每预约一行”，并说明改期、晚回调和候补竞态。常见误区是把搜索缓存当库存、用幂等替代容量仲裁、让客户端时钟决定过期，或把 no-show 当成系统已完成。

二十分钟应守住区间不变量、接口和条件写；四十分钟补充热点队列、hold 公式、候补与支付补偿；六十分钟再讨论多资源套餐、分区迁移、隐私、公平和灰度。追问可固定在回调晚于释放的一秒边界，要求逐事件说明谁有写权限和用户看到什么。
