# DNS 与服务发现

## 表面题目

设计 DNS 与服务发现，表面动作是把服务名解析成一个或多个 IP，再建立连接。真正的状态变化发生在注册表：一个端点加入、变健康、失效或被摘除，权威发布面把某一完整成员快照映射到名称。解析成功只证明调用方拿到了一个格式正确且尚可复用的答案，不证明答案里的端点此刻有容量，更不证明连接一定成功。

题名容易把名称解析、健康检测和请求级负载选择混成一件事。DNS 回答的是“这个名字在某个 generation 对应哪些候选地址，以及答案可缓存多久”；健康检查负责产生端点事实；连接层才处理某一次连接失败与重试。本设计聚焦动态服务端点集合的发现与缓存新鲜度，不让 DNS 参与每请求鉴权、重试预算或持久数据键迁移。

## 反问与边界

先问服务实例变化频率、客户端和递归解析器数量、连接是短连接还是长连接、单服务端点规模、允许的摘除陈旧窗口、解析 `p99` 与连接建立预算。还要确认客户端是否遵守 TTL、是否有 JVM 或操作系统无限缓存，递归器是否支持负缓存、SRV/AAAA 和分答案截断，服务名是区域内还是跨地域。若长连接在端点摘除后仍可继续，本设计只约束“新连接选择”，不承诺主动断开全部旧连接。

正确性需要拆三段时间：故障检测 `δ_detect`、注册表到权威发布 `δ_publish`、已有缓存剩余 TTL `T_remaining`。业务要选择暂时连接失败与提高权威查询量之间的风险。安全上，注册写入需要工作负载身份，租户命名空间不能互相覆盖，答案传输与缓存要防污染；端点元数据不向未授权租户泄露。非目标包括用 DNS TTL 证明服务健康、给每个 HTTP 请求排队，以及保证网络分区时所有解析器瞬时看到相同集合。

## 客观模型

最小操作是 `Register(service, endpoint, lease)`、`ReportHealth(endpoint, observation)`、`Publish(service, expected_generation, endpoint_set, TTL)` 与 `Resolve(service)`。注册表拥有 `(service, endpoint_set, endpoint_health, generation)` 的权威写入；发布器只从一个已提交 generation 生成完整 RRset；递归器和客户端拥有带到期时间的只读快照。端点 agent 可以续租，健康控制器可以建议摘除，但只有注册表条件提交才改变成员身份。

核心不变量是：同一 generation 的答案来自一个完整端点集，不能把注册和摘除的不同中间状态拼接；已从权威集合摘除的端点，在声明的新鲜度上界之外不能再被新解析选择；未授权注册者不能劫持服务名。设有 `C` 个独立缓存客户端、TTL 为 `T`，每个客户端每 TTL 至少刷新一次，则权威查询率量级约为 `Q_authority ≈ C/T`。把 TTL 从六十秒降到六秒会把该部分查询压力放大约十倍。

摘除后的最坏旧答案窗口应分解为 `δ_detect + δ_publish + T_remaining`，而不是只写 TTL。例：检测十秒、发布两秒、缓存最多还剩六十秒，总窗口可达七十二秒。服务名热点会放大权威未命中，端点频繁抖动又会让 generation 和缓存失效持续推进；这与某个后端的每请求队列容量是不同的资源问题。

## 必然约束

[DEDUCED:dns-service-discovery-ttl-is-a-staleness-contract-not-a-health-check] TTL 的含义是解析器在多长时间内可复用某份答案，它并不触发端点存活证明。端点 B 可以在 TTL 开始后一秒崩溃，缓存仍合法持有包含 B 的集合；反过来，TTL 到期也不表示 B 已坏。故障检测和答案传播是独立延迟链，因此健康探针频率、发布延迟和缓存寿命必须分别观测。只有静态端点且业务把连接失败完全交给上层重试时，才可忽略快速摘除。

[DEDUCED:dns-service-discovery-cannot-guarantee-immediate-withdrawal-through-cached-dns] 12:00:00 权威答案为 `{A,B}`、TTL 六十秒；客户端在 12:00:01 缓存，B 于 12:00:05 崩溃并在 12:00:10 被权威摘除；12:00:30 客户端首次建立连接仍可从合法缓存选择 B。权威记录此时完全正确，却不能撤回已经发出的旧答案。只要任一解析层可在 TTL 内复用，立即全局摘除就不可能由缓存 DNS 单独保证；要么缩短新鲜度上界，要么把快速失败反馈放到连接层。

[DEDUCED:dns-service-discovery-endpoint-generation-prevents-partial-membership-answers] 若发布器逐条更新 `{A,B}` 到 `{B,C}`，并发读取可能在删除 A 之后、加入 C 之前得到只有 `{B}`，或在顺序相反时得到从未作为目标集合提交的 `{A,B,C}`。以 generation 原子发布完整集合，使答案可追溯到一个已提交快照，并允许解析指标关联该版本。它不能让所有缓存同时更新，但能防止单份答案本身是撕裂状态。

## 从简单方案演进

最简单正确基线是静态配置文件，服务发布时人工更新并重启客户端。它适合极少变化的端点，却让扩缩容和故障摘除依赖发布周期。当端点变化到配置传播 `p99` 超过允许陈旧窗口，先引入权威注册表和版本化全量发布；缓存 DNS 继续提供低成本名称读取。新增成本是注册租约、健康判定、generation 回滚和权威服务容量。

第一个待校准切换指标是“旧答案中的死端点导致的新连接失败率”连续十分钟超过 `0.1%`，且 TTL 是主要贡献：缩短服务 TTL 或启用客户端推送增量，并同时观测权威 QPS。第二个指标是缩短 TTL 后权威利用率超过 `60%`，或缓存未命中 `p99` 已超过连接建立预算的 `20%`：保留较长 DNS TTL 做粗发现，把快速健康反馈、连接重试与瞬时摘除放到 L7 或 sidecar。`0.1%/10 分钟` 与 `60%/20%` 都是待压测和故障注入校准的初始策略；调低前者会增加查询但更快收敛，调低后者会更早采用双层发现并增加数据面复杂度。

端点数量太大导致 UDP 截断、答案轮转和客户端选择偏斜时，DNS 只返回区域或代理层入口，不把成千上万个 pod 全塞进 RRset。注册表按服务分片，单服务的 generation 仍由一个权威序列发布；健康抖动通过最小稳定期与退出优先级抑制，但不得让已确定致命故障长时间留在答案中。

未选择把 TTL 设为零并让每次连接同步查询注册表，因为它把注册表故障和延迟放到所有连接关键路径，并制造惊群。当客户端数量很小、成员变化极快且每次选择必须读取最新标签时，直接客户端发现会重新变优。也未选择让 DNS 根据每个请求的实时队列长度返回节点；缓存会迅速使负载信号过期，这应由连接或请求级代理承担。

## 设计决定

本设计以注册表为端点成员权威，以 generation 为服务级发布单元。工作负载用短租约注册，健康控制器综合主动探针、被动连接失败和发布保护期后提出状态变化；发布器以 `expected_generation` 条件提交完整 endpoint set，再生成带 TTL 的答案。DNS 读取不在每次查询协调，只返回最近已发布快照及其版本。

客户端对答案随机化并在连接失败时有限次尝试集合内其他地址，同时上报被动失败；它不会把一次失败写回注册表直接摘除共享端点，避免单客户端网络故障污染全局健康。负缓存 TTL 远短于正常服务答案，防止服务刚创建却长期保持不存在。权威面不可用时，未过期缓存可继续粗发现，过期后是否短时 stale-if-error 由服务风险明确配置。

反选方案是给所有客户端维持注册表推送流。它能在秒级内传播摘除，却增加长连接规模、增量丢失后的全量同步和 SDK 一致性。对高价值、成员高频变化且客户端平台可控的内部服务，它会重新变优；对开放、异构和低变化客户端，缓存 DNS 更稳健。

## 运行与演进

核心 SLI 是解析成功率与 `p99`、答案 generation 年龄、权威与递归 QPS、缓存命中率、从健康判死到发布的延迟、旧答案死端点连接失败率和各端点被选择偏斜。过载时先降低非关键元数据、合并健康抖动和延后扩容加入；已确认故障的摘除优先。不得通过无限延长过期答案掩盖注册表故障而不暴露 stale 状态。

故障演练时间线：12:00:01 客户端缓存 generation 18 的 `{A,B}`、TTL 六十秒；12:00:05 B 崩溃，12:00:08 探针达到判死阈值，12:00:10 注册表提交 generation 19 的 `{A}`；递归器 R 仍持有 generation 18。12:00:30 新连接选 B 失败后在预算内尝试 A 并成功；12:01:01 缓存刷新后不再选择 B。演练分别测量检测、发布、剩余 TTL 和连接恢复，避免只把总失败窗归因于 DNS。

发布新记录类型先让解析器双读并记录旧新答案差异，再按服务灰度；回滚继续保留 generation 单调性，不能重新发布更低版本。注册协议升级要容忍新旧 agent 并存，租约字段只增不改。跨地域服务名明确区域优先和灾备 generation，租户注册权限、答案审计与域名变更记录保持可追踪。

## 面试考察本质

这题考察的是：给定“已摘除端点在新鲜度边界后不再接收新连接”的不变量，因为递归器和客户端无法即时得知注册表变化，候选人能否把健康检测、权威发布与 TTL 缓存三段延迟分开，并用连接失败预算和权威查询成本决定 DNS 是粗粒度发现还是快速发现。

优秀回答会说明 TTL 是陈旧契约而非健康证明，写出 `C/T` 查询量和 `δ_detect + δ_publish + T_remaining` 失败窗，用 `{A,B}` 中 B 崩溃的时间线解释权威正确仍会连接失败，并用 generation 防止撕裂成员集。常见误区是把 TTL 当心跳、承诺立即撤回已缓存答案，或让 DNS 读取瞬时后端队列来做每请求均衡。

二十分钟回答应完成注册表、缓存层和新鲜度不变量；四十分钟加入查询放大、健康发布链、连接重试与最小反例；六十分钟再讨论推送发现、记录规模、跨地域和注册安全。追问要固定在“12:00:30 客户端为什么还能选 B”，要求候选人指出它持有哪个 generation、谁有权改变成员以及哪一层承担快速恢复。
