# 协同编辑器

## 表面题目

设计协同编辑器，让多个用户同时修改同一文档，在线时低延迟互见，断线时继续编辑，重连后不重复、不丢失并最终收敛。表面上像发送击键消息，真正状态变化却是多个副本各自生成操作，系统必须定义并发插入、删除和格式化的确定合并语义。单个网关的到达次序不是所有作者共同观察到的事实。

成功至少包括本地操作耐久保存、权威层接受并合并、其他副本最终拥有同一操作集合，以及可见文档收敛，但这些是不同提交点。本地保存只产生可恢复、可重试的 `LOCAL_PENDING`，不授予未来写入共享文档的权利；只有权威层按当前成员世代、协议版本和历史保留窗校验并提交后，操作才属于共享真相。在线光标和选区是可丢失提示；未被客户端或服务端持久保存的离线编辑无法凭空恢复。

## 反问与边界

先问文档模型是纯文本、富文本还是树，是否要求完全离线编辑，最大离线时长、并发作者数、单文档大小和撤销语义是什么；OT 还是 CRDT 不是偏好题，而由中心版本可用性、客户端复杂度和合并要求决定。还需确认本地回显预算、远端可见延迟、恢复字节、历史保留和协议升级窗口。

安全边界包括文档成员版本、操作授权、恶意大操作、租户隔离与离线设备撤权。非目标是让所有用户观察完全相同的瞬时光标位置、用墙钟决定冲突胜负、永久保留所有墓碑，或在客户端丢失未上传本地数据后仍保证恢复。

## 客观模型

最小接口为 `Open(document_id,snapshot_epoch)`、`SubmitOps(document_id,actor,ops,context)`、`Sync(frontier)` 与 `PublishSnapshot(compaction_epoch)`。权威状态是 `document -> {snapshot,protocol_version,compaction_epoch}` 和带 `(actor,counter)`、因果上下文或服务器基准版本的操作集合。在线广播只加速传播；操作身份、转换规则/CRDT 合并和压缩世代拥有收敛语义。

不变量是：同一逻辑操作只合并一次；所有兼容副本在得到相同已提交操作集合后产生相同可见文档；删除与格式化引用稳定元素或明确版本，不只引用会漂移的字符下标；`LOCAL_PENDING` 不能被读成共享提交；上传时必须验证当前成员世代、协议版本和 compaction/保留窗，被撤权或超窗客户端的旧操作只能导出，不能越过权威提交点；压缩后仍允许回归的操作必须可解释。

对中心 OT，单次变换成本可近似随未确认窗口 `u` 与并发作者数 `e` 增长为 `O(u×e)`；朴素作者向量元数据约为每操作 `O(e)`。恢复成本约为 `snapshot_bytes + missing_ops_bytes`。热点来自少数超大文档、长离线分支和墓碑增长，而非全站平均击键率。

## 必然约束

[DEDUCED:collaborative-editor-convergence-is-not-a-single-arrival-order] 文档 `abc` 上，甲离线在位置 1 插入 `X`，乙同时插入 `Y`。两个操作都合法且双方断线时不存在共同即时总序；若“后到覆盖前到”，必丢一个编辑。协议必须定义确定 tie-breaker 或 OT 转换，使相同操作集合得到同一结果。只允许在线串行编辑时，中心到达序才足够。

[DEDUCED:collaborative-editor-operation-identity-and-context-prevent-replay-duplication] 重连会重发未确认操作，只有 `(actor,counter)` 能识别同一逻辑操作，只有因果上下文或基准版本能判断它与已提交操作是先后还是并发。正文相同不能去重，因为用户可能合法输入两个相同字符；裸下标也会因并发插入漂移。

[DEDUCED:collaborative-editor-compaction-must-preserve-offline-merge-meaning] 删除墓碑和旧位置元数据可缩小状态，但一个仍被允许回归的离线操作可能引用它们。若服务在其保留窗内静默压缩，操作要么无法定位，要么作用到错误字符。压缩必须产生 compaction epoch，要求旧副本先转换或重新基于快照；否则只能拒绝超出窗口的离线分支并明确告知用户。

[DEDUCED:collaborative-editor-local-durability-is-not-shared-commit] 甲在仍有权限时离线保存操作 A，随后管理员撤权并推进成员世代；设备本地能恢复 A，只证明磁盘没有丢数据，不能证明甲在未来仍有共享写权限。重连时服务必须用当前成员世代、操作协议和 compaction/保留窗重新授权、解释并提交 A，成功提交后才进入共享操作集合。任一校验失败时保留本地内容供复制或导出，但不得自动合并、广播或让旧 ACK 伪装成共享成功。

## 从简单方案演进

最简单正确基线是必须在线的单文档主节点：按服务器版本接收操作，客户端重连从最后版本补拉。它容易审计，但离线操作只能拒绝或手工冲突。当产品要求离线编辑，或版本冲突重试率达到待校准回显预算的 10%，引入带身份与上下文的 OT/CRDT 合并，而不是继续最后写胜。

单文档并发作者超过 50，或 OT 变换 `p99` 超过编辑回显预算的 30% 时，把文档按稳定块/子树拆协作域，或选择适合离线的 CRDT；跨块移动仍需明确事务边界。拆分降低热点，却新增跨块引用、局部快照和权限传播。

当墓碑/元数据字节超过可见正文的 2 倍，或最长离线跨度达到历史保留窗的 70%，创建新 compaction epoch 快照，旧客户端先同步转换后再上传。10%、50 人、30%、2 倍与 70% 都是待压测和产品校准参数：调低增加快照与协议复杂度，调高放大 CPU、内存和旧操作风险。

反选“服务器为所有击键分配唯一序号，客户端只接受该顺序”；它在必须在线、作者很少且审计简单回放优先时更合适，一旦允许离线多写就会把冲突变成拒绝或丢数据。

## 设计决定

本设计按文档类型选择一种固定合并协议，不混用含糊语义。客户端先把本地操作以稳定 ID 持久化为 `LOCAL_PENDING` 并立即回显，界面明确区分“保存在此设备”和“已进入共享文档”。在线上传时携带 frontier、成员世代、协议版本与 compaction epoch；服务读取当前权限与保留窗，去重、解释并权威提交后才返回 shared commit ID、广播新的 frontier，并把本地状态升级为 `SHARED_COMMITTED`。光标引用稳定元素并带短 TTL，丢失不影响文档真相。

故障时间线是：0 秒甲乙都在版本 100；1 秒甲断线并把 A 保存为 `LOCAL_PENDING`；2 秒乙提交 B；10 秒管理员撤销甲的权限并推进成员世代；30 秒甲上传 A 与旧世代，服务按当前权限拒绝，A 仍可在本机导出但共享文档只含 B。若甲未被撤权且协议与保留窗仍兼容，服务才可合并 A、返回 shared commit ID；旧网关重放已提交 A 时因 ID 已见被忽略。能保证本地 A 可恢复并重试、权威接受的操作只合并一次并最终收敛；不能保证任意离线编辑最终进入共享文档。

未采用裸字符下标和最后写胜，因为并发插入会改变下标并丢编辑。若对象是一个不可合并的单值字段，且产品明确接受最后写胜，简单版本寄存器才重新变优。

## 运行与演进

核心 SLI 是本地回显延迟、权威接受 `p99`、远端收敛时间、重复操作命中、转换/合并 CPU、单文档 frontier 差距、墓碑比、快照恢复字节与无法转换的旧操作数。按文档、作者数和离线跨度分桶。过载时先丢光标、降低远端广播频率，再限制大批操作；已保存编辑的同步路径保留。

演练两个断线副本在相同位置插入并让旧网关重放，验证已权威提交的结果字节一致且每个 op 只出现一次；再分别在上传前撤权、推进协议版本和越过 compaction 保留窗，验证 `LOCAL_PENDING` 仍可导出但不会进入共享文档。协议升级先由读端理解双版本，影子计算新旧结果并比较摘要，小比例文档进入新 epoch；回滚不能把已写新操作解释成旧语义。

跨地域可让操作在本地暂存，但合并协议决定冲突，不用墙钟。撤权提升成员世代，离线客户端保留本地导出权但不能继续写共享真相。文档内容、操作日志与 presence 游标分别保留和加密，成本按活跃合并 CPU、历史元数据与快照带宽归因。

## 面试考察本质

给定“相同的权威已提交操作集合最终必须产生同一文档，且本地持久化不等于共享提交”的不变量，因为断线副本不知道他人的并发修改、字符下标会漂移，权限和协议也会在离线期间变化，候选人应推导出稳定操作身份、因果上下文、当前授权、确定 OT/CRDT 语义与压缩世代；再依据离线要求、作者数和元数据成本选择复杂度，而非把本地 ACK 或网关到达顺序伪装成协作成功。

优秀信号包括先定义文档模型、给出两个离线插入反例、区分操作收敛与光标提示、说明客户端本地持久化边界。常见误区是只说 WebSocket、最后写胜、按正文去重，或压缩墓碑后仍无限接纳旧操作。

二十分钟回答覆盖操作 ID、版本与中心基线；四十分钟加入并发合并、离线恢复和最小反例；六十分钟再讨论分块、压缩 epoch、撤权和协议迁移。追问要求候选人展示同一操作集为何收敛、未提交数据在哪里，以及哪两个指标触发拆分或压缩。
