# 作业调度器

## 表面题目

设计作业调度器，表面上是接收一次性或周期计划，在到期时选择 worker 执行，并让用户查看取消、成功或失败。真正的状态变化是一个由“计划版本 + 逻辑时间槽”确定的触发，从 `PENDING` 变成一个权威 run，再由当前 claim 持有者把 run 推进到终态。成功不能只表示某台机器启动过进程，而要说明副作用是否已提交、失败能否重试、迟到结果如何处理。

题名容易掩盖两类不同语义：数据压缩等可幂等批任务允许至少一次执行，发工资或发邮件则不能把进程只运行一次等同于业务效果只发生一次。本设计负责触发、claim、重入、取消和审计；作业内业务事务由目标系统拥有。DAG 依赖、容器放置和代码隔离分别属于其他题。

## 反问与边界

先问触发语义：错过的每个时间槽都补跑，还是只跑最新一次；时区和夏令时怎样解释；允许提前、最大迟到和抖动是多少。再问运行语义：任务能否并发重入，失败最大重试次数，超时后是取消还是结果未知，取消是否必须阻止尚未提交的副作用。SLO 应拆为到期延迟、队列等待、运行结果确定时间和漏触发数，而不是一个平均成功率。

规模要用活动计划数 `N`、每秒到期触发 `D`、运行时长分布 `T`、热点整点比例和租户偏斜描述。故障域包括调度领导者、队列、worker、状态库和外部目标。安全边界要求任务身份只能获得本次 run 所需的短期能力。非目标是从任意不支持幂等键或查询的外部 API 中凭空制造 exactly-once；这种目标只能通过业务对账与补偿收敛。

期望状态是每个有效计划版本应产生哪些 run，以及每个 run 的取消/完成目标；观测状态是当前逻辑时间、run 状态、claim epoch、心跳和目标回执。调度分区领导者创建触发，run reconciler 回收失租约运行，worker 只执行被授予的单次 claim。墙钟用于发现候选时间槽，但数据库提交序决定同一槽是否已创建。

## 客观模型

最小接口为 `PutSchedule(spec, version)`、`CancelSchedule`、`Claim(run_id)`、`Heartbeat(run_id, epoch)`、`Complete(run_id, epoch, result)` 与 `GetRun`。`trigger_id = hash(schedule_id, schedule_version, logical_slot)`；run 状态为 `PENDING -> CLAIMED(epoch, lease_until) -> SUCCEEDED|FAILED|CANCELLED`，失租约可转回 `PENDING` 并把 epoch 加一。触发记录是计划到 run 的唯一线性化点，完成记录是运行控制状态的仲裁点。

不变量是同一 trigger identity 最多有一个权威 run；只有当前 claim epoch 能改变 run 终态；取消世代之后创建的旧计划触发无效；外部副作用必须携带 `{run_id, epoch}` 或稳定业务幂等键。若到期率为 `D`、平均运行时间为 `E[T]`、一次尝试成功率为 `p`，需要的并发下界约为 `C = D*E[T]/p`；整点有偏斜系数 `s` 时入口要承受 `sD`，不能用全天平均容量规划。

反例：12:00 的槽由领导者甲创建 run R，worker W1 获得 epoch 7，网络分区后租约过期；W2 获得 epoch 8 并成功调用扣费。W1 随后恢复并返回。状态库必须拒绝 W1 的完成；若扣费端不比较 epoch 或业务幂等键，两个扣费都可能真实发生。因此 lease 只控制内部写权限，外部目标也必须参与 fencing，或任务必须能查询和补偿。

## 必然约束

[DEDUCED:job-scheduler-trigger-identity-must-survive-recomputation] 周期触发必须由稳定的计划版本与逻辑时间槽标识，否则领导者切换或补跑会把同一次触发误建为多个运行。最小事件序列是甲扫描到 12:00、插入前失联，乙重扫同一分钟并插入，甲恢复后也插入。若 ID 是随机值，两条都合法；稳定 trigger identity 加唯一条件写才能把重算变成幂等。边界是业务明确允许每次扫描都形成独立采样时，此不变量才可放宽。

[DEDUCED:job-scheduler-lease-needs-side-effect-fencing] 执行租约只能证明 worker 暂时有权运行；若外部副作用不校验更高 fencing epoch，过期 worker 仍可在新 worker 之后提交。网络不能让 W1 知道自己的心跳未被接受，而暂停进程也不能撤回已发请求。故数据库 status、分布式锁或心跳都不足以隔离外部写；目标系统应以 run ID 去重、拒绝低 epoch，或把副作用放进同一事务 outbox。

[DEDUCED:job-scheduler-retry-cannot-prove-exactly-once-effects] 调度器在响应丢失后无法同时确定外部操作是否成功与是否应重试，必须依赖幂等目标、查询或可对账补偿。比如扣费已提交但 200 响应丢失；不重试可能漏扣，换新身份重试可能双扣。至少一次执行是调度器能独立提供的边界，“恰好一次业务效果”必须由端到端协议定义。

## 从简单方案演进

最简单正确基线是一张带唯一 trigger ID 的表，单个扫描器按索引查询到期计划，事务中插入 run，worker 用条件更新 claim。它适合低到期率，失败后任何实例可重扫。第一个待压测切换指标是到期扫描 `p99` 连续十分钟超过允许迟到预算的三分之一，或一次扫描返回超过五万候选；此时按下一触发时间分桶和分区所有权，代价是分桶迁移、热点整点和跨分区补跑。

第二个待业务校准指标是 claim 冲突超过百分之十且重试流量占写入百分之二十五，或 ready queue 的 `p99` 等待超过任务 SLO 的一半；此时从抢占式轮询转为按租户公平派发和有界预取。百分比不是测量事实：调低会更早引入调度复杂度，调高会让数据库承担无效竞争。过载时延迟低优先级补跑，不能丢失权威 run。

当作业平均时长远超租约时，心跳维持续租；可检查点任务把进度和 checkpoint generation 一起提交。未选择为每次触发启动一个永久计时器，因为进程重启会丢计时且百万计时器迁移困难；若计划数少、单机故障由外部监督且允许人工补跑，它会重新变优。未选择无限 lease，因为失联 worker 会永久占住运行权。

## 设计决定

计划采用不可变版本，修改产生新版本并以 activation slot 切换。分区领导者根据权威时间窗口枚举 logical slot，通过唯一插入建立 run；扫描可重复且不承担唯一正确性。worker 获得单调 claim epoch 和短租约，所有心跳、checkpoint、完成及取消确认都条件匹配 epoch。旧 worker 过期后返回时，日志进入带 attempt 标签的诊断存储，但完成、checkpoint 和 outbox 提交被拒绝；若外部返回未知，则 run 进入 `RECONCILING` 查询目标，而非覆盖新结果。

正常路径以 run ID 作为目标幂等键。目标支持 fence 时比较 epoch；只支持幂等键时，同一 run 的所有 attempt 复用键；两者都不支持时，系统明确提供至少一次，失败进入对账队列。取消先增加 run epoch、撤销新 claim，再向当前 worker 发中断；已提交副作用不伪装成已撤回。调度存储不可用时停止创建与 claim，已运行任务是否继续由能力有效期决定。

反选是消息 TTL 到期即触发。它简化扫描，但计划编辑、补跑、夏令时和队列丢失很难审计；当仅有短期一次性延迟任务、消息系统自身提供持久定时与唯一键时会重新变优。

## 运行与演进

关键 SLI 是漏触发计数、重复权威 run 计数、due-to-claim 延迟、按租户 queue age、租约回收次数、过期 epoch 写拒绝数、结果未知年龄和业务对账差异。演练时间线固定为：00 秒 W1 获 epoch 41；20 秒心跳响应丢失；40 秒租约过期，W2 获 42；45 秒 W2 用 `{R,42}` 提交；50 秒 W1 恢复提交 `{R,41}`。预期目标与状态库都拒绝 41，日志仍可查，用户只看到 42 的结果。

待演练切换指标一是未知结果超过一百条或最老超过十五分钟时暂停同目标新任务并优先查询；二是某租户连续五分钟占据超过百分之三十 worker 或 ready age 达全局中位数五倍时启用更严格配额。阈值随副作用风险和租户合同校准：工资任务可能一条未知就停，缓存刷新可容忍更多。

发布新的触发算法先影子枚举并比较 trigger ID 集合，再按计划分区灰度；回滚仍能读取新 schedule version。时区数据库升级要保留旧版本解释已生成槽。灾备恢复从最后持久 trigger 游标重扫，唯一键吸收重复。密钥按 run 注入短期能力，审计记录操作者、计划版本、epoch 和目标回执，不保存敏感参数明文。

## 面试考察本质

给定“一个计划版本的一个逻辑时间槽只能形成一个权威运行，且过期执行者不能提交业务效果”这一不变量，因为领导者和 worker 都只能看到可能迟到的心跳与外部响应，候选人应推导出稳定 trigger identity、可重算扫描、claim epoch 和端到端 fencing，并在迟到容忍、恢复速度与重复副作用风险之间决策。

优秀信号是主动区分触发、执行进程和业务效果三层，画出旧 worker 在新 worker 之后返回的时间线，承认 exactly-once 的端到端边界，并给出补跑与取消语义。常见误区是用 cron + 队列即宣布完成、把数据库锁当外部 fence、随机生成每次 run ID，或让 worker 超时自动认定目标失败。

二十分钟应完成 trigger ID、状态机和租约；四十分钟加入公平队列、幂等副作用与结果未知；六十分钟再讨论时区、补跑、checkpoint、灾备和能力隔离。追问应落到“W1 过期但扣费已发出”时每个参与者知道什么，这正是作业调度器区别于普通任务队列的客观本质。
